## M6 Post-Merge Fix — Auth Lockout Test CI Reliability (KI-035) (2026-07-12)

**Root Cause**: `tests/auth/test_service.py::test_authenticate_locks_out_after_repeated_failures` constructed a real, Redis-backed `AccountLockout` with no reachability guard, unlike every other Redis-dependent test in the project. `.github/workflows/ci.yml` has never provisioned a `redis` service (only `postgres`) — in CI, `AccountLockout` correctly failed open (its documented, production-correct behavior), so the account was never actually locked after 3 wrong-password attempts, and the test's final assertion (expecting `AccountLockedError` on a 4th, correct-password call) failed because authentication legitimately succeeded. Not an M6 regression — no file this milestone touched relates to authentication — but discovered via CI immediately after M6 merged.

**Tests Changed**: `tests/auth/conftest.py` gained `FakeAccountLockout`, an in-memory double matching `AccountLockout`'s exact interface. `tests/auth/test_service.py`'s `_lockout()` helper now returns the fake by default (removing an implicit Redis dependency from every test in the file that doesn't specifically need real enforcement — 20 of the file's 22 tests, none of which were ever meant to test Redis itself). Two new tests were added: `test_authenticate_enforces_lockout_when_redis_available` (real `AccountLockout`, skips gracefully if Redis is unreachable, matching `tests/auth/test_lockout.py`'s own convention) and `test_authenticate_fails_open_when_redis_unavailable` (a real `AccountLockout` pointed at an always-unreachable address — `redis://localhost:1/0`, matching `test_lockout.py`'s existing pattern — asserting both that `authenticate()` never raises `AccountLockedError` and that the `"failing open"` warning is actually logged, via `caplog`).

**Security Tests**: The two new tests are, collectively, this fix's own security-relevant coverage: they prove `authenticate()`'s lockout-enforcement and fail-open behaviors are both intact, at the integration level, in both Redis-reachable and Redis-unreachable conditions — a stronger guarantee than existed before this fix, which only ever tested the "happy path" (Redis reachable) implicitly and untested the fail-open path at this level at all.

**Performance Notes**: `test_authenticate_fails_open_when_redis_unavailable` costs ~8s (two real connection attempts against an unreachable address, ~2s each on this environment, matching `test_lockout.py::test_fails_open_when_redis_unreachable`'s existing ~4s/2-call baseline) — reduced from an initial ~16.6s draft (which looped 3 wrong-password attempts before the correct one) by cutting to a single wrong-password attempt, since `max_attempts=1` already exercises the "would have locked a reachable Redis" threshold without added looping.

**Failed Tests**: 0 at final verification (278 total across the full project: 276 pre-fix + 2 new). No other test was affected — `app/auth/lockout.py` itself was not modified, and every other file's Redis-dependent test already followed the skip-gracefully-or-target-unreachable convention this fix now also applies consistently within `test_service.py`.

**Fixes Applied**: The change described above. `ruff`/`black` unchanged (no style violations introduced).
