## ADR-001 — Resolve the Deployment Trigger Ambiguity

- **Date**: 2026-07-02
- **Status**: Accepted
- **Context**: Founder Specification Part 2.11 names two production-deploy triggers ("merge to `main`" and "tagged release") without reconciling them.
- **Problem**: An unreconciled dual trigger risks ambiguity about which commit is actually live, or accidental double-deploys.
- **Options Considered**: (1) Merge to `main` deploys straight to production. (2) Tagged release only, no auto-deploy on merge. (3) Merge to `main` auto-deploys to staging; a tag deploys to production.
- **Final Decision**: Option 3 — merge to `main` auto-deploys to staging; a tagged release (`vX.Y.Z`) deploys to production.
- **Rationale**: Gives a deliberate, human-gated production deploy step without inventing a `develop` branch the spec explicitly defers.
- **Tradeoffs**: Requires discipline to actually cut tags for production releases; adds one manual step vs. pure continuous deployment.
- **Future Implications**: Documented in `.claude/GIT_WORKFLOW.md`; applies starting at the Deployment & Observability milestone (M8).

## ADR-002 — `calculation_version` Is Non-Negotiable From Migration #1

- **Date**: 2026-07-02
- **Status**: Accepted
- **Context**: Founder Specification permits overwriting/correcting historical data at any time (reimport) while also mandating that identical historical data yields identical simulation results — and explicitly allows deferring the `calculation_version` field that would reconcile the two.
- **Problem**: Deferring the field guarantees an eventual retrofit against real, already-created simulation records once reproducibility is finally enforced.
- **Options Considered**: (1) Defer per the spec's literal allowance. (2) Add `calculation_version` (or equivalent) to the simulation/results schema from the very first migration, even if unused initially.
- **Final Decision**: Option 2.
- **Rationale**: Retrofitting a versioning column onto a table with live production data is far more expensive and riskier than including an unused column now.
- **Tradeoffs**: A small amount of unused schema surface exists until versioning logic is actually implemented.
- **Future Implications**: Binding on the Database Schema milestone (M1); documented in `.claude/DATABASE_RULES.md`.

## ADR-003 — Nullable Output Columns on `simulations` and `ai_explanations`

- **Date**: 2026-07-02
- **Status**: Accepted
- **Context**: The Founder Specification marks output columns (`final_value`, `explanation_text`, etc.) as NOT NULL, while the same spec's own status enums include `pending`/`failed` states where no output can exist yet.
- **Problem**: A literal implementation would violate its own schema constraints the first time a simulation or AI generation fails or is left pending.
- **Options Considered**: (1) Implement literally as NOT NULL and special-case failures elsewhere. (2) Make output columns nullable, matching the documented status states.
- **Final Decision**: Option 2.
- **Rationale**: The NOT NULL constraint as written is an internal spec bug, not an intentional design choice; fixing it at the schema level is simpler and safer than working around it in application code.
- **Tradeoffs**: None material — this is a correction, not a design tradeoff.
- **Future Implications**: Binding on the Database Schema milestone (M1); documented in `.claude/DATABASE_RULES.md`.

## ADR-004 — Redis Deferred Until Actually Needed

- **Date**: 2026-07-02
- **Status**: Accepted
- **Context**: The Founder Specification lists Redis as part of the approved MVP stack but marks it "optional" during early development.
- **Problem**: Provisioning Redis in the M0 repository foundation before any feature (caching, rate limiting) actually needs it adds operational surface with no corresponding value yet.
- **Options Considered**: (1) Include Redis in `docker-compose.yml` from M0 for parity with the eventual target stack. (2) Exclude Redis from M0 entirely; introduce it exactly when a milestone needs caching or rate limiting.
- **Final Decision**: Option 2 — `docker-compose.yml` at M0 contains Postgres and the backend only.
- **Rationale**: Matches the project's own "don't provision speculatively" discipline; Redis is scheduled to arrive at the API Layer milestone (M4) when rate limiting is actually implemented.
- **Tradeoffs**: The API Layer milestone must explicitly add Redis to Docker Compose and wire it in — tracked as part of that milestone's scope, not silently assumed.
- **Future Implications**: Documented in `.claude/SYSTEM.md` and `.claude/PERFORMANCE_BUDGET.md`.

## ADR-005 — Adopt Conventional Commits

- **Date**: 2026-07-02
- **Status**: Accepted
- **Context**: The Founder Specification requires descriptive commit messages but does not mandate a specific convention.
- **Problem**: Without a shared convention, commit history becomes harder to scan and changelog-generation harder to automate later.
- **Options Considered**: (1) Free-form descriptive commits only. (2) Adopt Conventional Commits (`feat:`, `fix:`, `refactor:`, `test:`, `docs:`, `chore:`, `perf:`).
- **Final Decision**: Option 2.
- **Rationale**: Low cost, high consistency payoff, and plays well with `CHANGELOG.md` maintenance.
- **Tradeoffs**: None material.
- **Future Implications**: Documented in `.claude/GIT_WORKFLOW.md`; applies to every commit from M0 forward.

## ADR-007 — Financial Analytics Is a Separate Future Milestone

- **Date**: 2026-07-02
- **Status**: Accepted
- **Context**: The founder directed that advanced financial metrics (Volatility, Sharpe, Sortino, Maximum Drawdown, Calmar Ratio, Rolling Return Analysis, correlation) must not be permanently folded into the Simulation Engine.
- **Problem**: Continuously reopening the Simulation Engine's highest-risk, most heavily-tested code path (90%+ coverage target, known-answer tests) to add new metrics increases regression risk in the platform's most critical component.
- **Options Considered**: (1) Add each new metric directly into the Simulation Engine as it's requested. (2) Reserve a dedicated, separate Financial Analytics module/milestone that reads simulation output rather than extending the engine itself.
- **Final Decision**: Option 2. Simulation Engine v1 owns only Final Value, ROI, CAGR, Inflation-Adjusted Return, Dividend Contribution, and Opportunity Cost Analysis.
- **Rationale**: Keeps the Simulation Engine's test surface small and stable; advanced analytics can iterate independently without touching the sole-source-of-financial-truth code path.
- **Tradeoffs**: Advanced metrics ship later than they otherwise might if bundled into M3.
- **Future Implications**: Documented in `.claude/MVP_RULES.md`; a Financial Analytics milestone is reserved post-MVP, positioned after the Simulation Engine.

## ADR-008 — Economic Indicators as a Separate Catalog + Time-Series Pair

- **Date**: 2026-07-05
- **Status**: Accepted — Pending Founder Review (see `docs/KNOWN_ISSUES.md` KI-005)
- **Context**: The Founder Specification names Economic Indicators as one of the nine database domains (Part 2.6.3) with a required index (2.6.13), but Part 2.6.27 (where its physical table would have been specified) was never written — a confirmed spec gap, not a design choice made by the founder.
- **Problem**: Economic indicators (CPI, unemployment rate, etc.) needed to be given a physical schema without founder guidance on its shape, and without over-designing beyond what M1 needs.
- **Options Considered**: (1) Add an `economic_indicator` value to `asset_type_enum` and store indicators as rows in `assets`/`historical_prices`. (2) Create a wholly separate catalog table (`economic_indicators`) plus a time-series table (`economic_indicator_values`), structurally parallel to `assets`/`historical_prices` but independent.
- **Final Decision**: Option 2.
- **Rationale**: Economic indicators are not investable instruments — they have no `symbol`, no `asset_type` in the tradable sense, and no simulation ever targets one directly (they're context, e.g. for inflation adjustment). Forcing them into `assets` would either add nullable, asset-specific columns that don't apply to indicators, or force indicator-specific columns onto every tradable asset row. A structurally-parallel-but-separate pair keeps both models clean.
- **Tradeoffs**: Two new tables instead of reusing `assets`/`historical_prices` directly; some duplicated structural pattern (catalog + time series) across the two domains.
- **Future Implications**: If the founder later decides indicators should be treated as a special asset type instead (e.g. to unify search/lookup code), this ADR would need to be superseded — flagged in `docs/KNOWN_ISSUES.md` KI-005 pending that review.

## ADR-009 — `audit_logs.user_id` Uses `ON DELETE SET NULL`

- **Date**: 2026-07-05
- **Status**: Accepted
- **Context**: `audit_logs.user_id` is a real (non-polymorphic) foreign key to `users.id`, recording who triggered an event. The Founder Specification does not specify delete behavior for this relationship.
- **Problem**: If a user account is later deleted (e.g. account deletion request), the default FK behavior would block the deletion entirely while audit rows referencing that user exist, or (if cascade were used) would destroy audit history — either outcome is wrong for a security/audit log.
- **Options Considered**: (1) Default `NO ACTION` — blocks user deletion while any audit log references them, forcing an awkward cleanup step. (2) `ON DELETE CASCADE` — deletes audit history along with the user, destroying exactly the record a security incident review would need. (3) `ON DELETE SET NULL` — the audit row survives, `user_id` becomes null, and `details`/`entity_type`/`entity_id` retain the rest of the event context.
- **Final Decision**: Option 3.
- **Rationale**: Audit logs exist to answer "what happened and who did it" even after the "who" no longer has an active account — this is the same reasoning that makes audit logs immutable (no `updated_at`) in the first place.
- **Tradeoffs**: A null `user_id` on an old audit row means "the acting user's account no longer exists," which must be handled gracefully wherever audit logs are displayed (not implemented yet — no admin UI exists at M1).
- **Future Implications**: Applies to any future FK from an immutable/audit-style table to `users` — default to `SET NULL`, not cascade or restrict, unless a specific reason argues otherwise.

## ADR-010 — Standardize on LF Line Endings, CRLF Only for Windows-Native Scripts

- **Date**: 2026-07-06
- **Status**: Accepted
- **Context**: Development happens on Windows (the primary environment), but the project runs in Docker (Linux containers), deploys to Linux-based hosting (Railway/Render), and runs CI on `ubuntu-latest` GitHub Actions runners. A repository hygiene pass (not a feature milestone) surfaced the need to make line-ending behavior explicit rather than left to each contributor's local Git/editor configuration.
- **Problem**: Without an explicit policy, line endings are governed by each contributor's local `core.autocrlf` setting (this machine's is `true`), which silently rewrites LF to CRLF on checkout and back on commit. That's invisible and harmless in isolation, but across Windows and Linux contributors, Docker builds, and CI runners, it produces spurious whole-file diffs, inconsistent `git blame`, and — worse — scripts with CRLF line endings fail or behave unexpectedly when executed inside Linux containers (`.sh` files, Dockerfiles) where a stray `\r` at the end of a shebang line or command can break execution outright.
- **Options Considered**: (1) Leave line endings entirely to each contributor's local Git config (the status quo, implicit and inconsistent). (2) Force LF everywhere, including `.bat`/`.ps1`. (3) Force LF for everything that is ever built, run, or interpreted inside Linux/Docker/CI (`.py`, `.md`, `.yml`/`.yaml`, `Dockerfile`, `.sh`), and CRLF only for the two file types that are Windows-native by definition and never execute in a Linux context (`.bat`, `.ps1`).
- **Final Decision**: Option 3, implemented via `.gitattributes` (repository-enforced, not dependent on contributor config) and mirrored in `.editorconfig` (so editors save matching line endings locally before Git ever has to normalize anything).
- **Rationale**: `.sh` files and `Dockerfile`s are interpreted by Linux shells inside containers — a CRLF there is a correctness bug, not a style issue. Python, Markdown, and YAML have no such hard requirement, but forcing LF keeps `git diff`/`git blame` clean across a Windows-primary, Linux-deployed project. `.bat`/`.ps1` are the inverse case: they only ever run on Windows, where CRLF is the native and expected convention, and forcing LF on them would be consistency for its own sake with no benefit.
- **Tradeoffs**: Contributors whose local Git config normalizes differently will see `.gitattributes` override their local behavior for tracked files — this is the intended effect (repository policy beats individual config) but can surprise a contributor unfamiliar with `.gitattributes` the first time they see `git add --renormalize` touch files they didn't expect.
- **Future Implications**: Any new source file type added to the project (e.g. `.json`, `.toml`, `.tsx` once the frontend milestone begins) should get an explicit `.gitattributes` entry at that time rather than falling back to the `* text=auto eol=lf` catch-all by accident — see `docs/KNOWN_ISSUES.md` for the tracked reminder.

## ADR-011 — Provider Capability Protocols Instead of One Uniform Interface

- **Date**: 2026-07-07
- **Status**: Accepted
- **Context**: Milestone 2 (Data Ingestion Pipeline) needs a Provider Layer supporting yfinance (prices, dividends, splits), CoinGecko (prices only — crypto has no dividends or splits), and FRED (economic indicator observations only — no prices at all).
- **Problem**: A single `Provider` interface requiring every adapter to implement `fetch_prices`/`fetch_dividends`/`fetch_splits`/`fetch_observations` would force CoinGecko and FRED to implement methods that make no domain sense for them (a `fetch_dividends` for crypto, a `fetch_prices` for an economic indicator), either raising `NotImplementedError` at call time or returning meaningless empty lists silently.
- **Options Considered**: (1) One large `Provider` protocol all adapters implement fully, unsupported methods raising `NotImplementedError`. (2) Separate capability protocols (`PriceProvider`, `DividendProvider`, `SplitProvider`, `IndicatorProvider`) that a provider implements only the subset it genuinely supports, with callers checking capability via `isinstance()` before calling.
- **Final Decision**: Option 2, implemented with `typing.Protocol` + `@runtime_checkable` (`app/ingestion/providers/base.py`).
- **Rationale**: Matches the actual shape of the domain — not every provider supports every capability, and pretending otherwise (via `NotImplementedError`) just moves the failure from "doesn't compile" to "fails at runtime in a way that looks like a bug." `isinstance()` capability checks let the orchestrator (and any future caller) ask "can this provider do X?" before attempting X, which is also exactly how `import_asset()`'s convenience wrapper decides whether to attempt dividend/split imports.
- **Tradeoffs**: Slightly more moving parts (four protocols instead of one) for a reader encountering the Provider Layer for the first time.
- **Future Implications**: A future provider (Polygon, Alpha Vantage, IEX) that supports a different subset of capabilities (e.g. prices + dividends but not splits) requires zero changes to existing providers or the orchestrator — it simply implements the protocols it can satisfy.

## ADR-012 — CoinGecko OHLC Approximation Is Disclosed, Never Fabricated

- **Date**: 2026-07-07
- **Status**: Accepted
- **Context**: CoinGecko's free-tier API endpoint that supports an arbitrary historical date range (`/market_chart/range`) returns one price observation per day, not true Open/High/Low/Close. The endpoint that does return real OHLC (`/ohlc`) is restricted to a fixed lookback window from "now" and cannot backfill arbitrary historical ranges — unusable for this platform's core use case (simulating decades-old investments).
- **Problem**: The `historical_prices` schema (M1) requires non-null Open/High/Low/Close for every row. Without true OHLC data available from the free API for historical ranges, some value has to go in those columns.
- **Options Considered**: (1) Silently set Open = High = Low = Close = the single observed price, with no indication anywhere that this differs from genuinely-observed OHLC. (2) Same substitution, but explicitly disclosed via a warning attached to every CoinGecko price import's Import Report, and documented prominently in code comments and this ADR.
- **Final Decision**: Option 2 (`COINGECKO_OHLC_APPROXIMATION_WARNING` in `coingecko_provider.py`, attached by the orchestrator to every CoinGecko price import report).
- **Rationale**: "Historical Truth Is Sacred" (Founder Specification, Part 2.1) does not forbid using an imperfect data source — it forbids *pretending* imperfect data is complete. Setting O=H=L=C is not fabrication in the sense of inventing a number that never existed (the single price point genuinely was CoinGecko's observation for that day); the fabrication risk would be in presenting it as if real intraday high/low variance had been captured when it wasn't. Disclosure closes that gap.
- **Tradeoffs**: Crypto price data ingested via CoinGecko has permanently lower fidelity than stock/ETF data via yfinance — any future feature computing intraday volatility from `high_price - low_price` would silently get zero for all CoinGecko-sourced rows. This is now a known, documented constraint rather than a silent trap.
- **Future Implications**: If a paid CoinGecko tier or an alternative crypto data provider with genuine historical OHLC becomes available, this adapter (or a replacement) should be updated — tracked in `docs/KNOWN_ISSUES.md`.

## ADR-013 — Per-Record SAVEPOINT in Storage Layer Upserts

- **Date**: 2026-07-07
- **Status**: Accepted
- **Context**: The Storage Layer's idempotent upsert (`INSERT ... ON CONFLICT DO NOTHING ... RETURNING`) can still raise a genuine constraint violation distinct from the anticipated natural-key duplicate (e.g. a foreign key pointing at a row that doesn't exist). Postgres aborts an entire transaction on the first unhandled statement error.
- **Problem**: Without isolating each record's insert attempt, one bad record partway through a multi-hundred-row import batch would silently discard every row already upserted earlier in the same transaction when the batch's outer transaction is later committed (or, if the error isn't caught at all, the whole transaction aborts).
- **Options Considered**: (1) Let a constraint violation propagate and abort the whole import's transaction. (2) Wrap each individual upsert in a SAVEPOINT (`session.begin_nested()`), so a failure rolls back only that one insert attempt, and the transaction as a whole remains valid and committable with everything upserted so far intact.
- **Final Decision**: Option 2 (`IngestionRepository._upsert`, `app/ingestion/storage/repository.py`).
- **Rationale**: A single malformed record (a genuine bug, not an ordinary duplicate) should be reported and skipped, not allowed to silently erase legitimate work already done in the same import run — directly serving "accuracy is more important than speed" and the general principle that partial success should be visible and preserved, not hidden by an all-or-nothing rollback.
- **Tradeoffs**: A SAVEPOINT per record adds minor overhead per insert; acceptable at MVP import volumes (per-asset historical backfills, not continuous high-frequency writes).
- **Future Implications**: Verified directly by `test_upsert_price_constraint_violation_raises_and_preserves_prior_rows` (`tests/ingestion/test_storage.py`) — any future change to `_upsert` should keep this test green, since it's the one thing standing between "one bad row" and "silently lose an entire import."

## ADR-014 — One Audit Log Row Per Real Import Attempt, No Separate "Start" Event

- **Date**: 2026-07-07
- **Status**: Accepted
- **Context**: The Founder Specification's audit requirements (as relayed in this milestone's brief) list "import start" and "import completion" as things to capture. `audit_logs.entity_id` (M1 schema) is NOT NULL, and dry runs must not write to the database at all.
- **Problem**: There is no real entity to attach a "start" audit event to before the asset/indicator row is resolved (for a brand-new symbol, that resolution is itself the first write of the import). Writing a start-of-import audit row would also directly conflict with the dry-run "must not modify the database" requirement, since dry runs need to log a start signal without writing anything.
- **Options Considered**: (1) Two audit rows per real import: one at start (against a synthetic, not-otherwise-referenced UUID) and one at completion. (2) One audit row per real import attempt, written at completion (success or failure), with the full structured Import Report (including `import_start`, `import_end`, `duration_seconds`) embedded in `details` — plus structured application logging (not persisted audit rows) for real-time start/progress visibility.
- **Final Decision**: Option 2.
- **Rationale**: The single completion row is a complete historical record — it contains the start timestamp and everything else needed for monitoring/debugging after the fact, without inventing a synthetic entity_id with no backing row (which would itself be an inconsistency with the platform's polymorphic-audit convention of `entity_id` pointing at something real). Application logs (see `app/core/logging.py`) cover the "is this import still running right now" concern that a persisted start-row would otherwise serve.
- **Tradeoffs**: There is no queryable database record of imports that are currently in-progress (only completed ones, successful or failed) — acceptable given MVP-scale imports complete in seconds to low minutes, not hours.
- **Future Implications**: If a future milestone introduces long-running or scheduled background imports where "is this stuck?" becomes an operational question, this design should be revisited — likely via a mutable in-progress audit row rather than the current write-once-at-completion shape. Tracked in `docs/KNOWN_ISSUES.md`.

## ADR-015 — Simulation Engine Calculation Model Built on `close_price`, Not `adjusted_close_price`

- **Date**: 2026-07-08
- **Status**: Accepted (implements Founder Decision 001 — see `docs/FOUNDER_DECISIONS.md`)
- **Context**: The Founder Specification's own text (Parts 2.6.7, 2.6.20, 2.6.22) recommends `adjusted_close_price` as the MVP default for the Simulation Engine, with raw price fields and `stock_splits` described as existing "for transparency and future advanced simulations." The founder has explicitly decided to build the Simulation Engine (M3) around the "future advanced" model instead, from the start.
- **Problem**: `adjusted_close_price` is a single number that bundles split adjustment and (per the specification's own description, "provider dependent") dividend adjustment together, opaquely. A calculation built on it cannot show its work — it cannot answer "how much of this return came from dividends versus price appreciation" without essentially re-deriving the raw event history anyway, undermining the auditability and explainability the Founder Specification names as top Simulation Engine priorities (Part 2.14.1).
- **Options Considered**: (1) Use `adjusted_close_price` as the specification's own text recommends for MVP, deferring explicit event-based modeling to a genuinely future milestone. (2) Use `close_price` with explicit, stored-event dividend reinvestment and split disclosure, per Founder Decision 001.
- **Final Decision**: Option 2, detailed in full in `docs/simulation_formulas.md`.
- **Rationale**: See Founder Decision 001. In engineering terms: an explicit event-processing model is directly testable against known-answer references (a specific dividend, on a specific date, at a specific share count) in a way an opaque adjusted-price calculation is not — this materially serves Part 2.14.18's requirement that the Simulation Engine "requires the highest test coverage in the platform."
- **Tradeoffs**: More implementation complexity in M3 (a per-event loop rather than a single price-ratio lookup); one unverified empirical assumption (raw `close_price` is already split-consistent within a single ingestion fetch) that must be confirmed by a known-answer test against a real historical split before this model is trusted in production — tracked as a blocking pre-launch item, not an optional nice-to-have.
- **Future Implications**: A future "Advanced Corporate Actions Engine" (spin-offs, mergers, special dividends, incremental-reimport-safe split reconciliation) is explicitly reserved as a later milestone, not part of M3. Any new source file or calculation added to the Simulation Engine must continue to read `close_price`, never `adjusted_close_price`, unless a new Founder Decision supersedes this one.

## ADR-016 — API Layer: Service-Owned Transaction Boundary, Engine-Computed Growth Series, Savepoint-Nested Test Sessions

- **Date**: 2026-07-10
- **Status**: Accepted
- **Context**: M4 adds the first real caller of the M3 Simulation Engine (`POST /api/v1/simulations`) and the first real consumer of a database `Session` that outlives a single function call (a FastAPI request). Three interlocking design questions had to be resolved together: who commits/rolls back the transaction, where the Founder-Specification-required "Growth Chart" output (Part 3.3.2, tracked as KI-021) gets computed, and how DB-integration tests exercise code that itself calls `session.commit()`.
- **Problem**: (1) `app.simulation.engine.run_simulation` only `flush()`es by design (established in M3: the caller owns the transaction) — for `MissingHistoricalDataError`/`CalculationError`, it flushes a failed `Simulation` row *before* re-raising, per Founder Specification Part 2.6.24's "failed simulations should be stored" requirement. If the API layer's DB-session dependency auto-rolled-back on any exception (the common FastAPI pattern), that already-flushed failed row would be silently destroyed on its way out. (2) Part 3.3.2 requires a value-over-time series; nothing in the API layer should perform financial calculation (approved decision #2), so it cannot be computed in a route handler. (3) The project's established DB-integration test pattern (`tests/simulation/conftest.py`, `tests/ingestion/conftest.py`) binds a `Session` to a single connection wrapped in one outer transaction, rolled back at the end of the test — but that pattern assumes the code under test never itself calls `commit()`/`rollback()`; `app.api.v1.services.simulation_service` does, by design, per (1).
- **Options Considered**: (1a) Give `get_db_session` FastAPI's common auto-commit-on-success/auto-rollback-on-exception behavior. (1b) Make `get_db_session` commit/rollback-agnostic (only `close()`s) and have the service layer explicitly own the boundary per exception type. (2a) Recompute the growth series in the route handler from already-serialized response data. (2b) Extend the Simulation Engine itself with a read-only `calculate_growth_series` formula and thread it through `SimulationOutcome`. (3a) Accept the `SAWarning: transaction already deassociated from connection` noise in test output as harmless. (3b) Bind test sessions with SQLAlchemy 2.0's `join_transaction_mode="create_savepoint"`, so a service-layer `commit()`/`rollback()` operates on a nested SAVEPOINT instead of ending the outer transaction outright.
- **Final Decision**: (1b), (2b), (3b) — implemented in `app/api/v1/dependencies.py::get_db_session`, `app/api/v1/services/simulation_service.py::create_simulation`, `app/simulation/formulas.py::calculate_growth_series` + `app/simulation/engine.py`, and `tests/api/conftest.py::db_session` respectively.
- **Rationale**: (1b) makes the commit point an explicit, auditable decision per exception type (pre-flight validation errors roll back since nothing was written; `MissingHistoricalDataError`/`CalculationError` commit since the engine already flushed a row worth preserving) rather than an implicit framework default that would silently violate Part 2.6.24 for exactly the error paths that matter most. (2b) keeps "no financial calculation logic in the API layer" a structural guarantee, not a code-review convention — `calculate_growth_series` is exercised by the same known-answer/cross-check test discipline as every other M3 formula (`tests/simulation/test_growth_series.py`), including a test proving its final data point exactly matches the independently-computed `apply_dividend_reinvestment` + `calculate_final_value` result. (3b) is the standard SQLAlchemy 2.0 pattern for testing code that itself manages transactions, and was confirmed necessary empirically: without it, the fixture's own final `transaction.rollback()` raised a real (if non-fatal) `SAWarning` for every test whose request path called `session.rollback()`/`session.commit()`.
- **Tradeoffs**: (1b) pushes transaction-boundary responsibility onto every future service function that writes to the database — each must know and handle its own commit/rollback, rather than getting it for free from the dependency. This is a deliberate cost accepted for correctness around the one existing case where it matters (failed-simulation persistence); documented in `simulation_service`'s module docstring so it isn't rediscovered by surprise. (2b) means `growth_series`/`disclosed_splits` are never persisted (no `simulations` columns exist for them) — a `GET /api/v1/simulations/{id}` call made after the initial `POST` cannot reconstruct them, so `SimulationResponse.from_simulation()` defaults both to empty tuples on the retrieval path. This is the founder-approved fallback ("if this is too large for M4, expose growth_series as an empty/deferred field and document the deferral") and is tracked, not silently accepted, at KI-021.
- **Future Implications**: If a future milestone persists `growth_series`/`disclosed_splits` (e.g., a dedicated `simulation_growth_points` table), `SimulationResponse.from_simulation()`'s retrieval-path default changes from "always empty" to "read from storage," with no route-handler-level calculation change required. Any new API-layer service function that writes to the database must follow the same explicit commit/rollback-per-exception-type pattern established here, not assume a framework-level auto-commit.

## ADR-017 — Refresh Tokens: Opaque + Hashed, Not a Second JWT, With Rotation-Chain Reuse Detection

- **Date**: 2026-07-11
- **Status**: Accepted (implements Founder Decision 002)
- **Context**: M5 needs a refresh mechanism (Founder Decision 002) that supports real server-side revocation — a bare stateless JWT refresh token cannot be revoked before its own expiry without an external denylist, which is itself a stateful store in disguise.
- **Problem**: Choose a refresh-token representation and storage design that supports revocation, rotation, and (designed-for-but-not-yet-built) multi-device session management, without over-building for M5's actual scope.
- **Options Considered**: (1) A second, longer-lived JWT as the refresh token, validated purely by signature, no DB row. (2) An opaque random value, hashed (SHA-256) and stored in a new `refresh_tokens` table, with `revoked_at`/`replaced_by_id` columns supporting rotation and reuse detection.
- **Final Decision**: Option 2 (`app/models/refresh_token.py`, migration `0002_refresh_tokens`).
- **Rationale**: A DB-backed opaque token makes "is this session still valid?" a real, queryable, revocable fact rather than a signature check against a token that remains valid until expiry no matter what the server later learns (e.g., a detected compromise). Storing only the SHA-256 hash (never the raw value) means a database read or leak alone can never be replayed as a working credential — the same discipline already applied to `users.password_hash`. `replaced_by_id` (a self-referential FK) chains each rotation, which is what makes reuse detection possible: presenting an already-revoked token that was itself the result of a prior rotation is a strong theft signal, not just an expired-token case.
- **Tradeoffs**: Every refresh call requires a DB round-trip (an `INSERT` + an `UPDATE`), unlike a pure-JWT refresh token's signature-only check. Accepted: refresh calls are infrequent relative to access-token-validated requests (once per ~15 minutes per active session at most), and this project's engineering priority order (Correctness > Security > Reliability > Maintainability > Performance) puts revocability ahead of that marginal cost.
- **Future Implications**: `user_agent`/`ip_address` columns and the lack of a uniqueness constraint on `user_id` (multiple concurrent rows per user are fully supported) exist now specifically so a future "manage your devices" feature (list active sessions, revoke one, revoke all) requires zero schema migration — only new routes calling `AuthRepository.revoke_all_for_user` (already implemented, unused by any route today) and a new "list active sessions" query. See docs/KNOWN_ISSUES.md for the tracked concurrent-refresh race this design's rotation step has not yet been hardened against.

## ADR-018 — Session Delivery via httpOnly/Secure/SameSite=Strict Cookies, Not Response Body or `localStorage`

- **Date**: 2026-07-11
- **Status**: Accepted (implements Founder Decision 002)
- **Context**: Neither the Founder Specification nor `.claude/API_STANDARDS.md` specifies where the JWT named in Part 2.15.3 should live on the client. `.claude/SECURITY_POLICY.md` already recommended httpOnly, `SameSite=strict` cookies as the safer default "given the frontend is same-origin-deployable via Vercel + a custom domain" — this ADR is the M5 implementation of that recommendation.
- **Problem**: A token readable by JavaScript (a JSON response body consumed into `localStorage` or a JS-visible cookie) can be exfiltrated by any successful XSS payload, turning a single unrelated frontend bug into a full session-theft vulnerability.
- **Options Considered**: (1) Return both tokens in the JSON response body; frontend stores them in `localStorage` and attaches the access token via an `Authorization` header. (2) Return both tokens exclusively as httpOnly, Secure, `SameSite=Strict` cookies, set directly by the API responses (`register`/`login`/`refresh`); nothing token-shaped ever appears in a response body.
- **Final Decision**: Option 2 (`app/api/v1/routers/auth.py`).
- **Rationale**: httpOnly means no JavaScript, including an XSS payload, can ever read either cookie's value — this closes off the single highest-impact token-theft vector for a browser-based frontend. `SameSite=Strict` means the cookies are never attached to a cross-site request at all, which is this project's primary CSRF mitigation (a dedicated CSRF token was considered and deliberately not added for M5 — SameSite=Strict is judged sufficient given this is a same-origin-deployable, non-public-API platform; tracked as a documented, not silent, simplification). The refresh-token cookie is additionally scoped to the `/api/v1/auth` path only, so it is never even transmitted to, or exposed by, any other endpoint.
- **Tradeoffs**: Cookie-based sessions require `allow_credentials=True` CORS configuration (already set since M4) and make the API awkward for a hypothetical future non-browser API client (a mobile app, a third-party integration) that can't rely on a cookie jar — accepted, since this platform's only client is its own same-origin web frontend for the foreseeable roadmap (Part 2.16/2.25).
- **Future Implications**: If a non-browser client is ever needed, it would require a parallel `Authorization`-header-based token issuance path — not a retrofit of this one, since mixing the two schemes on the same cookie-trusting routes would reopen the CSRF surface SameSite=Strict currently closes.

## ADR-019 — Account Lockout as a Distinct, Email-Keyed Mechanism From IP-Based Rate Limiting

- **Date**: 2026-07-11
- **Status**: Accepted (implements Founder Decision 002)
- **Context**: Founder Specification Part 3.6.7 (Credential Stuffing) names "Account lockout policies" as a required mitigation, distinct from and additional to the existing rate-limiting requirement (Part 2.8.13). M4 already has a per-IP `RateLimiter` (`app/core/rate_limit.py`).
- **Problem**: An attacker distributing password-guessing attempts against one target account across many source IPs (or via a botnet/proxy pool) defeats a purely per-IP rate limit entirely, while never tripping any single IP's threshold.
- **Options Considered**: (1) Rely solely on the existing per-IP `RateLimiter`, applied to the new `/auth` endpoints at the spec's 10/min figure. (2) Add a second, independent mechanism (`AccountLockout`, `app/auth/lockout.py`) keyed by the normalized email address being attempted, locking the account after a fixed number of failures regardless of source IP.
- **Final Decision**: Both — Option 2 in addition to, not instead of, Option 1.
- **Rationale**: The two mechanisms defend against different attacker shapes: per-IP rate limiting throttles a single source hammering many endpoints or accounts; per-account lockout stops a distributed attack against one specific account that per-IP limiting cannot see. This is exactly why the Founder Specification lists both as separate line items in its threat mitigation table rather than treating rate limiting as sufficient on its own.
- **Tradeoffs**: A per-account lockout is itself a potential denial-of-service vector against a specific victim (an attacker who knows a target's email can lock them out by deliberately failing 5 logins) — accepted as a known, standard tradeoff of any lockout policy; mitigated in part by a bounded, time-limited window (15 minutes) rather than a permanent lock requiring manual admin intervention.
- **Future Implications**: Both mechanisms share the same Redis instance and fail-open policy on Redis unreachability (an outage of this dependency must never itself become the reason a legitimate user can't log in) — any future third rate-governing mechanism should follow the same fail-open convention rather than introduce a new failure philosophy.

## ADR-020 — Reject the Default `JWT_SECRET` Placeholder Outside Development at Startup

- **Date**: 2026-07-11
- **Status**: Accepted
- **Context**: `Settings.jwt_secret` needed a default value so local development works with zero required setup, matching the existing convention for `database_url` (which defaults to `itm_password`). Discovered during the M5 red-team self-review, not requested by any design document.
- **Problem**: Unlike a wrong database password (which fails loudly — the app simply can't connect), a forgotten `JWT_SECRET` in a real deployment is silently catastrophic: the signing key becomes public knowledge (it's committed in `.env.example`/source), letting anyone forge a valid access token for any user, including `is_admin: true`, with no failed request or error anywhere to reveal the problem.
- **Options Considered**: (1) Ship the placeholder default with a code comment warning not to use it in production, trusting deployment discipline. (2) Add a `pydantic` `model_validator` on `Settings` that raises at startup if `environment` is not `development`/`test`/`testing` and `jwt_secret` still equals the placeholder.
- **Final Decision**: Option 2 (`app/core/config.py::_reject_default_jwt_secret_outside_development`).
- **Rationale**: A comment is advice; a startup failure is a guarantee. This mirrors the general principle that a security-critical misconfiguration should fail the deployment immediately and obviously, not silently degrade the platform's actual security posture in a way that produces no error anyone would ever see until it's exploited.
- **Tradeoffs**: None material — the guard only fires when `environment` is deliberately set to something other than the three explicitly-exempted values, which any real deployment must already do correctly for other reasons (e.g. `.claude/PERFORMANCE_BUDGET.md`'s environment-specific behavior).
- **Future Implications**: Any future secret with a development-only placeholder default (were one ever introduced) should get the same startup-time guard rather than relying on documentation alone.

## ADR-021 — AI Provider Abstraction: Protocol-Based, Anthropic First, `NullProvider` Fallback

- **Date**: 2026-07-12
- **Status**: Accepted (implements Founder Decision 003)
- **Context**: Founder Specification Part 2.7.15 requires the AI layer to "communicate through an abstraction layer rather than directly depending on a specific vendor," naming OpenAI, Anthropic, Google, and Local Models as candidates without selecting one. This project already has a directly analogous precedent: `app/ingestion/providers/base.py`'s capability-`Protocol` design (ADR-013), adopted specifically because not every provider supports every capability.
- **Problem**: A single hard-coded call to a vendor SDK anywhere in router or service code would violate Founder Specification Principle 3 (100% functional with AI removed) the moment that code ran without a configured key, and would make switching or adding a provider a cross-cutting change rather than a one-file one.
- **Options Considered**: (1) Call the Anthropic SDK directly from `explanation_service.py`. (2) Define a single `AIProvider` Protocol (`app/ai/providers/base.py`) with one `generate()` method, implement `AnthropicProvider` and a `NullProvider`, and select between them via a `Settings.ai_provider` factory (`app/ai/providers/get_ai_provider`).
- **Final Decision**: Option 2 — reusing the exact Protocol-based shape ADR-013 already validated for the ingestion providers.
- **Rationale**: `NullProvider` (selected by default, `AI_PROVIDER=none`) makes "the platform must remain 100% functional with every AI component removed" (Principle 3) literally true rather than aspirational — no network call, no API key, and every caller already has a working failure path for a provider that cannot generate. No module outside `app/ai/providers/anthropic_provider.py` imports the `anthropic` package, so adding a second vendor later touches one new file, not the router or service layer.
- **Tradeoffs**: The `Protocol`'s single `generate(system_prompt, user_content, max_tokens) -> ProviderResult` method is deliberately minimal (no streaming, no function/tool calling) — sufficient for M6's synchronous, single-turn explanation/follow-up generation, but would need extension for a future streaming-response or agentic-tool-use feature (neither in scope for M6, per Founder Decision 003).
- **Future Implications**: A second provider (OpenAI, Google, a local model) is a new file implementing the same `Protocol` plus one new branch in the factory — no change to `app/ai/service.py`, `app/api/v1/services/explanation_service.py`, or any router.

## ADR-022 — Explanation/Follow-Up Caching Matches on (simulation, prompt_version, model_name[, question_text]), Not Filtered to `COMPLETED` for the Explanation Engine

- **Date**: 2026-07-12
- **Status**: Accepted (implements Founder Decision 003)
- **Context**: The M6 founder decision requires that an unchanged `simulation_id` + `prompt_version` + model never re-spends a model call — "return the stored explanation instead of calling the model again." The Explanation Engine exposes an explicit `regenerate` flag for a deliberate retry; the Financial Tutor's follow-up-question endpoint does not.
- **Problem**: A cache lookup filtered to `generation_status == COMPLETED` only would silently re-attempt generation on every single non-regenerate call after any failure (provider outage, integrity violation) — for the Explanation Engine, this defeats the entire purpose of a regeneration cap, since every "free" non-regenerate call would keep spending a real model call as long as the prior attempt failed. For the Financial Tutor, the opposite problem exists: a follow-up question has no `regenerate` override at all, so caching a `FAILED` answer against that exact question text forever would create a permanent dead end for a transient provider hiccup.
- **Options Considered**: (1) Cache-match only `COMPLETED` rows for both features. (2) Cache-match regardless of status for both features. (3) Match regardless of status for the Explanation Engine (which has an explicit `regenerate` escape hatch), but `COMPLETED`-only for the Financial Tutor (which has none).
- **Final Decision**: Option 3 — `_find_cached(..., only_completed: bool)` in `app/api/v1/services/explanation_service.py`, `False` (the default) for the Explanation Engine, `True` for the Financial Tutor.
- **Rationale**: The regeneration cap, not the cache filter, is what should bound a retry-after-failure loop for the Explanation Engine — matching cache to "any status" makes the cap's count meaningful (an unbounded number of free non-regenerate calls would otherwise never touch the model at all after the first failure, silently defeating cost-control intent). The Financial Tutor has no equivalent override, so it needs the opposite default to remain usable after a transient failure; its own per-simulation follow-up cap (not the cache filter) is what bounds *its* cost exposure instead.
- **Tradeoffs**: The two features now have asymmetric caching semantics, which is a genuine, if narrow, inconsistency a future maintainer must understand rather than assume uniform — documented directly in `_find_cached`'s docstring for exactly this reason.
- **Future Implications**: If a future milestone adds an explicit "regenerate this follow-up answer" action, its cache lookup should switch to `only_completed=False` to match the Explanation Engine's semantics, consistent with the pattern established here (an explicit retry action pairs with any-status caching; no retry action pairs with completed-only caching).

## ADR-023 — Educational Disclaimer Is Code-Appended, Never Model-Generated

- **Date**: 2026-07-12
- **Status**: Accepted (implements Founder Decision 003)
- **Context**: The M6 approved output structure lists "Educational Disclaimer" as the seventh required section, alongside six others (Summary, What Happened, Why It Happened, Financial Concepts, Key Takeaways, Limitations) that are naturally AI-generated content.
- **Problem**: Asking the model to generate the disclaimer text itself means its presence depends on the model's compliance with a prompt instruction — a request, not a guarantee. A disclaimer is exactly the kind of compliance-relevant content where "the model usually includes it" is a materially weaker property than "the platform always includes it."
- **Options Considered**: (1) Instruct the model to produce all seven sections, including the disclaimer, and verify its presence with the same structure check used for the other six. (2) Treat the disclaimer as a fixed, version-controlled string (`app.ai.service.EDUCATIONAL_DISCLAIMER`) that application code appends after every successful generation, and explicitly instruct the model *not* to write one itself (to avoid a duplicate).
- **Final Decision**: Option 2.
- **Rationale**: A fixed string is deterministic, cannot be worded inconsistently across generations, cannot be omitted by an off-distribution model response, and needs no safety-check coverage of its own (the six-section structure check in `app.ai.safety.check_output_structure` deliberately excludes it, per that module's own docstring). This is one place M6's implementation goes beyond the letter of the approved instruction, for a documented, narrow safety reason — not a silent deviation.
- **Tradeoffs**: None material — the disclaimer's wording cannot vary per-simulation (e.g., it cannot reference the specific asset or figures involved) since it is not generated per-request; judged acceptable since a disclaimer's purpose is a fixed compliance statement, not simulation-specific content.
- **Future Implications**: Any future addition to the required output structure that is itself a fixed, compliance-relevant statement (not simulation-specific content) should default to this same code-appended pattern rather than a prompt instruction.

## ADR-024 — `ai_explanations` Extended with `explanation_type`/`question_text` Rather Than a New Table

- **Date**: 2026-07-12
- **Status**: Accepted (implements Founder Decision 003)
- **Context**: The Financial Tutor's follow-up question-and-answer needs to be persisted, audited, and cached the same way the Explanation Engine's initial explanation already is. `.claude/DATABASE_RULES.md` prohibits adding a new table/domain beyond the nine listed without approval, but has an established precedent for extending an existing domain instead (`refresh_tokens` extending Users at M5, ADR-017).
- **Problem**: A follow-up answer is structurally very close to an initial explanation (same `simulation_id` FK, same `model_name`/`prompt_version`/`input_summary`/`explanation_text`/`generation_status`/`error_message` shape) but additionally needs to record which question was asked, and the two need to be distinguishable for caching, capping, and listing.
- **Options Considered**: (1) A new `ai_followup_questions` table, duplicating most of `ai_explanations`' columns. (2) Extend `ai_explanations` with `explanation_type` (`initial` | `follow_up`) and a nullable `question_text`, plus a composite index on `(simulation_id, explanation_type)` supporting the new cache/cap lookup pattern.
- **Final Decision**: Option 2 (migration `0003_ai_explanation_type`).
- **Rationale**: The two record types share every column except one (`question_text`, already nullable-by-necessity the same way `explanation_text` is), and `ai_explanations` already supports "multiple explanations per simulation" (Founder Specification Part 2.6.26) — a follow-up answer is not a new logical domain, it is another row in the domain that already exists. This mirrors the exact reasoning ADR-017 used for `refresh_tokens` extending Users rather than standing alone.
- **Tradeoffs**: `ai_explanations` now serves two logically distinct purposes from one table, requiring every query to filter by `explanation_type` explicitly (enforced by the new composite index, not by a separate physical table boundary) — a minor query-ergonomics cost, accepted the same way ADR-008 accepted a comparable tradeoff for Economic Indicators' two-table split in the opposite direction.
- **Future Implications**: A hypothetical third AI-generated content type (e.g., a full multi-simulation comparison report, if Report Generation is ever built per Part 3.3.12) should evaluate this same question fresh — this ADR is not a blanket license to keep extending `ai_explanations` indefinitely, only a specific, justified precedent for the two types M6 actually needs.

## ADR-025 — Frontend Design Token Architecture: CSS-Variable Three-Layer System Bridged into Tailwind v4, Plus a New Z-Index Scale

- **Date**: 2026-07-15
- **Status**: Accepted
- **Context**: M7 Phase 0 (`docs/BRAND_CONSTITUTION.md`, `docs/frontend_design_system.md`) approved a primitive → semantic → component token architecture and a specific color/type/spacing/radius/elevation/motion/breakpoint system, but M7 Phase 1 is the first milestone to actually scaffold the frontend — `create-next-app` provisioned Next.js 16 with Tailwind v4, which removed the `tailwind.config.ts` JavaScript config file in favor of a CSS-first `@theme` directive, a genuine framework change from the config-file approach `docs/frontend_design_system.md` implicitly assumed when it was written.
- **Problem**: Tailwind v4's `@theme` values are normally static at build time, which is the wrong fit for a token system that must switch every color at runtime for light/dark (FD-005) without a page reload or a `dark:` variant sprinkled onto every utility class.
- **Options Considered**: (1) Use `@theme` directly with hard light-mode values and `dark:` variants on every component — matches Tailwind v4's own quickstart docs but reintroduces exactly the per-component dark-mode duplication the design system's token architecture exists to avoid. (2) Three physical CSS files (`tokens/primitives.css`, `tokens/semantic.css`, `tokens/components.css`) holding real runtime CSS custom properties, switched via a `[data-theme]` attribute selector, then a thin `@theme inline { --color-x: var(--color-x); }` bridge layer that lets Tailwind generate ordinary utility classes (`bg-background`, `text-ink-primary`, ...) whose actual value still resolves at paint time from the switchable variable.
- **Final Decision**: Option 2, implemented in `frontend/src/styles/tokens/{primitives,semantic,components}.css` and bridged in `frontend/src/app/globals.css`'s `@theme inline` block. Also introduces a z-index scale (`--z-dropdown` through `--z-tooltip`, six tiers) as a new primitive token — not previously specified in any Phase 0 document — since Tailwind v4 does not ship a themeable z-index namespace and the design system was silent on stacking order.
- **Rationale**: This is the standard pattern the shadcn/ui v4 template itself converged on for the identical problem (CSS vars for the switchable layer, `@theme inline` purely as a Tailwind-utility-generation bridge), so it is a well-precedented solution rather than a bespoke one. It also means every component only ever writes semantic utility class names (`bg-surface`, `border-border-hairline`) and never a `dark:` variant, which is the concrete implementation of the design system's own "components reference semantic tokens only" rule (`docs/BRAND_CONSTITUTION.md` component checklist item 1).
- **Tradeoffs**: Two color-utility names read slightly awkwardly as a direct consequence of the naming convention (`border-border-hairline`, `text-ink-primary`) — accepted rather than shortening the semantic names, since a longer-but-unambiguous utility name is preferable to a token-naming scheme that drifts from `docs/frontend_design_system.md`'s own vocabulary. The z-index scale is a genuinely new, unreviewed addition; it should be treated as provisional until a real overlay/modal/toast stacking scenario is built in Phase 2 and can validate the six tiers are sufficient.
- **Future Implications**: Any new semantic color added in a future phase must be added to all three token files (or explicitly deferred) plus the `@theme inline` bridge — omitting the bridge step silently produces a CSS variable that has no corresponding Tailwind utility, a class of bug worth a lint/review checklist item in Phase 2.

## ADR-026 — Theme Switching: `data-theme` Attribute + Inline Head Script, Not a Third-Party Theme Library

- **Date**: 2026-07-15
- **Status**: Accepted
- **Context**: FD-005 requires light/dark theme architecture from day one, with no flash of the wrong theme on load and no visible toggle mandated yet. The obvious off-the-shelf solution (`next-themes`) is small and well-tested, but this Next.js version (16.2) ships its own documented flash-prevention pattern (bundled guide: "Preventing Flash," `node_modules/next/dist/docs/01-app/02-guides/preventing-flash-before-hydration.md`) built specifically around this exact scenario.
- **Problem**: Server-rendered HTML has no access to `localStorage` or the client's OS theme preference; without correcting the DOM before first paint, a user with a stored or system dark preference sees a flash of light content on every load.
- **Options Considered**: (1) `next-themes` — a small, well-known dependency that handles this internally. (2) Hand-roll the exact pattern this Next.js version's own docs recommend: a synchronous inline `<script>` in `<head>` that reads `localStorage` (falling back to `prefers-color-scheme`) and sets `data-theme` on `<html>` before the browser paints, paired with a React `ThemeProvider`/`useTheme()` context whose lazy `useState` initializer reads the same source so client and script never disagree.
- **Final Decision**: Option 2 — `frontend/src/providers/theme-script.ts` (the inline script string) and `frontend/src/providers/theme-provider.tsx` (the context/hook), with `<html data-theme="light" suppressHydrationWarning>` as the SSR fallback default in `frontend/src/app/layout.tsx`.
- **Rationale**: Avoids a dependency for a problem this specific Next.js version already documents a first-party solution for, and keeps the implementation legible to any future contributor reading this version's own bundled guide. `useTheme()` is exported now so a future toggle control is a one-line `setTheme()` call, not a redesign — deliberately satisfying `docs/BRAND_CONSTITUTION.md`'s "no toggle unless it naturally falls out of the architecture" instruction without leaving the capability unbuilt.
- **Tradeoffs**: A hand-rolled solution carries slightly more maintenance surface than a maintained library, and re-implements a small amount of logic (system-preference listening, storage sync) that `next-themes` provides for free. Accepted because the surface is genuinely small (two files, under 90 lines combined) and the behavior is fully covered by `frontend/src/__tests__/lib/theme-provider.test.tsx`.
- **Future Implications**: If a future phase needs SSR-aware theme-dependent server rendering (e.g., a server component branching on theme), this client-only approach will need revisiting — not a concern for Phase 1, since no server component currently reads theme.

## ADR-027 — Centralized API Client: One Axios Instance, One Error-Normalization Function, One Error-Code-to-Copy Table

- **Date**: 2026-07-15
- **Status**: Accepted
- **Context**: `docs/frontend_design_system.md`'s shared-component list calls for "an API client hook that maps the `{success, data}` / `{success: false, error: {code, message, request_id}}` envelope... to one central error-code→copy table" (`docs/api_design.md`). `.claude/CODING_STANDARDS.md` requires the frontend to remain presentation-only and never duplicate backend business logic.
- **Problem**: Without one enforced boundary, every future screen would be tempted to branch on `response.data.success` itself and invent its own error copy, silently duplicating logic and producing inconsistent error messages for the same backend error code across different screens.
- **Options Considered**: (1) A thin wrapper per endpoint that each does its own success/failure branching. (2) One `axios` instance (`withCredentials: true` for the httpOnly session cookies Founder Decision 002 established) with a response interceptor that unwraps `{success:true,data}` and converts every failure — a backend error envelope or a transport-level failure with no response at all — into one typed `ApiError` class, plus a single `ERROR_COPY` record covering every documented code.
- **Final Decision**: Option 2, in `frontend/src/lib/api/{client,errors}.ts`. The interceptor's error-normalization step is factored into an exported pure function (`normalizeApiError`) specifically so it is unit-testable against constructed error shapes without depending on axios's transport/`validateStatus` internals inside a test.
- **Rationale**: Every endpoint function (`frontend/src/lib/api/endpoints/*.ts`) can now simply `await apiRequest<T>(...)` and either get typed data back or have a single `ApiError` thrown — no screen ever needs to know the envelope shape exists. This is the direct implementation of "the frontend only calls the versioned API" and "never duplicate backend business logic": the only place that understands the API contract's shape is this one module.
- **Tradeoffs**: `ERROR_COPY` deliberately does not include an entry used for AI-unavailable states — that is a normal successful response per Founder Decision 003, not an error code, and routing it through this table would be a category error; this is called out explicitly in `errors.ts`'s own comment so a future contributor doesn't "helpfully" add one.
- **Future Implications**: Adding a new backend error code (`docs/api_design.md`) requires adding it to the `ApiErrorCode` union (`frontend/src/types/api.ts`) and `ERROR_COPY` in the same change — TypeScript's exhaustiveness checking on the `Record<ApiErrorCode, ErrorCopy>` type will fail the build if a code is ever added to one but not the other.

## ADR-028 — Status and Muted-Ink Colors Split Light/Dark, Fixing a Real Contrast Bug (and a Self-Reference Bug) in the M7 Phase 1 Tokens

- **Date**: 2026-07-16
- **Status**: Accepted
- **Context**: M7 Phase 1.5's hardening review computed real WCAG contrast ratios for every ink/status color against its background, rather than assuming the Phase 0/0B design-review hex values were correct as shipped. `docs/frontend_design_system.md`'s status palette (§3) and the muted-ink neutral were both designed as a single hex shared by light and dark mode — unlike the chart-data palette and every other neutral, which already had separate light/dark values.
- **Problem**: Two distinct, real bugs were found, not one. (1) **Contrast**: a single hex cannot pass 4.5:1 against both a near-white and a near-black background at once — computed contrast ratios showed muted-ink at 3.41:1 (fail) on light, status-good at 3.18:1 (fail) on light, status-warning at 1.74:1 (fail badly) on light, status-serious at 2.50:1 (fail) on light, and status-critical at 4.05:1 (fail) on dark; only status-critical-on-light (4.56:1) happened to pass, and only barely. (2) **Self-reference**: `frontend/src/styles/tokens/semantic.css`'s original `:root` block declared `--color-status-good: var(--color-status-good)` (and the same pattern for warning/serious/critical) — a custom property referencing its own name, which is invalid per the CSS Custom Properties spec and causes the declaration using it (`color: var(--color-status-good)` in `badge.tsx`) to fall back to an inherited/initial color instead. The dark-mode override blocks additionally never set `--color-status-*` at all, meaning dark mode was equally affected. Confirmed via the actual compiled production CSS (`.next/static/chunks/*.css`), not just source inspection.
- **Options Considered**: (1) Leave the hues as a single value and only fix the self-reference bug, accepting the contrast failures as a known, documented gap. (2) Split every status color and the muted-ink neutral into light/dark primitive pairs — computing new, verified-safe hex values for each failing case — matching the pattern the chart-data and other neutral tokens already correctly used, and fix the self-reference in the same change.
- **Final Decision**: Option 2. New primitives: `--color-status-{good,warning,serious,critical}-{light,dark}` and `--color-ink-muted-{light,dark}` (`frontend/src/styles/tokens/primitives.css`), referenced correctly (no self-reference) per theme in `semantic.css`. New verified hex values: good `#0b7d0b`/`#0ca30c`, warning `#8a5a00`/`#fab219`, serious `#a8451f`/`#ec835a`, critical `#c23434`/`#e8605f`, muted `#6b6963`/`#898781` (light/dark respectively) — dark-mode values were already safe for good/warning/serious/muted and were kept unchanged; only critical needed a new, lighter dark-mode value.
- **Rationale**: Every new hex was chosen by solving for the relative-luminance threshold needed to hit ≥4.5:1 against its theme's background (`#f9f9f7` light / `#0d0d0d` dark), then verified programmatically, not eyeballed — the same values are asserted directly in `frontend/src/__tests__/lib/contrast.test.ts` as a permanent regression guard (a known-answer test, mirroring this project's own backend testing philosophy applied to a design token). Keeping the hue identity (still recognizably green/amber/orange-red/red) while shifting the exact shade per theme is the same tradeoff the chart-data palette already made for the identical reason, so this isn't a new design pattern, just a gap in Phase 1's original application of it.
- **Tradeoffs**: None of the shifted hex values change any prior product decision or Founder Decision — this is a pure bug fix within an already-approved design direction (BRAND_CONSTITUTION.md §7's status-palette *philosophy* — "functional, never celebratory" — is unaffected; only the literal hex values are corrected). `docs/frontend_design_system.md` §3 was updated to match, since that document is meant to reflect implementation truth, not a stale proposal.
- **Future Implications**: Any future new status/semantic color must be verified against both theme backgrounds with the same known-answer-test discipline before shipping — `contrast.test.ts` is the place to add it. Any change to `--color-status-*`'s dark-mode selectors must set all four status variables explicitly in every theme block (`:root`, the `prefers-color-scheme` media block, and `[data-theme='dark']`) — the missing-override half of this bug would have recurred silently if a future contributor added a fifth status color and forgot one of the three blocks.

## ADR-029 — Financial-Math Guardrail: Branded `DecimalString` Type + ESLint Rule Banning Numeric Coercion in Product Code

- **Date**: 2026-07-16
- **Status**: Accepted
- **Context**: `.claude/CODING_STANDARDS.md` states "the frontend is presentation-only: it must never calculate returns, CAGR, dividend reinvestment, or inflation adjustment" as a rule, but M7 Phase 1 enforced it only by convention (every financial field was typed `string`, indistinguishable at the type level from any other string, and nothing prevented a future component from writing `Number(sim.final_value) - Number(sim.investment_amount)` inline). M7 Phase 1.5's hardening review was asked to add a practical guardrail, not just restate the rule in a comment.
- **Problem**: A documented rule with no structural enforcement is exactly the kind of thing that survives a design review intact and then gets silently violated eight screens later, once whoever is under deadline pressure to build a Results page reaches for the obvious (and wrong) `Number(...)` call because nothing stops them and the type checker has no opinion.
- **Options Considered**: (1) Document the rule more prominently and rely on code review. (2) A branded `DecimalString` type (`frontend/src/lib/format/decimal-string.ts`) marking every backend-sourced financial field as distinct from a plain `string` at the type level, plus a real ESLint rule (`no-restricted-syntax`, a core ESLint rule requiring no new dependency) banning `Number(`, `parseFloat(`, `parseInt(`, and unary `+` numeric coercion in `src/app/**` and `src/components/**` (product/UI code), plus a static-analysis test scanning the formatting module's own source for the same banned tokens. (3) A custom type-aware ESLint rule (using `@typescript-eslint`'s type-checking API) that specifically flags numeric coercion only when the operand's type is `DecimalString` — more precise than option 2, but requires building and maintaining a custom lint rule package, a meaningfully larger undertaking than justified for this phase.
- **Final Decision**: Option 2. Verified working against a deliberate scratch violation (four coercion patterns, all four caught) before being treated as done, then confirmed the real, already-written codebase passes cleanly with zero changes needed.
- **Rationale**: The branded type and the lint rule are complementary, not redundant: the type makes a financial value visually and structurally distinct from a stray string in an editor and in code review; the lint rule makes the actual ban enforced automatically, in CI, on every future page, without relying on a reviewer noticing. Choosing a repo-wide restriction (rather than option 3's precise, type-aware version) is a deliberate breadth-over-precision tradeoff: it is coarser (it also flags a hypothetically legitimate `Number()` call on genuinely non-financial data inside a component), but `src/app/**`/`src/components/**` is exactly the surface where non-financial numeric coercion is rare and financial-value coercion is the realistic risk, so the false-positive cost is low and the rule needed zero new tooling to ship today.
- **Tradeoffs**: A component that genuinely needs to parse a non-financial number (e.g. a pagination page index from a query string) will need a narrowly-scoped `// eslint-disable-next-line no-restricted-syntax` with a comment explaining why — an accepted, visible escape hatch rather than a silent one. The rule cannot see through re-exports or aliasing (e.g. `const N = Number; N(x)` would not be caught) — accepted as a deliberate, documented gap; the goal is to catch the obvious, common-case mistake, not to be adversarially unbypassable.
- **Future Implications**: If a genuine, repeated need for numeric coercion on non-financial data emerges in `src/app/**`/`src/components/**`, revisit toward option 3 (a type-aware rule) rather than accumulating scattered disable comments. Any new backend financial field added to `src/types/api.ts` must be typed `DecimalString`, not `string`, in the same change.

## ADR-030 — API Contract Strategy: Hand-Written Types, Deferred Codegen, Live-Schema Drift Detection

- **Date**: 2026-07-16
- **Status**: Accepted
- **Context**: `frontend/src/types/api.ts` was hand-written against `docs/api_design.md` and spot-read backend Pydantic schemas during M7 Phase 1, with no formal codegen pipeline and no automated way to notice if the backend's actual response shape ever diverges from what the frontend assumes. M7 Phase 1.5's contract-drift review, prompted to evaluate this informally, found the risk was not hypothetical: reading the real Pydantic schemas directly turned up two genuine mismatches already shipped (KI-036: `GrowthSeriesPoint`/`DisclosedSplit`'s date fields are named `point_date`/`split_date`, not `date`; KI-038: six `SimulationResponse` fields are nullable, and `error_message` was missing entirely).
- **Problem**: FastAPI generates a complete OpenAPI schema (`/openapi.json`) for free, which could in principle drive full type codegen (`openapi-typescript` or similar) — but hand-written types currently encode intent codegen can't produce automatically (the `DecimalString` branded type, ADR-029's guardrail, and deliberate per-field documentation of *why* a field is nullable), and replacing them wholesale is a real migration, not a drop-in.
- **Options Considered**: (1) Adopt `openapi-typescript` (or equivalent) codegen now, regenerating `types/api.ts` from a live or exported schema as part of the build. (2) Keep hand-written types, but add a concrete, automated drift-detection mechanism rather than relying on manual re-reading of backend source during design reviews. (3) Keep hand-written types with no automated check at all, relying on manual review — the status quo M7 Phase 1 shipped with, which this same review just showed is insufficient.
- **Final Decision**: Option 2. `frontend/src/__tests__/lib/api-contract-drift.test.ts` fetches the backend's live `/openapi.json` and asserts the specific field names the frontend's hand-written types depend on are present (and that specific renamed/removed fields, like the old `date`, are genuinely absent) — skipping gracefully, not failing, when no backend is reachable (matching the existing Redis-unreachable test convention, KI-035's resolution). Verified against a real, running backend during this same review (postgres/redis containers already running, `uvicorn app.main:app` started directly): all four live-path assertions passed.
- **Rationale**: Full codegen is deferred, not rejected — it remains the more scalable long-term answer once the API contract stabilizes past M7's remaining phases (Simulator/Results screens will need several more response shapes) and once a decision is made about how to preserve `DecimalString` branding and guardrail-relevant documentation through a generated file (e.g. a thin hand-written wrapper layer over generated base types). Shipping a working drift detector today, cheaply, closes the actual gap (silent, undetected mismatch) without blocking on that larger design question.
- **Tradeoffs**: This drift detector only catches a mismatch when someone runs the test suite against a live backend — it will not catch drift in CI unless a backend service is provisioned there (not currently the case), so it is a strong *local development and manual-review-time* safety net, not a CI gate yet. It also only checks field *names*, not full type/shape compatibility (a field could be renamed to something present-but-differently-typed and this test would not catch a type mismatch, only a presence mismatch).
- **Future Implications**: Revisit full codegen once the API surface this frontend depends on stabilizes past Phase 2's initial build-out. If CI ever provisions a backend service (mirroring the `postgres`-in-CI precedent), promote this drift test from a locally-run safety net to an enforced CI gate. Any new hand-written type added to `types/api.ts` for a new endpoint should get a corresponding drift-detection case in the same change, not as an afterthought.

## ADR-031 — Isolate Next.js's `unstable_retry` Behind a Shared Type, Not a Runtime Wrapper

- **Date**: 2026-07-16
- **Status**: Accepted
- **Context**: `frontend/src/app/error.tsx` and `global-error.tsx` both receive an `unstable_retry` prop from Next.js — a function explicitly marked unstable by Next.js itself (introduced at v16.2.0, per that version's own bundled docs). M7 Phase 1.5 was asked to isolate this volatility rather than let two files each inline the same unstable prop shape.
- **Problem**: If a future Next.js release renames, changes the signature of, or removes `unstable_retry` (which its own naming explicitly signals as possible), every file that inlines its type shape needs a coordinated edit, and it is easy to miss one.
- **Options Considered**: (1) Leave the prop shape inlined in both files (the M7 Phase 1 state). (2) Extract a shared type (`frontend/src/lib/next-error-boundary.ts::RouteErrorBoundaryProps`) that both files import. (3) Build a runtime wrapper component around `error.js`/`global-error.js` that abstracts `unstable_retry` behind a stable, custom-named API.
- **Final Decision**: Option 2.
- **Rationale**: Next.js itself dictates the exact prop signature of the `error.js`/`global-error.js` file-convention special files — it calls these components directly as part of its own routing/rendering pipeline, with a prop shape it controls, not one this project defines. Option 3 (a runtime wrapper) would not actually reduce coupling to the unstable API: Next would still invoke the wrapper with exactly this shape, so a future rename would still require updating the wrapper. A shared *type* accomplishes the real, achievable goal — one place to update, not two or more — without pretending to insulate against a framework-level API change that isolation can't actually prevent.
- **Tradeoffs**: `global-error.tsx` deliberately imports nothing else from the app (it is the "everything else failed" boundary) — the shared type is imported there too, but as a `import type`, which TypeScript fully erases at compile time and which therefore carries no runtime dependency, preserving that file's "nothing left to fail" property.
- **Future Implications**: If Next.js's `error.js`/`global-error.js` prop contract changes in a future upgrade, `RouteErrorBoundaryProps` is the only place requiring an edit; `frontend/src/app/error.tsx` and `global-error.tsx` themselves would not need to change beyond whatever new behavior the new API enables.

## ADR-033 — Decimal-Comparison Guardrail: `compareDecimalStrings` Helper + ESLint Ban on Bare Relational Operators, Scope Expanded Beyond `app`/`components`

- **Date**: 2026-07-18
- **Status**: Accepted
- **Context**: A pre-Simulator foundation review (M7 Phase 2 punch list) found that ADR-029's `DecimalString` guardrail covered numeric *coercion* (`Number()`, `parseFloat()`, `parseInt()`, unary `+`) but not *comparison*. Nothing stopped a future component from writing `if (sim.final_value > sim.investment_amount)` — which compiles, since both sides are typed `DecimalString` (a branded `string`), and appears to work for small test cases, but is silently wrong the moment the two values' integer-part lengths differ (`"9" > "10"` is `true` as a string comparison, backwards from the numeric answer), since JS compares strings lexicographically, not numerically.
- **Problem**: The Simulator's own upcoming needs (ordering/sorting, badge-choice logic that might compare two already-fetched figures) make this a live risk, not a hypothetical one — the exact "documented rule survives review, then gets violated once someone reaches for the obvious syntax" pattern ADR-029 was written to prevent for coercion, unaddressed for comparison.
- **Options Considered**: (1) Document "never compare DecimalStrings directly" in the format README and rely on review. (2) Add a sanctioned `compareDecimalStrings(a, b): -1 | 0 | 1` helper (`src/lib/format/compare-decimal-string.ts`) performing sign/integer-length/digit-string comparison — the same string-only-digit-math discipline `decimal-string.ts` already uses — plus an ESLint `no-restricted-syntax` selector banning bare `<`/`>`/`<=`/`>=` `BinaryExpression`s in product/UI code, mirroring ADR-029's mechanism exactly. (3) A type-aware ESLint rule that only flags comparisons where an operand's static type is `DecimalString` — more precise, but (like ADR-029's own rejected option 3) requires adopting `@typescript-eslint`'s type-checked linting in this config for the first time, a larger undertaking than this guardrail warrants on its own.
- **Final Decision**: Option 2, plus expanding the *existing* ADR-029 rule's file scope in the same change: `eslint.config.mjs`'s guardrail block now covers `src/hooks/**`, `src/providers/**`, and `src/lib/**` in addition to `src/app/**`/`src/components/**` — the punch-list's own explicit ask, since a hook or a shared `lib` helper is just as capable of an accidental `Number()`/bare-comparison mistake as a component is, and M7 Phase 1.5 only ever wrote the rule against the two directories that existed at the time.
- **Rationale**: `compareDecimalStrings` is exported from `src/lib/format/index.ts` alongside the other sanctioned formatters — ordering two already-correct values is a display/UI concern (like rounding or grouping), not a derivation of a new financial figure, so it belongs in this module per its own README contract, with the README's "NOT allowed" list edited to state the one narrow exception precisely (compare via the helper; never combine via arithmetic). `src/lib/format/**` itself is added to the guardrail's `ignores` — the same treatable exception ADR-029 implicitly relied on (the format module's own static-analysis test already proves it uses no banned coercion tokens) made explicit now that a real *comparison* rule exists and `compare-decimal-string.ts`'s own implementation legitimately does raw digit-string `<`/`>` comparisons to build the sanctioned helper.
- **Tradeoffs**: Identical breadth-over-precision tradeoff to ADR-029: the selector matches any bare relational-operator comparison in the covered directories, not only ones on a `DecimalString` operand, so a genuine non-financial comparison (e.g. `params.query.length > 0` in `use-asset-search.ts`, the one pre-existing case this expansion surfaced) needs a scoped, commented `eslint-disable-next-line` — an accepted, visible escape hatch, not a silent one. Like ADR-029, the rule cannot see through aliasing or a helper that re-wraps a comparison operator.
- **Future Implications**: Any future code needing to order two `DecimalString` values must import `compareDecimalStrings`, not write a bare comparison. If genuine, repeated false positives from the broad selector accumulate in practice, revisit toward a type-aware rule (this ADR's rejected option 3) rather than accumulating scattered disable comments, matching ADR-029's own stated revisit condition.

## ADR-034 — Lock the Tailwind Breakpoint Namespace to the Three Approved Breakpoints

- **Date**: 2026-07-18
- **Status**: Accepted
- **Context**: `frontend_design_system.md` §12 approves exactly three breakpoints (`sm`/`md`/`lg` at 640/1024/1440px, matching mobile/tablet/desktop-and-wide) as a deliberate three-tier scale, explicitly "rather than Tailwind's default five-tier scale." `globals.css`'s `@theme inline` block redeclares `--breakpoint-sm/md/lg` at those exact values — but a pre-Simulator foundation review found this redeclaration does not remove Tailwind v4's own built-in `--breakpoint-xl` (1280px) and `--breakpoint-2xl` (1536px), since `@theme inline` only overrides the specific keys it declares; it does not clear the rest of the namespace. Concretely, `xl:text-lg` or `2xl:grid-cols-4` would compile and work today, silently reintroducing the five-tier scale the design system explicitly rejected.
- **Problem**: An unapproved breakpoint utility compiling successfully (rather than failing to resolve, or failing lint) means the only thing preventing its use is a contributor remembering a rule stated in a markdown design doc, not the codebase itself — exactly the "documented rule, no structural enforcement" pattern ADR-029 and ADR-033 both already treat as insufficient for financial-math guardrails, here applied to a design-system boundary instead.
- **Options Considered**: (1) Leave `xl`/`2xl` available but rely on code review/design-review convention to never use them (the status quo). (2) Explicitly reset the entire `--breakpoint-*` namespace to `initial` inside the same `@theme inline` block, immediately before redeclaring only the three approved breakpoints — Tailwind v4's own documented mechanism for clearing a theme namespace before repopulating it. (3) Reconfigure breakpoints via a separate `tailwind.config.*` file instead of CSS-first `@theme` — rejected outright, since this project has no such file and introducing one solely for this would be a larger, unrelated architectural change (CSS-first configuration is Tailwind v4's own recommended default, and every other token in this codebase already follows it).
- **Final Decision**: Option 2. `--breakpoint-*: initial;` added directly above the three `--breakpoint-sm/md/lg` declarations in `globals.css`.
- **Rationale**: This is the same "make the rule structural, not just documented" mechanism this project already applies to financial-math guardrails (ADR-029/ADR-033), applied here to a design-token boundary: after the reset, `xl:`/`2xl:` utility classes no longer resolve to anything (Tailwind treats an unset breakpoint variant as unrecognized), so an accidental use fails to produce the intended responsive behavior immediately and visibly during development, rather than silently compiling and shipping. A permanent regression test (`src/__tests__/lib/breakpoints.test.ts`) asserts both the reset's presence/ordering in `globals.css` and that no `xl:`/`2xl:` utility exists anywhere under `src/`.
- **Tradeoffs**: None identified against the approved three-tier design — this is a pure enforcement fix within an already-approved token scale (`frontend_design_system.md` §12's three breakpoints are unchanged), not a new design decision. If a future milestone genuinely needs a fourth breakpoint tier, it must be added as a deliberate, founder-reviewed change to both `globals.css` and `frontend_design_system.md` §12 in the same change, not silently reintroduced via Tailwind's defaults.
- **Future Implications**: Any future breakpoint-scale change must update `globals.css`'s three explicit declarations and `breakpoints.test.ts`'s expectations together — the reset means the namespace no longer inherits anything from Tailwind's built-in defaults, so a forgotten declaration here is now a build-visible gap (no breakpoint at all), not a silent fallback to an unapproved Tailwind default.

## ADR-035 — `DevSeedProvider`: A Local-Development-Only Fixture Provider, Not a Fix to Yahoo's Rate Limiting

- **Date**: 2026-07-18
- **Status**: Accepted
- **Context**: With the Simulator (frontend) built and able to reach a running backend, manual end-to-end testing needed real asset/price data in the local database. `YFinanceProvider` failed for every symbol tried (AAPL, SPY, BTC-USD) with an opaque JSON-parse error; root-caused via yfinance's own debug mode to Yahoo rate-limiting (`429`) yfinance 0.2.44's internal crumb-negotiation endpoint (KI-044) — confirmed to reproduce identically outside Docker, ruling out a container-specific networking cause, and confirmed the underlying Yahoo chart API itself was reachable and returning valid data via a plain unauthenticated request, ruling out a wholesale IP block.
- **Problem**: Manual Simulator testing needed to be unblocked now, without depending on Yahoo's rate-limit state clearing (unpredictable, outside this project's control) and without weakening the ingestion pipeline's own validation/normalization/audit guarantees to do it, per `.claude/SYSTEM.md`'s "Historical Truth Is Sacred" principle and this project's standing rule that seed/fixture data must never be mistaken for real provider data.
- **Options Considered**: (1) Fix yfinance's Docker networking/headers — rejected, since the diagnosis showed this was never a networking/header issue; no such fix exists to make. (2) Run ingestion outside Docker — rejected; empirically reproduced the identical failure on the bare host, so it isn't a fix either, only a different environment exhibiting the same root cause. (3) Bump the `yfinance` dependency version now — deferred, not rejected: a real fix for the underlying cause, but unverifiable without a live, non-rate-limited window to test against, and a moving target given Yahoo's own evolving anti-bot behavior; tracked as KI-044's actual planned resolution, not solved by this ADR. (4) Add a dev-only fixture provider (`DevSeedProvider`) plugged into the existing Provider Layer extension point (`app/ingestion/providers/__init__.py::get_provider`'s own docstring: "adding a future provider means adding one entry here and one adapter module — nothing downstream needs to change"), going through the unmodified orchestrator/normalization/validation/repository/audit pipeline.
- **Final Decision**: Option 4, `DevSeedProvider` (`backend/app/ingestion/providers/dev_seed_provider.py`, registered as `--provider dev_seed`). Explicitly a local-dev mitigation for KI-044, not a fix to it — options 1–3 remain the real resolutions, tracked separately.
- **Rationale**: Implementing this as a `PriceProvider` (the same Protocol `YFinanceProvider`/`CoinGeckoProvider` implement, per `providers/base.py`) rather than a standalone script bypassing the pipeline means every existing guarantee stays intact: `get_or_create_asset` still runs, `validate_price_record` still rejects a malformed row, `record_import_audit` still logs the import — a `dev_seed` import is auditable and structurally indistinguishable from a real one *except* for its `data_source` label and origin, which is exactly the property "seed data must be clearly marked, never contaminate real data" requires. Three concrete safeguards enforce that property: (a) `name = "dev_seed"` is the literal string written to `assets.data_source` — the same column every real provider populates, never disguised as `"yfinance"`; (b) the constructor raises `RuntimeError` immediately if `Settings.environment` is not `development`/`test`/`testing`, mirroring `Settings`'s own existing `JWT_SECRET`/`AI_PROVIDER` production guards (`app/core/config.py`) — this provider cannot be reached in a production deployment even if `--provider dev_seed` were mistakenly passed to the CLI there; (c) only three fixed symbols are defined, at deliberately round, obviously-fake price levels (AAPL $100.00, SPY $400.00, BTC-USD $20,000.00) rather than any attempt to approximate real historical prices — a design choice specifically so a screenshot or log line showing this data could never be mistaken for a real historical figure out of context.
- **Tradeoffs**: A real, if minor, data-hygiene bug was found applying this fix: `get_or_create_asset` only sets `data_source` when it *creates* an `Asset` row, never when it reuses an existing one (`app/ingestion/storage/repository.py:43`) — pre-existing `AAPL`/`SPY`/`BTC-USD` rows from earlier failed `--provider yfinance` attempts (KI-044) had already been created with `data_source="yfinance"` before their price fetch failed, so the first `dev_seed` import silently inherited that stale label on reuse. Confirmed (by inspecting `historical_prices` row counts/date ranges before deleting anything) that no real price data existed under those rows — a failed yfinance fetch never gets far enough to write one — then deleted the three stale rows and re-ran `dev_seed` cleanly, with explicit user sign-off before the delete given its destructive, cross-table nature. This `get_or_create_asset` behavior (label frozen at first creation, even by an since-abandoned provider attempt) is a real, general data-hygiene gap beyond just this fixture provider's use — not fixed here, since fixing it is a change to shared, real-ingestion-path code, out of this ADR's scope; tracked as a follow-up note in `docs/KNOWN_ISSUES.md` KI-044's resolution text rather than a separate KI, since it was only ever observed in the course of applying `dev_seed` and has no known real-provider trigger today (every real provider's `get_or_create_asset` call already passes its own correct `data_source` at first creation; the gap only bites when a *different* provider than the one that first created the row imports the same symbol later).
- **Future Implications**: `dev_seed` must never be extended with more symbols or richer data than "unblock manual testing" strictly requires — if a future need calls for a larger or more realistic local dataset, prefer a proper fixtures/factory layer for tests specifically, not growing this provider. If `get_or_create_asset`'s frozen-`data_source`-on-reuse behavior ever causes a real (non-dev-seed) mislabeling incident, promote the tradeoff above from a documented note to its own KI and fix `get_or_create_asset` directly (e.g., update `data_source` whenever a reused asset's existing value doesn't match the current call's `provider.name`, with disclosure).

- **Context**: M7 Phase 3B's Results foundation requires a "basic technical details disclosure" showing the simulation ID, calculation version, and created timestamp (task scope, matching `docs/EXPERIENCE_CONSTITUTION.md` §5's "every number carries a legible source" principle extended to metadata). `calculation_version` has existed on the `Simulation` ORM model since the first migration (`app/models/simulation.py`, `docs/ARCHITECTURE_DECISIONS.md` ADR-002's non-negotiable) but was never included in `SimulationResponse` (`app/api/v1/schemas/simulations.py`) — a pure omission, not a deliberate exclusion recorded anywhere.
- **Problem**: The frontend must not calculate or invent this value (`.claude/SYSTEM.md` Principle 3/`docs/api_design.md`'s "no financial calculation logic in the frontend"), so it can only be shown if the backend returns it. Decide how to expose an already-stored, already-correct field with the least risk to the surrounding milestone (a growth-series persistence fix, Founder Decision 014, was under design review in the same pass but is explicitly not implemented yet).
- **Options Considered**: (1) Do nothing and omit calculation version from the Results foundation's technical-details disclosure, deferring the schema change to whenever Founder Decision 014's persistence work lands — rejected, since it directly contradicts the M7 Phase 3B task's explicit technical-details requirement for no real benefit (the field already exists and is already correct on every row). (2) Add `calculation_version: str` to `SimulationResponse` and populate it from `simulation.calculation_version` in `from_simulation` — a pure, additive, backwards-compatible field exposure with no schema migration, no engine change, and no behavior change to any existing field. (3) Bundle this exposure into the same future change that implements Founder Decision 014's `growth_series` persistence — rejected as unnecessary coupling: the two are independent (one exposes an existing scalar column, the other persists a new series column) and gating the trivial, zero-risk one behind the larger, unscheduled one serves no purpose.
- **Final Decision**: Option 2. `calculation_version` added to `SimulationResponse` (`backend/app/api/v1/schemas/simulations.py`), populated in `SimulationResponse.from_simulation`, and returned identically on both `POST /api/v1/simulations` and `GET /api/v1/simulations/{id}` — the field is already correct and versioned identically regardless of retrieval path, so no `POST`/`GET` asymmetry (unlike `growth_series`'s current KI-021 gap) is introduced.
- **Rationale**: This is the narrowest possible fix for the actual need — one field, one classmethod, no schema migration (the column already exists), no test-breaking change beyond updating the two integration tests and one component-test fixture that construct a full `SimulationResponse`/mock. Exposing it now, independent of Founder Decision 014, means the Results foundation's technical-details disclosure is honest and complete today rather than showing a partial disclosure with a note to "check back later" for a field that was always available.
- **Tradeoffs**: None identified. The field was always intended to be visible per its own docstring ("so reproducibility can be enforced from day one") — this ADR simply closes an oversight, not a deliberate historical exclusion.
- **Future Implications**: Any future change to how `calculation_version` is derived or incremented (e.g., a v2 calculation methodology) is now immediately visible to the frontend without a further schema change — the exposure only needs to happen once. `docs/api_design.md`'s `SimulationResponse` example JSON and §5's retrieval-path description were updated in the same pass to keep documentation and implementation synchronized.

## ADR-037 — Results Foundation Page: Server/Client Split, Status-Aware Rendering, and `StatTile` Compact Size Variant

- **Date**: 2026-07-19
- **Status**: Accepted
- **Context**: M7 Phase 3B builds `/simulation/[id]`, the platform's first dynamic route and the second half of the Simulator's "simulate-and-explain loop" (`docs/EXPERIENCE_CONSTITUTION.md` §4). Three genuinely new problems needed a decision, none of which the Simulator (M7 Phase 2, a single static route) had to solve: (1) how a dynamic-segment page should be structured given this Next.js version's async `params` (a `Promise`, not a synchronous prop — confirmed directly against `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/dynamic-routes.md`, not assumed from prior training data, per `frontend/AGENTS.md`'s standing instruction that this Next.js version has breaking changes from what a general-purpose model would expect); (2) how to render a simulation whose `status` is `pending` or `failed`, a case KI-038 explicitly flagged as "not yet done, since no Results screen exists yet"; (3) a real, found-during-build layout defect — `StatTile`'s existing `text-3xl`/`sm:text-4xl` value typography, exercised for the first time in a genuine three-tiles-in-a-row layout (the Simulator's own `StatTile` usages in `dev/playground` are shown singly, `w-56`, never three abreast), visibly overflowed its card for a real five-figure `final_value` (`$11,814.09`) at the page's `max-w-2xl` container width — caught by live-rendering the page against a real backend-created simulation and inspecting a screenshot, not assumed correct from passing unit tests alone (this project's established verification discipline, e.g. KI-043).
- **Problem**: Structure the route, the status-aware rendering, and fix the layout defect without over-building — this is explicitly a "foundation" pass (no growth chart, no AI panel, no methodology section, per direct task scope) that must still be genuinely correct for every reachable `status` value, not just the happy path.
- **Options Considered — dynamic segment structure**: (1) A single async server component reading `params` directly and rendering everything server-side — rejected, since the Results data itself must come from `useSimulation` (TanStack Query, the established GET-query convention per ADR-032), which requires a Client Component boundary; a server component cannot call it. (2) Mirror the Simulator's existing page/client split exactly (`frontend/src/app/simulator/page.tsx` + `frontend/src/components/simulator/simulation-form.tsx`): a thin async server component (`app/simulation/[id]/page.tsx`) that only awaits `params` and passes the resolved `id` string down to one `'use client'` component (`SimulationResultClient`) that owns all data-fetching and rendering.
- **Final Decision (dynamic segment structure)**: Option 2 — identical shape to the Simulator's own precedent, so a contributor who has read `simulator/page.tsx` can predict `simulation/[id]/page.tsx`'s structure without re-deriving it.
- **Options Considered — status-aware rendering**: (1) Only build the `completed` case now and treat `pending`/`failed` as a later phase's problem — rejected; `SimulationResponse.status` is a real, typed union today (`SimulationStatus`), and every hero-number field is already nullable specifically because `pending`/`failed` are real states (KI-038's own finding), so building only one of three reachable branches would ship a page that can crash or silently render `null` as a number for two-thirds of its own type's valid values. (2) Render all three states from one component, switching on `sim.status`: `completed` shows the hero `StatTile` row; `pending` shows a calm "hasn't finished calculating yet" card with a manual `refetch()`-backed "Check again" action (no polling/auto-refresh added, since simulations complete synchronously within the `POST` request today — no background worker exists that would make a pending row later transition to completed without a new user action, so an auto-poll would only add complexity with nothing to observe); `failed` renders the existing `error_message` through the same `ErrorState` primitive the Simulator's own success-card sub-case already uses for a failed simulation (`simulation-form.tsx`), rather than inventing a second failure-rendering pattern.
- **Final Decision (status-aware rendering)**: Option 2, verified against two real backend-created rows in this pass (not only unit-tested): a `completed` AAPL simulation and a genuinely `failed` one (`MISSING_HISTORICAL_DATA`, the same error path the Simulator's own form already exercises), both fetched live through `/simulation/[id]` and screenshotted.
- **Options Considered — the `StatTile` overflow defect**: (1) Add `break-words` only, letting a long value wrap onto a second line inside the existing type scale — technically prevents the value being clipped/hidden (the actual bug: overflowing text was being visually painted over by the next grid cell's opaque background, not merely rendered outside a box), but produces an ugly mid-digit-group break (`$11,814.` / `09`) for the common case, which is a real legibility regression for a product whose brand identity is built on numeric precision. (2) Widen the page's container (`max-w-2xl` → `max-w-3xl`) to give three columns more room at the existing type scale — tried and measured insufficient (a five-figure currency value still wrapped even at `max-w-3xl`'s column width, per live screenshot comparison). (3) Add an explicit `size?: 'default' | 'compact'` prop to the shared `StatTile` primitive (`frontend/src/components/ui/stat-tile.tsx`), defaulting to `'default'` (today's unchanged `text-3xl`/`sm:text-4xl`, `p-6`) so every existing call site (the Simulator, `dev/playground`) is pixel-identical to before, with `'compact'` (`text-xl`/`sm:text-2xl`, `p-4`) opted into only by a genuine multi-tile row.
- **Final Decision (the overflow defect)**: Option 3, plus keeping `break-words` and `min-w-0` (added to the tile's outer `flex` container so it can actually shrink below its content's intrinsic width) as a defensive floor under `'compact'` too — so an even longer future value (e.g., a seven-figure `final_value`) degrades to a wrap rather than a silent visual clip, never the reverse. Verified by live screenshot at `'compact'` with the same real `$11,814.09` value rendering on one line.
- **Rationale**: The `StatTile` fix is scoped as a new opt-in variant rather than changing the component's default typography, because the default size is already correct and pixel-tested for its existing single/standalone use — changing it globally to fix a problem specific to a three-abreast layout would be an unrelated visual regression to every other call site. Finding this defect required actually rendering the page against live data (this session created a real simulation via the running backend and screenshotted three states with Playwright) rather than trusting the component/unit test suite alone, which could not have caught a cross-component layout overflow — consistent with this project's `verify` skill and its established precedent (KI-043) that a passing test suite and a correct live render are not the same claim.
- **Tradeoffs**: `'compact'` is a second typography scale to keep in sync with `'default'` if `StatTile`'s visual language changes later (e.g., a font-weight or tabular-nums adjustment must be applied to both branches). Judged acceptable: the two scales are visually related (same font, same weight, same `.figure` tabular-numeral treatment — only `text-*`/`p-*` sizes differ), not a forked component.
- **Future Implications**: Any future screen placing three or more `StatTile`s in a single row should default to `size="compact"` rather than rediscovering this overflow independently; a fourth-or-more-tile row should be reviewed against the same live-render-with-real-data discipline before shipping, not assumed safe by extrapolation from three.

## ADR-038 — Results Opening Sequence: A `?new=1` Query Marker, Not `sessionStorage`, as the "Play Exactly Once" Mechanism

- **Date**: 2026-07-20
- **Status**: Superseded by ADR-041
- **Context**: M7 Phase 3B.1 builds the Results Opening Sequence — a one-time, calm phrase-by-phrase reveal of the worked-example sentence and its answer, played only immediately after a newly-created, `completed` simulation, and never on a refresh, browser-back, shared link, or revisited simulation (the approved storyboard's explicit Replay Rules). This requires a way for `/simulation/[id]` to distinguish "the user just arrived here from `SimulationForm`'s own successful submission" from every other way the same URL can be reached, and to make that distinction exactly once per creation.
- **Problem**: Whatever marker is chosen must (a) be set by `SimulationForm` right before navigating, (b) be readable by the Results screen on first mount, and (c) reliably clear itself so a subsequent visit to the identical URL never re-reads a truthy value — while surviving React 18 Strict Mode's development-only double-invocation of render-phase code (component bodies, state initializer functions) without the double-invocation itself corrupting the read.
- **Options Considered**: (1) `sessionStorage`, keyed by simulation id, set by `SimulationForm` and read-and-cleared (`getItem` + `removeItem`) on the Results screen's first mount — the more common pattern for this kind of one-shot signal, and initially attempted. Rejected after tracing a genuine correctness bug it would introduce: reading `sessionStorage` is a side-effecting operation (clearing it) as well as a read, so computing the initial "should this play?" decision inside a `useState` lazy initializer or a bare `useEffect` — both of which Strict Mode deliberately double-invokes in development specifically to catch impure render-phase code — would consume (clear) the marker on the first, discarded invocation and find nothing on the second, "real" one, silently disabling the sequence in dev half the time. Avoiding that would require a `useRef` guard threaded through the effect, adding real complexity to something that should be simple. (2) A React context/prop passed directly from `SimulationForm` to the Results screen — rejected outright; the two are different routes, mounted independently by the router, with no shared parent component instance to carry state through. (3) A `?new=1` query-string marker, appended by `SimulationForm`'s own `router.push`, read via `useSearchParams()` and immediately scrubbed from the URL via `router.replace()` (stripped, not left in place) once read.
- **Final Decision**: Option 3. `frontend/src/hooks/use-just-created-flag.ts`'s `useJustCreatedFlag()` reads `searchParams.get('new') === '1'` inside a `useState` lazy initializer — a **pure** read (`URLSearchParams.get` has no side effect), so Strict Mode's double-invocation is harmless; both calls return the identical answer. A separate `useEffect` (run-once, empty dependency array) then calls `router.replace(pathname, { scroll: false })` to strip the marker from the URL and history entry — also safe to double-invoke, since replacing the same clean URL twice is a no-op. `SimulationForm` only appends `?new=1` when the just-created simulation's `status === 'completed'` (a `failed` result has no answer to reveal, so it navigates to the plain URL).
- **Rationale**: The deciding property is that Option 3's mount-time read is provably pure, sidestepping the entire class of "impure code in a place React may invoke more than once" bugs Option 1 required a manual `useRef` guard to avoid. It also has a secondary, unplanned-for benefit: the marker's absence-vs-presence is directly inspectable in the URL bar during manual testing and trivially mockable in a component test (`useSearchParams: () => new URLSearchParams('new=1')`) without touching global browser storage APIs, which is exactly how `opening-sequence-heading.test.tsx` and `use-just-created-flag.test.tsx` exercise every Replay Rule case (fresh creation, revisited/shared link, reduced motion) without any `sessionStorage` mocking machinery.
- **Tradeoffs**: A query-string marker is, in principle, copy-pasteable by a user before the `useEffect` strips it (a `router.replace` happens on the same tick React commits the first paint, in practice faster than a human could copy the address bar) — accepted as a non-issue given the marker only ever *skips a wait*, it never changes what information is shown (per the Replay Rules, "information must remain identical" regardless of whether the sequence plays), so even a mis-shared `?new=1` link would, at worst, replay a calm three-second reveal for its recipient, not expose or alter anything.
- **Future Implications**: Any future one-shot "just arrived from a specific prior action" signal between two independently-routed pages in this product should default to this query-marker-plus-immediate-strip pattern rather than `sessionStorage`, unless the signal must survive an actual full-page reload before being consumed (which this one deliberately does not need to).

## ADR-039 — Results Reading Experience: Replacing the Dashboard Layout with an Editorial One, and Deferring the Growth Chart Behind an Honest Placeholder

- **Date**: 2026-07-20
- **Status**: Accepted
- **Context**: M7 Phase 3B built the completed-simulation Results screen as a KPI-card dashboard: three bordered `StatTile`s in a row, a bordered "Simulation Snapshot" card, and a bordered "Technical Details" disclosure card, all beneath a conventional page-title heading. M7 Phase 3B.2's direct instruction explicitly rejects that pattern by name ("the previous implementation followed a generic finance-dashboard pattern... do NOT continue from that design") and specifies a fixed seven-section editorial reading order instead, with the worked-example sentence (already built in M7 Phase 3B.1's `OpeningSequenceHeading`) as the page's sole hero — "not the chart, not the cards, not the number — the sentence."
- **Problem**: Two decisions, one straightforward and one genuinely constrained by an existing, already-tracked backend gap: (1) how to restructure the page's visual composition without discarding the reusable pieces (routing, `useSimulation`, the opening sequence's animation/replay logic, `SimulationSnapshot`'s and `TechnicalDetails`' underlying data) the instruction explicitly says to keep; (2) what "Section 5 — Growth Over Time" should actually render, given `growth_series` is empty on every retrieval today (KI-021 — Founder Decision 014 approved the persistence fix, not yet implemented, scheduled for M7 Phase 3C) — a real chart would have no real data to draw, and `docs/BRAND_CONSTITUTION.md`/`docs/EXPERIENCE_CONSTITUTION.md` both explicitly forbid fabricating or smoothing over a genuine data gap.
- **Options Considered — layout**: (1) Keep `StatTile`'s bordered-card treatment for Section 4's "Supporting Facts," just re-labeled — rejected; the instruction's own visual rules ("avoid... unnecessary borders... decorative chrome") name exactly this pattern, and `StatTile`'s `rounded-...border border-border-hairline bg-surface p-6` wrapper is definitionally a KPI card, regardless of its label. (2) Write new, unstyled `<dl>`/`<dt>`/`<dd>` markup directly for Section 4's facts, keeping `StatTile` itself unchanged for any future caller that legitimately wants a bordered tile (e.g., `dev/playground`'s own display of it) — chosen, since the redesign's rejection is of the *card treatment for this specific screen*, not of `StatTile` as a component. (3) Fold the former "Simulation Snapshot" card's content (asset/amount/date range/dividends/inflation) into the new Section 7 "The Proof" disclosure as its own "Assumptions" subsection, alongside the former "Technical Details" card's content (simulation ID/calculation version/created timestamp) as a "Technical details" subsection, under one shared `<details>` rather than two separate bordered cards — chosen, matching the instruction's explicit "collapsed by default... nothing technical appears before the explanation" for exactly this content.

(2) Add a charting library now and wire it to `growth_series`, accepting that it will only ever render for the always-empty case until KI-021 lands — rejected as premature scope: introducing a charting dependency, its color/accessibility/table-fallback requirements (`docs/BRAND_CONSTITUTION.md` §9's "every chart ships a table-view fallback"), and its own design review is a substantially larger decision than this phase's stated scope ("do not implement motion yet... perfect the static composition first"), and doing it against literally zero real data would mean shipping and testing a chart component this project could never actually verify against a real render. (3) Render Section 5's kicker and heading structurally (preserving the fixed reading order), with a single calm, factual paragraph stating the data gap plainly when `growth_series.length === 0`, and a minimal one-line fallback ("N data points recorded...") for the day a `growth_series`-bearing simulation is actually retrievable — chosen.
- **Final Decision**: Option 2 for the layout (plain `<dl>` facts, `StatTile` untouched elsewhere) plus Option 3 (Snapshot + Technical Details folded into one "The Proof" disclosure); Option 3 for the growth chart (an honest, structurally-future-proofed placeholder, no charting library added this pass).
- **Rationale**: `docs/EXPERIENCE_CONSTITUTION.md` §5's "a genuine data gap... is stated as a plain fact, never smoothed over, never dressed up as an error the user caused" directly governs the growth-chart decision — the Final Value and Total Return figures on the same page are already exact and complete, so the honest move is to say precisely that ("only this specific view of how it got there is still being built") rather than imply the whole result is somehow incomplete. Keeping `StatTile` itself unmodified (rather than adding a "flat" variant to its public API) avoids growing a shared component's surface area for a single caller's specific rejection of its default look — the new Section 4 markup is bespoke to this screen because the *screen's* aesthetic changed, not because `StatTile` was wrong.
- **Tradeoffs**: Section 7 now duplicates a small amount of structure the old `SimulationSnapshot`/`TechnicalDetails` components already had (two `<dl>` blocks under one `<details>` instead of two separate components) — accepted, since the instruction's reading order requires exactly one collapsed disclosure holding all secondary/technical content, not two. The growth-chart placeholder's copy references KI-021 conceptually ("a known, tracked limitation") without naming it by number in the UI itself (a user-facing sentence never cites an internal issue tracker ID) — the ADR/DEVLOG entries are where that specific traceability lives, per this project's usual split between user-facing copy and internal documentation.
- **Future Implications**: When Founder Decision 014's `growth_series` persistence actually lands (M7 Phase 3C), `GrowthOverTime` (`frontend/src/components/simulation-result/results-sections.tsx`) is the one place to replace with a real chart — its `hasSeries` branch already exists and is already tested for the populated case, so that phase's work is additive (swap the placeholder paragraph for a real chart, keep the empty-state branch for any simulation created before the backfill), not a redesign of this section's structure. Any future Results-screen content should default to this phase's editorial posture (typography/whitespace/rhythm, no dashboard-card chrome) rather than reintroducing bordered tiles, unless a future founder decision explicitly reverses M7 Phase 3B.2.

## ADR-040 — CAGR Percentage-Scale Fix: `calculation_version` "v2", Backfill Design, and the `NUMERIC(10,6)` Overflow Bound Check

- **Date**: 2026-07-21
- **Status**: Accepted
- **Context**: [Founder Decision 016](FOUNDER_DECISIONS.md) approved Option 1a — fix `calculate_cagr` at the source to return a percentage (matching `calculate_total_return_percent`'s existing convention), bump `calculation_version`, and backfill every existing `completed` simulation's stored `cagr_percentage` immediately, closing `docs/KNOWN_ISSUES.md` KI-045. That decision fixed *what* to do; this ADR is the engineering record of *how*, per this project's standing rule that every Founder Decision generates a corresponding ADR.
- **Problem**: Three engineering questions Founder Decision 016 itself left to implementation: (1) what the new `calculation_version` string should be; (2) how to structure the backfill so it is safe to run against live data and idempotent if re-run; (3) what to do about the one edge case the "multiply by 100" backfill isn't trivially lossless for — a "v1" `cagr_percentage` large enough that ×100 would overflow the `NUMERIC(10, 6)` column's bound (max magnitude `9999.999999`).
- **Options Considered — version string**: (1) `"v1.1"` or similar, signaling a minor correction — rejected; `calculation_version` is compared for exact equality everywhere it's read (this backfill migration, any future reproducibility check), and a minor/major convention this project has never used before would need to be invented and documented for a one-time fix. (2) `"v2"` — a new, semantically-clean version distinct from `"v1"`, consistent with `DEFAULT_CALCULATION_VERSION`'s existing flat-string convention (`backend/app/simulation/engine.py`, unversioned beyond a bare string since M3).
- **Final Decision (version string)**: Option 2, `"v2"`. `docs/simulation_formulas.md` §4a documents the "v1" (raw fraction) vs. "v2" (percentage) semantics explicitly, so any future reader of a stored row can resolve `cagr_percentage`'s scale from `calculation_version` alone, without needing to know this ADR's history.
- **Options Considered — backfill mechanism**: (1) A standalone Python script (`backend/scripts/...`), run manually and out-of-band from Alembic — rejected; this codebase has no prior script-based data-migration precedent (Founder Decision 014's `growth_series` backfill, the only comparable precedent, is itself still unimplemented and was scoped as "a one-time migration/script" without picking one), and an out-of-band script has no built-in ordering guarantee relative to the schema change it depends on, nor a downgrade path. (2) An Alembic data migration (`backend/alembic/versions/0004_...py`), using the same `upgrade()`/`downgrade()` shape as `0001`–`0003` — chosen, since this repo's actual precedent (three prior migrations, all schema-only) is Alembic, and a data-only migration (no DDL) is a well-supported, ordinary use of the same tool with no new dependency.
- **Final Decision (backfill mechanism)**: Option 2. `0004_cagr_percentage_v2_backfill.py`'s `upgrade()` runs a single `UPDATE simulations SET cagr_percentage = cagr_percentage * 100, calculation_version = 'v2' WHERE calculation_version = 'v1' AND cagr_percentage IS NOT NULL AND cagr_percentage <= 99.999999` (the pre-scale bound below which ×100 cannot overflow `NUMERIC(10,6)`'s `9999.999999` ceiling) in one statement — idempotent by construction (a second run matches zero rows, since every previously-matched row is now `calculation_version = 'v2'`), and safe against concurrent inserts (`run_simulation` as of this fix always writes `calculation_version = "v2"` directly, per `DEFAULT_CALCULATION_VERSION`, so no new `"v1"` row can appear after this fix ships to be missed or double-counted).
- **Options Considered — the overflow bound**: (1) Let the `UPDATE` run unconditionally and allow Postgres's own `NUMERIC` column constraint to reject any row whose result overflows, failing the whole migration — rejected; a single oversized outlier would abort the entire backfill transaction, leaving every other row (including ones with no overflow risk at all) unfixed until the outlier is manually resolved, and Founder Decision 016's own text ("flag any row where × 100 would overflow rather than silently failing") explicitly asks for a flag, not a hard stop. (2) Silently cap/clamp an overflowing value at the column's max — rejected; a clamped CAGR is a fabricated number, not a corrected one, and this project's "Historical Truth Is Sacred" principle forecloses ever writing a number to a financial-result column that isn't the actual calculated value. (3) Exclude overflow-risk rows from the `UPDATE`'s `WHERE` clause (leaving them at `"v1"`, unscaled) and separately `SELECT` and log them so the migration's own output makes the exclusion visible and actionable, rather than a silent gap discoverable only by later, unrelated inspection.
- **Final Decision (the overflow bound)**: Option 3. Before the `UPDATE`, `upgrade()` runs a `SELECT id, cagr_percentage FROM simulations WHERE calculation_version = 'v1' AND cagr_percentage IS NOT NULL AND cagr_percentage > 99.999999` and logs each matching row's `id` via `op.get_bind()`'s connection (visible in the migration's own output, `alembic upgrade` logs) with a message naming it as skipped and requiring manual review — these rows are left at `calculation_version = "v1"` with their original, unscaled value, which is a true, findable data state (`calculation_version` itself is the flag), not a mask over an unresolved problem. In practice, a `cagr_percentage > 99.999999` "v1" row means an annualized return exceeding roughly 10,000% before scaling — a value that would already have been implausible display data even under the pre-fix bug, making this a defensive bound check for correctness under all inputs, not one expected to actually fire against this platform's real historical simulation data (no known asset/date-range combination in the current dataset produces a CAGR anywhere near this magnitude).
- **Rationale**: Every choice above optimizes for the same property Founder Decision 016 itself named as the deciding one: a wrong value stored at the source keeps resurfacing in every future consumer, so the correction must be structurally safe to run against real, live data exactly once, self-evident in its own output about anything it couldn't safely fix, and never destructive (no row is deleted; an unfixable row is left exactly as it was, correctly flagged by its own `calculation_version`, rather than forced through a lossy or fabricated correction).
- **Tradeoffs**: A flagged, unbackfilled "v1" row (should one ever exist) continues to display an incorrect CAGR on its Results page indefinitely, exactly like Founder Decision 016's own rejected Option 1b — accepted here only as the narrow, logged exception for a bound-overflow case Option 1b's general rejection didn't anticipate, not a reversal of that decision's "no silent staleness" principle. `downgrade()` is intentionally **not** a perfect inverse: it restores `calculation_version = 'v1'` and divides by 100 for rows this migration touched, but cannot distinguish "was always v1, untouched by this migration" from "was v2 before this migration ran" for any row created between `upgrade()` and a hypothetical `downgrade()` — acceptable since a downgrade of a data-correction migration, after new v2 rows have been created by normal traffic, is not a supported operation this project expects to actually perform (matching this codebase's existing migrations, none of which have ever been downgraded in production).
- **Future Implications**: Any future Simulation Engine formula-correctness fix (not just a new feature) should default to this same shape — fix the source function, bump `calculation_version`, write an idempotent Alembic data migration with an explicit, logged carve-out for any row a lossless rescale can't safely reach, rather than either skipping the backfill or forcing it through unconditionally.

## ADR-041 — Removing the Staged Opening-Sequence Timeline: `useSettleIn`, and Retiring the `?new=1` Replay Marker

- **Date**: 2026-07-22
- **Status**: Accepted
- **Context**: [Founder Decision 017](FOUNDER_DECISIONS.md) rejected the M7 Phase 3B.1 staged composing→pause→reveal timeline (`useOpeningSequence`, ADR-038's `?new=1` replay marker feeding it) while keeping the surrounding editorial Results Reading Experience (ADR-039) unmodified, and specified the one motion still permitted: "at most one ~200ms ease-in settle (opacity/translate), applied once, interruptible, fully disabled under prefers-reduced-motion." This ADR is the engineering record of how that ruling was implemented.
- **Problem**: Three implementation questions the Founder Decision itself left open: (1) what replaces `useOpeningSequence`'s phase state machine for the one permitted transition; (2) whether the `?new=1`/`useJustCreatedFlag` replay-gate mechanism (ADR-038) still serves a purpose once there is no staged timeline left to gate; (3) what to do with the dev-only opening-sequence preview route and the Next.js `Suspense` boundary ADR-038's `useSearchParams` call had required.
- **Options Considered — the settle mechanism**: (1) Keep `useOpeningSequence`'s `setTimeout`-driven phase machine, shrinking it to a single `'composing' -> 'settled'` transition — rejected; the state machine's entire shape (an array of scheduled timeouts, a `phase` enum, a `skip()` escape hatch) exists to serialize *multiple* sequential stages, which is exactly what Founder Decision 017 rejects in principle, not just in duration. Keeping the machinery around for one stage would leave a skip affordance with nothing meaningful to skip (a `~200ms` wait is not worth interrupting) and a phase enum where only two of its four states could ever be reached. (2) A pure CSS entrance animation (`@keyframes`, `animation-fill-mode`) with no JavaScript at all — rejected; a `prefers-reduced-motion` CSS override already exists globally (`globals.css`) and would correctly zero out the animation duration, but the component would have no way to know synchronously whether the animation is playing, which the hero heading's other logic (none remaining after this change, but a defensive property worth keeping for any future caller) benefits from. (3) A minimal boolean hook, `useSettleIn(active)`, using a paired-`requestAnimationFrame` mount trick (commit the initial "unsettled" paint on the first frame, flip to "settled" on the next) to drive a Tailwind/CSS `transition` class pair — chosen.
- **Final Decision (the settle mechanism)**: Option 3. `frontend/src/hooks/use-settle-in.ts` replaces `use-opening-sequence.ts` entirely. `active` is captured once via a lazy `useState` initializer (the same Strict-Mode-safe "pure read" pattern ADR-038 established for `useJustCreatedFlag`) rather than tracked live against `useReducedMotion()`'s own live-updating value — so a `prefers-reduced-motion` OS-setting change mid-session cannot retrigger or interrupt a transition that already resolved one way or the other for that mount, satisfying "applied once" literally, not just "once per timeline." `opening-sequence-heading.tsx` applies the resulting boolean to a single `transition duration-[var(--duration-transition)] ease-in` (reusing the existing `--duration-transition: 220ms` token, `frontend/src/styles/tokens/primitives.css` — close enough to "~200ms" that introducing a second, near-duplicate duration token was judged unnecessary) opacity/translate pair on the `<h1>` only; every other element (the kicker label, and all of `children`) renders with no transition at all, matching "at most one."
- **Options Considered — the `?new=1` mechanism**: (1) Keep `useJustCreatedFlag`/`?new=1` in place, unused by the heading component but preserved for a hypothetical future need — rejected; this codebase's own standing convention (`.claude/SYSTEM.md`, "don't add features/infrastructure beyond what the task requires") argues against retaining a mechanism with no current caller, and ADR-038's entire rationale was specifically "gate a staged timeline so it plays exactly once" — a purpose that no longer exists once every visit renders identically. (2) Remove `useJustCreatedFlag` and the `?new=1` query parameter entirely, so `SimulationForm` navigates to the plain `/simulation/{id}` URL for both `completed` and `failed` outcomes alike — chosen.
- **Final Decision (the `?new=1` mechanism)**: Option 2. `frontend/src/hooks/use-just-created-flag.ts` is deleted. `frontend/src/components/simulator/simulation-form.tsx`'s post-submit `router.push` no longer branches on `sim.status` to decide the URL — every completed or failed simulation navigates to the same plain URL shape. This is a direct, in-scope consequence of Founder Decision 017, not independent scope creep: the mechanism had exactly one caller, and that caller's reason for needing it is the thing the Founder Decision removed.
- **Options Considered — the dev preview route and `Suspense` boundary**: (1) Keep `frontend/src/app/dev/opening-sequence-preview/` for manually eyeballing the new, much shorter settle transition — rejected; the route's own header comment already scoped it as "TEMPORARY, verification-only preview for M7 Phase 3B.1... removed once the opening sequence is manually confirmed," and a ~200ms CSS opacity/translate fade has no multi-stage timing worth a dedicated preview harness the way a 2.75-second choreographed sequence did. (2) Delete the route outright — chosen. Separately, `frontend/src/app/simulation/[id]/page.tsx` previously wrapped `SimulationResultClient` in a `<Suspense>` boundary solely because `useJustCreatedFlag` (transitively, via `OpeningSequenceHeading`) called `useSearchParams()`, which Next.js requires a `Suspense` boundary around or a production build fails (`node_modules/next/dist/docs/.../use-search-params.md`). With `useJustCreatedFlag` removed, no component under `SimulationResultClient` calls `useSearchParams()` anywhere (confirmed by a full-tree grep) — the `Suspense` boundary is dead weight, not a defensive no-op worth keeping, so it is removed along with the call that required it.
- **Final Decision (dev route / Suspense)**: Both removed, as above.
- **Rationale**: Every decision above follows the same principle: once Founder Decision 017 removed the staged timeline, every piece of supporting infrastructure that existed *only* to serve that timeline (the replay marker, the phase state machine, the preview route, the `Suspense` boundary it incidentally required) loses its sole justification at the same time, and this project's standing anti-dead-code convention treats "no longer has a reason to exist" the same as "was never built" — remove it rather than leave it as an unused, confusing vestige a future reader would have to reverse-engineer the purpose of. The one thing deliberately *not* removed is all motion whatsoever — Founder Decision 017 itself distinguishes "theatrical, multi-stage choreography" (rejected) from "a single short, restrained transition" (explicitly permitted), and `useSettleIn`'s minimalism (one boolean, no phases, no skip affordance because there is nothing worth skipping) is the implementation of that distinction, not a loophole reintroducing the rejected pattern at a smaller scale.
- **Tradeoffs**: `frontend/src/__tests__/lib/use-settle-in.test.tsx` and `opening-sequence-heading.test.tsx`'s settle-transition assertions depend on real (or `requestAnimationFrame`-flushed) animation frames rather than `vi.useFakeTimers()`'s `setTimeout` mocking the old phase-machine tests used — a real/flushed-frame test is marginally slower and, in principle, more susceptible to environment timing flakiness than a fully mocked fake-timer test, though jsdom's `requestAnimationFrame` polyfill has proven stable in this suite. Judged acceptable given the mechanism itself (`requestAnimationFrame`, not `setTimeout`) is the more correct primitive for a paint-synchronized CSS transition trigger, and mocking it away would test less of the actual mechanism, not more.
- **Future Implications**: Any future "brief, restrained entrance transition" elsewhere in this product should default to `useSettleIn`'s shape (one boolean, captured once at mount, `requestAnimationFrame`-paired, reduced-motion-aware) rather than reaching for a `setTimeout`-based phase machine unless a genuinely multi-stage sequence is separately, explicitly approved the way the now-rejected M7 Phase 3B.1 sequence once was.

## ADR-042 — `growth_series` Persistence: JSONB Column, String-Serialized Values, Read-Through on GET, Script-Based Backfill

- **Date**: 2026-07-23
- **Status**: Accepted (implements Founder Decision 014, Option A)
- **Context**: [Founder Decision 014](FOUNDER_DECISIONS.md) approved persisting `growth_series` on the `simulations` row at creation time, backfilling pre-existing completed rows by re-running `calculate_growth_series` against each row's own stored inputs and stamping the result with that row's own `calculation_version`, and requiring `GET /api/v1/simulations/{id}` to never return an empty `growth_series` for a `completed` simulation (M7 Phase 3C-2, closing `docs/KNOWN_ISSUES.md` KI-021). That decision fixed *what* to build (persist, version, backfill, read-through); this ADR is the engineering record of *how*, per this project's standing rule that every Founder Decision generates a corresponding ADR. `disclosed_splits` is addressed by the same pass but needed no new storage at all — see its own section below.
- **Problem**: Four engineering questions Founder Decision 014 left to implementation: (1) what column type stores a `growth_series` array, and in what shape; (2) how a stored series round-trips through the same Decimal-safe convention the API already uses (`app.api.v1.schemas.common.DecimalStr`), without ever passing through a JSON number; (3) how the `GET` path serves `disclosed_splits` without a new column, given splits already live in `stock_splits`; (4) how the backfill for pre-existing rows is structured, given — unlike KI-045's CAGR backfill — recomputing a `growth_series` requires replaying `app.simulation.formulas.calculate_growth_series` against stored price/dividend history, not a single SQL arithmetic expression.
- **Options Considered — column type/shape**: (1) A new child table (`simulation_growth_points`, one row per point) — rejected; a `growth_series` is always read and written as one atomic unit (the whole series for one simulation, never a single point in isolation), so a relational one-to-many table adds join/ordering complexity a document-shaped column doesn't need, and this project already has a native-JSONB precedent for exactly this kind of "always read as a whole" structured value (`ai_explanations.input_summary`, `audit_logs.details`). (2) A single nullable `JSONB` column on `simulations`, storing the array directly (`[{"point_date": "...", "value": "..."}, ...]`) — chosen, matching the API response's own `GrowthSeriesPoint` shape closely enough that encode/decode is a thin, obvious mapping, not a schema translation.
- **Final Decision (column type/shape)**: Option 2. `app/models/simulation.py::Simulation.growth_series: Mapped[list[dict] | None] = mapped_column(JSONB, nullable=True)`, added by `alembic/versions/0005_growth_series_persistence.py` (`ALTER TABLE simulations ADD COLUMN growth_series JSONB`, matching the `ALTER TABLE ... ADD COLUMN` shape `0003_ai_explanation_type.py` already established for a non-`0001` schema addition). Nullable and un-defaulted: NULL means "not yet computed/backfilled" for a `pending`/`failed` row or a not-yet-backfilled pre-existing completed row — the same nullable-until-populated convention every other output column on this table already follows (ADR-003).
- **Options Considered — value serialization**: (1) Store `value` as a JSON number — rejected outright; this project's `DecimalStr` convention (`app/api/v1/schemas/common.py`) exists specifically so a JS client's float64 `JSON.parse` can never silently round a financial figure, and storing the *source of truth* in a representation the wire format deliberately avoids would reopen exactly the risk `DecimalStr` was built to close, one layer earlier. (2) Store `value` as a fixed-point string, produced the same way (`format(value, "f")`) the response layer already serializes every Decimal field — chosen, and implemented in a small new module, `app/simulation/growth_series_codec.py` (`serialize_growth_series`/`deserialize_growth_series`), rather than inline in `engine.py`/`simulation_service.py`, so both the write path (engine, at creation) and the read path (service, on `GET`) share one encode/decode contract instead of two independently-maintained ones.
- **Final Decision (value serialization)**: Option 2. `point_date` serializes via `date.isoformat()` / deserializes via `date.fromisoformat()`; `value` serializes via `format(value, "f")` / deserializes via `Decimal(value)` — an exact round trip (verified by `tests/simulation/test_growth_series_codec.py::test_round_trip_preserves_exact_decimal_value`), since `format(d, "f")` preserves a Decimal's own digits and exponent rather than normalizing them, and `Decimal(s)` parses that exact string back losslessly.
- **Options Considered — `disclosed_splits`**: (1) Add a parallel `simulations.disclosed_splits` JSONB column, mirroring `growth_series` — rejected; Founder Decision 014 clause 5 is explicit that no new column is warranted, since `stock_splits` already stores every split queryably by `(asset_id, split_date range)`, and a second, redundant copy would be one more place a future split correction/reimport could silently drift out of sync with its own source of truth. (2) Re-query `stock_splits` fresh on every `GET`, via the exact same `SimulationRepository.get_splits_ordered` range query `run_simulation` already uses at creation — chosen, since splits are rare, low-cardinality, and reading them is a plain indexed range query, not a calculation.
- **Final Decision (`disclosed_splits`)**: Option 2. `simulation_service.get_simulation_by_id` now instantiates a `SimulationRepository` and calls `get_splits_ordered(simulation.asset_id, simulation.start_date, simulation.end_date)` on every retrieval, returning it alongside the simulation and the decoded `growth_series` as a three-tuple; the router passes all three straight into `SimulationResponse.from_simulation`, which already accepted `disclosed_splits` as a parameter (only the `GET` call site was previously omitting it).
- **Options Considered — backfill mechanism**: (1) A pure-SQL Alembic data migration, matching `0004_cagr_percentage_v2_backfill.py`'s precedent exactly — rejected; that backfill was a lossless arithmetic rescale (`× 100`) expressible in one `UPDATE` statement entirely within Postgres. Recomputing a `growth_series` requires walking each simulation's stored price rows in date order, replaying dividend-reinvestment share-count compounding event by event, and reusing `app.simulation.formulas.calculate_growth_series` exactly — a calculation this project deliberately keeps out of raw SQL and out of the API layer alike (ADR-016), so a migration is the wrong place to duplicate it. (2) A standalone Python module, `app/simulation/backfill_growth_series.py`, run manually (`python -m app.simulation.backfill_growth_series[--dry-run]`) against a real database via the same `session_scope()` helper `app.ingestion.cli` already uses for its own manual-operator entrypoint — chosen.
- **Final Decision (backfill mechanism)**: Option 2. The module's `backfill(session, *, dry_run=False)` function selects every `Simulation` row where `status = 'completed' AND growth_series IS NULL`, and for each one, re-derives `initial_shares` from the row's own stored `initial_investment_amount` and the stored `close_price` on `start_date`, re-fetches dividend rows only if `dividends_reinvested` was `True` on that row, and calls `calculate_growth_series` with exactly the inputs `run_simulation` itself would have used — then writes the result via the same `serialize_growth_series` the live creation path uses, **without** touching `calculation_version` (Founder Decision 014 clause 3: never re-stamp to a newer version than the row was originally computed under — `growth_series`'s methodology is untouched by Founder Decision 016's CAGR fix, so a `"v1"` row backfilled today is exactly as methodologically valid as if it had been computed at creation). `backfill()` itself never commits or rolls back — matching this project's other session-taking service functions — so the caller (`main()`'s `session_scope()`, or a test's transactional fixture) owns the transaction boundary.
- **Options Considered — the unrecoverable-row edge case**: (1) Let a `MissingHistoricalDataError` (a row whose start/end price, or a dividend ex-date's price, no longer exists — e.g. after a since-corrected reimport) propagate and abort the whole backfill run — rejected; one bad row would block every other, recoverable row's fix, the same reasoning `ADR-013`'s per-record `SAVEPOINT` isolation already established for ingestion upserts. (2) Catch `MissingHistoricalDataError` per row, leave that row's `growth_series` at `NULL`, and record it in a returned `BackfillResult.skipped` list (with the reason) that `main()` logs individually — chosen, matching the CAGR backfill's own "flag and log, never silently fail" precedent (ADR-040).
- **Final Decision (the unrecoverable-row edge case)**: Option 2. A skipped row remains a `completed` simulation whose `GET` response returns an empty `growth_series` — the one narrow, logged, reported exception to Founder Decision 014 clause 4's "never empty for a completed simulation" rule, exactly analogous to ADR-040's own logged carve-out for an overflow-bound CAGR row. In practice this requires a row's underlying `historical_prices` data to have been deleted or never fully imported for that exact date range after the simulation was created — not expected to match any row in this platform's actual data, but implemented and tested (`tests/simulation/test_backfill_growth_series.py::test_backfill_skips_row_with_missing_start_price`) rather than assumed impossible.
- **Rationale**: Every choice above keeps the same three properties this project's prior persistence/backfill work (Founder Decision 014's own inspiration, ADR-040) already established: the storage format is Decimal-safe by construction, not by convention a future writer could forget; the backfill is safe to run against real, live data exactly once (idempotent — a second run's `WHERE growth_series IS NULL` matches zero rows); and any row the backfill genuinely cannot fix is visibly flagged, never silently masked as fixed.
- **Tradeoffs**: `simulation_service.get_simulation_by_id`'s return type changed from a bare `Simulation` to a three-tuple `(Simulation, tuple[StockSplit, ...], tuple[GrowthSeriesPoint, ...])` — its only caller (`routers/simulations.py::get_simulation`) was updated in the same change, but any future caller must unpack the tuple rather than treat the return value as the ORM row alone. `disclosed_splits` is now computed on every `GET`, not just `POST` — an extra small, indexed query per retrieval, judged negligible against `historical_prices`-scale query volume elsewhere in this codebase.
- **Future Implications**: The skip-and-report pattern (`BackfillResult.skipped`, logged individually with a reason, never a hard failure) is now this project's second precedent (after ADR-040's overflow carve-out) for a data backfill that cannot guarantee 100% coverage — any future backfill should default to this shape rather than either failing the whole run or silently marking an unrecoverable row as fixed. If a future milestone adds a third persisted, engine-computed structure to `simulations`, `app/simulation/growth_series_codec.py`'s split-module pattern (a dedicated, two-function encode/decode module shared by the write path and the read path) is the template to follow, rather than inlining serialization logic into `engine.py` or the service layer directly.

## ADR-043 — The Growth Chart: Recharts Plotting, the Disclosed `toChartPlotNumber` Exception, Split-Marker Disclosure, and the Two-State Dividend Paragraph

- **Date**: 2026-07-24
- **Status**: Accepted
- **Context**: M7 Phase 3C-3 builds the Growth Chart, the Why section, and The Proof's expanded methodology/assumptions/provenance — the frontend half of the Results Experience that M7 Phase 3C-2 (ADR-042) made possible by persisting a reliably non-empty `growth_series`/`disclosed_splits` on every completed simulation's `GET`. This ADR is the engineering record for the choices that ruling left to implementation: how to plot a `DecimalString` series in a codebase whose lint/test guardrails (ADR-029, ADR-033) make converting one to a JS number a build-breaking violation everywhere except one exempted module; how to disclose a stock split without fabricating a figure the platform cannot safely derive; how the Why section's dividend paragraph should behave given the API exposes no dividend-event signal at all; and which of `frontend_design_system.md` §8–9's pre-FD-017 chart-motion language still applies now that a chart finally exists to apply it to.
- **Problem**: Four engineering questions, each forced by a real, discovered constraint rather than a free stylistic choice: (1) Recharts (the locked charting library, `.claude/SYSTEM.md`) requires JS numbers for all SVG plotting geometry, but this codebase's `no-restricted-syntax` ESLint rule bans `Number()`/`parseFloat()`/`parseInt()` in every product directory, and `src/lib/format`'s own static guardrail test (`format.test.ts`) scans every file in that module for the same tokens and fails if found — so there is no location in the entire frontend where a `DecimalString` can become a plottable number without either breaking a guardrail or weakening one. (2) A stock split's `split_ratio` is a `DecimalString` (e.g. `"4.000000"` or `"0.100000"`); phrasing it as "4-for-1" for a forward split is a pure display-rounding operation this codebase already sanctions, but phrasing a reverse split as "1-for-N" requires inverting the ratio — genuine arithmetic (division), which this codebase's financial-math guardrail permanently forbids in product code. (3) `SimulationResponse` exposes `include_dividends` (the user's *choice*) but no dividend-event count, flag, or amount anywhere — so a three-state dividend paragraph (reinvested-with-events / excluded-by-choice / asset-paid-none) as originally specified cannot be built without either a new backend field (out of this task's scope) or the frontend deriving one itself from other fields (banned arithmetic, and unreliable even if attempted). (4) `frontend_design_system.md` §8–9 (written in M7 Phase 0, before any chart existed) describes an approved "line draws in once, event markers appear with a slight delay after" chart-motion pattern — but Founder Decision 017 (2026-07-22, after that document was written) rejected exactly this class of staged, multi-stage reveal choreography for the Results page's hero sentence, on Experience Constitution grounds (§6: motion communicates state only, never spectacle) that apply with equal force to a chart.
- **Options Considered — plotting a `DecimalString`**: (1) Hand-roll pixel-position math in string-space (no `Number()` anywhere), computing each point's vertical offset via integer digit arithmetic on the decimal strings directly — rejected; positioning a mark on a continuous axis is inherently `(value − min) / (max − min) × height`, which is subtraction, division, and multiplication regardless of whether it's expressed via `Number()` or via hand-rolled string digit manipulation — the *letter* of "never derive a new value via arithmetic" would be violated by this approach too, just less legibly and auditable, while adding a large amount of bespoke, undertested arbitrary-precision math this project has no other use for. (2) Add a `none_as_null`-style disclosed exception directly inside `src/lib/format/`, updating `format.test.ts`'s static scan to carve out the new file — rejected; that test's entire purpose is to make `src/lib/format`'s "never touches numeric coercion" guarantee independently, mechanically verifiable, not just documented — weakening the scanner's own assertion to accommodate an exception is a bigger, more precedent-setting change than keeping the guarantee intact and putting the exception somewhere the scanner was never supposed to cover. (3) A single-purpose function, `toChartPlotNumber(value: DecimalString): number`, placed in a new file *outside* `src/lib/format/` (`src/components/simulation-result/chart-plot-value.ts`) — still inside an ESLint-watched directory (`src/components/**`), so it requires one explicit, narrowly-scoped `eslint-disable-next-line` comment at the single call site, with an extensive doc comment explaining exactly why the conversion is safe (chart geometry only, never displayed, never compared, never fed into a calculation; every displayed figure independently re-formats the *original* `DecimalString` via `src/lib/format`) — chosen.
- **Final Decision (plotting a `DecimalString`)**: Option 3. `chart-plot-value.ts` exports exactly one function, used in exactly one place (`growth-chart.tsx`'s `toPlotPoints`), with the disable comment pointing back to this ADR. `src/lib/format/format.test.ts`'s guardrail scan is untouched — it remains literally, unconditionally true for every file in that directory, with zero carve-outs. Every displayed value on the chart (tooltip, axis date ticks, the accessible data table in The Proof) independently formats the *original* `DecimalString` via `formatCurrency`/`formatDate`, never `toChartPlotNumber`'s output — verified directly by `chart-plot-value.test.ts` (the conversion itself) and `growth-chart.test.tsx` (every displayed figure traces to a real, formatted source).
- **Options Considered — reverse-split disclosure**: (1) Compute "1-for-N" by inverting the ratio (`1 / split_ratio`) — rejected outright; this is exactly the derived-financial-arithmetic this codebase's guardrails exist to prevent, and a reverse split is rare enough (no known example in this platform's real ingested data) that inventing an exception for it is not justified. (2) Disclose the ratio as-provided, without attempting the "1-for-N" phrasing, honestly labeled as a reverse split — chosen. `compareDecimalStrings(split_ratio, asDecimalString('1'))` (an ordering operation, already sanctioned) decides which phrasing branch to take; the ratio itself is never inverted, only compared and — for the forward case — display-rounded to a whole number via the existing `formatCurrency(..., { decimals: 0 })` path, exactly the same "display-precision rounding of an already-correct value" operation `roundDecimalString` was built for.
- **Final Decision (reverse-split disclosure)**: Option 2. A forward split (`ratio > 1`) reads "A 4-for-1 stock split occurred on {date}..."; a reverse split (`ratio < 1`) reads "A reverse stock split (ratio {split_ratio}) occurred on {date}...", showing the raw ratio rather than fabricating a "1-for-N" figure — an intentional, disclosed asymmetry, not an oversight, per `growth-chart.test.tsx::'honestly labels a reverse split rather than fabricating a "1-for-N" figure it cannot safely derive'`.
- **Options Considered — the dividend paragraph's missing third state**: (1) Derive an "asset paid none" signal from existing fields — e.g. comparing `shares_purchased` against a hand-computed `investment_amount / initial_price` baseline — rejected; this is division of two `DecimalString` values, the exact operation `src/lib/format/README.md` (ADR-029/033) forbids "for any reason, including 'just for display'," and the derived comparison would itself be exactly the kind of frontend-side financial calculation `.claude/CODING_STANDARDS.md`'s presentation-only rule exists to prevent — a UI branch chosen by a calculation is still a calculation. (2) Request a new backend field (a dividend-event count or boolean) — out of scope for a frontend-only task and not requested by Founder Decision 014's already-shipped `GET` contract; deferred, not silently dropped. (3) Implement only the two states the available data actually supports (`include_dividends` true/false), phrasing the "true" case with an explicit "any... if any occurred" hedge rather than asserting reinvestment definitely happened — chosen.
- **Final Decision (the dividend paragraph)**: Option 3. `dividendText()` (`results-sections.tsx`) never overclaims: "Any dividends {symbol} paid during this period were reinvested automatically..." is true whether zero or many dividend events actually occurred, closing the gap honestly rather than fabricating a state the API cannot support. Documented in the function's own doc comment (not just this ADR) so a future contributor adding the third state later finds the reasoning at the point of change, not only in this document.
- **Options Considered — chart motion**: (1) Follow `frontend_design_system.md` §8–9 literally — the line "draws in once" on first render, event markers "appear with a slight, deliberate delay after the line completes" — rejected; this is precisely the class of staged, sequenced reveal choreography Founder Decision 017 rejected for the hero sentence one milestone earlier, and the Experience Constitution's §6 rule ("motion communicates state only... the permanent rejection of theatrical treatment") does not carve out an exception for charts specifically — it is a general rule this document predates. (2) Treat Founder Decision 017 as governing the chart too (it postdates and supersedes the relevant frontend_design_system.md language, even though that document was never formally amended to say so), disabling Recharts' built-in line-draw animation entirely and applying only the same single ~200ms settle already used for the hero sentence — chosen, per this task's own direct instruction.
- **Final Decision (chart motion)**: Option 2. `<Line isAnimationActive={false} ...>` disables Recharts' default draw-in animation outright; the chart's outer wrapper uses the same `useSettleIn`/`useReducedMotion` pair `opening-sequence-heading.tsx` already established (ADR-041), producing one shared, consistent settle treatment across every animated element on the Results page rather than a second, chart-specific motion vocabulary. `frontend_design_system.md` §8–9's chart-motion language is now superseded for this specific case by Founder Decision 017, the same way ADR-041 itself superseded ADR-038 — noted here rather than left as a silent contradiction between two design documents.
- **Options Considered — the chart's accessible text alternative**: (1) An `sr-only` (visually hidden, always in the accessibility tree) full data table rendered directly beneath the chart — meets WCAG's letter but adds a large, dense table to every completed Results page's DOM even for users who never need it, in tension with EXPERIENCE_CONSTITUTION.md §5's progressive-disclosure default. (2) A short, visible, plain-English summary sentence (built from the same already-formatted first/last points and point count) immediately below the chart, functioning as the actual text alternative for a screen-reader user, paired with the full per-point table living inside The Proof's existing collapsed disclosure (never hidden, per that section's own standing rule) — chosen, matching this task's own "(or summarized description + table in The Proof)" alternative explicitly.
- **Final Decision (accessible text alternative)**: Option 2. The Recharts SVG wrapper is marked `aria-hidden` (redundant with the adjacent real text, not the sole source of the information); the summary sentence is ordinary visible/accessible prose; the full table (`GrowthDataTable` in `results-sections.tsx`) lives in The Proof, satisfying `frontend_design_system.md` §8/§11's "every chart ships a table-view fallback" without duplicating a large table on the primary reading path. Keyboard access to per-point data is served by this same table (a native `<table>`, fully keyboard-navigable) rather than a custom keyboard-driven Recharts tooltip interaction — Recharts ships no built-in keyboard-accessible tooltip, and building one was judged disproportionate scope for this task; the chart itself states this explicitly ("reachable without a mouse") so the equivalence is disclosed to the user, not just asserted in this document.
- **Options Considered — Provenance's "data source" field**: (1) Omit it — `SimulationResponse` doesn't carry it, and adding scope for one line item is arguably disproportionate. (2) Fetch it eagerly alongside the simulation itself, on every completed Results page load — rejected; a supplementary provenance detail should never delay or complicate the page's primary loading state (EXPERIENCE_CONSTITUTION.md §5's progressive-disclosure posture, and this project's now-standing precedent from `useAssetAvailability`/`useAssetDetail` for lazy, purpose-scoped fetches). (3) A new `useAssetDetail` hook, mirroring `useAssetAvailability`'s existing shape exactly, fetching `GET /api/v1/assets/{symbol}` only once The Proof's `<details>` is actually opened (tracked via the native `onToggle` event) — chosen.
- **Final Decision (Provenance's data source)**: Option 3. `useAssetDetail(symbol, enabled)` (`src/hooks/use-asset-detail.ts`) is `enabled` only by `TheProof`'s own `open` state; `getAssetDetail` (already implemented, pre-existing in `src/lib/api/endpoints/assets.ts`) required no new API-layer code, only a new query hook wiring it to React Query with the established key (`queryKeys.assets.detail`, already defined). Live-verified: the field renders "Loading…" transiently and then the real value once the query resolves, confirmed against the running backend.
- **Rationale**: Every choice above shares one property: where this codebase's existing guardrails (ADR-029, ADR-033, the progressive-disclosure posture of EXPERIENCE_CONSTITUTION.md §5, Founder Decision 017's motion ruling) collided with what a growth chart technically requires, the resolution kept the guardrail's *guarantee* intact and disclosed the narrow, unavoidable exception explicitly — in a doc comment, an ADR, and a test — rather than either quietly weakening the guardrail or silently working around it in a way a future reviewer would have to rediscover.
- **Tradeoffs**: The dividend paragraph's missing third state is a real, disclosed scope gap, not a hidden one — a future milestone that wants a true "this asset paid no dividends" state must add a backend signal for it, at which point this ADR's Option 3 should be revisited (Option 2, deferred here, becomes the live option). The reverse-split disclosure's "ratio {split_ratio}" phrasing is less polished than a hypothetical "1-for-N" figure would be, accepted as the honest alternative to a figure this codebase cannot safely derive. `toChartPlotNumber`'s existence means a *second* narrow numeric-coercion exception now exists in the frontend (after none previously) — kept to exactly one call site, by design, so a future `grep` for the function name finds the entire exception's surface area in one place.
- **Future Implications**: Any future chart in this codebase (a second series, a future comparison chart per `frontend_design_system.md` §8's "nominal vs. real" chart type) should reuse `chart-plot-value.ts`'s pattern — one disclosed, single-purpose, heavily-documented conversion function outside `src/lib/format/`, never a new ad hoc `Number()` call at the point of use — rather than each new chart inventing its own exception. `useAssetDetail`'s open-triggered lazy-fetch pattern is now this project's second precedent (after `useAssetAvailability`'s selection-triggered fetch) for "fetch only once the user's own action indicates they want this data," and should be the default shape for any future supplementary, non-blocking data need on an already-loaded page.
- **Related**: Founder Decision 013/017 (Experience Philosophy, motion ruling), ADR-041 (the settle-in motion pattern this chart reuses), ADR-042 (the `growth_series`/`disclosed_splits` persistence this chart's data depends on), `docs/frontend_design_system.md` §8–9 (superseded for chart motion specifically, per above), `src/lib/format/README.md` (ADR-029/033, the guardrail this ADR's central exception is disclosed against).

## ADR-044 — M7 Phase 3D Design Elevation: Font Self-Hosting, Elevated Token Architecture, the Scramble Hook, and What Was Deliberately Not Ported

- **Date**: 2026-07-25
- **Status**: Accepted
- **Context**: The founder explored visual directions in an external design tool and approved a final mockup as the product's new visual source of truth (`DOCUMENTS_AND_IDEAS/Investment Time Machine Results/Investment Time Machine.dc.html` — see the Founder Decision 018 Context note on this file's actual path and untracked/uncommitted status, a documentation-hygiene gap this ADR does not resolve and has no git authority to fix). This ADR is the engineering record for implementing that approved direction across tokens, the Results Experience, and the Simulator, per Founder Decision 018's motion/atmosphere rules.
- **Problem**: Six engineering questions the mockup approval and Founder Decision 018 leave to implementation: (1) how to self-host three new Google Fonts (Newsreader, IBM Plex Mono, Public Sans) without an external font CDN at runtime; (2) how to introduce a materially different dark-mode palette (oklch-based, warm near-black gradient, single gold accent) without either rewriting every existing token consumer app-wide or scoping the change so narrowly it can't satisfy "app shell" cohesion (STEP 4); (3) how to implement FD-018's JS-driven digit-scramble without violating this codebase's `no-restricted-syntax` relational-operator guardrail or `react-hooks/set-state-in-effect`; (4) whether the Growth Chart's data-line color should switch to the mockup's warm accent or stay the CVD-validated portfolio blue; (5) how the invested-amount baseline reference line's Y-position should be computed given the chart's existing single-call-site `toChartPlotNumber` exception (ADR-043); (6) what, if anything, from the mockup's own design exploration should be explicitly excluded rather than silently ignored.
- **Options Considered — font self-hosting**: (1) Load Newsreader/IBM Plex Mono/Public Sans from Google Fonts' CDN via a `<link>` tag — rejected outright by direct instruction ("self-host via next/font, no external font CDN at runtime") and by this project's own standing preference for no uncontrolled third-party runtime dependencies. (2) `next/font/google` (already the mechanism `layout.tsx` used for Inter/JetBrains Mono) — chosen; downloads and self-hosts the font files at build time, serving them from the app's own origin with zero runtime request to Google.
- **Final Decision (font self-hosting)**: Option 2. `layout.tsx` now imports `Public_Sans`, `IBM_Plex_Mono` (weights 400/500/600), and `Newsreader` (`style: ['normal', 'italic']`, for the Why section's italic sub-heading and the mockup's italic asset-symbol treatment) via `next/font/google`, each bound to a CSS variable (`--font-public-sans`, `--font-ibm-plex-mono`, `--font-newsreader`) applied on `<html>`. `globals.css`'s `@theme inline` block redirects `--font-sans`/`--font-mono` to the new fonts (replacing Inter/JetBrains Mono outright, not adding a second parallel stack) and adds a new `--font-serif` key Tailwind's `font-serif` utility now resolves to. **Bundle-size note** (STEP 5): a full production build (`.env.local` moved aside, CI-parity) emits 24 self-hosted `.woff2` subset files under `.next/static/media/`, totaling 373,728 bytes (~365 KB) across all three families' declared weights/styles combined — this is the aggregate build output, not a single page's network cost; a browser only fetches the specific weight/style/subset files its own rendered text actually needs (`next/font`'s generated `@font-face` rules are per-weight/style), typically 2–4 files for any one page in practice.
- **Options Considered — elevated palette architecture**: (1) Overwrite the existing dark-theme values of `--color-paper-dark`/`--color-ink-*-dark`/`--color-hairline-dark` in place with the mockup's oklch values — would satisfy "app shell" cohesion automatically (every existing dark-theme consumer app-wide picks it up for free) but directly contradicts "leave the stale root page as-is": the root `/` page and `/dev/playground` would silently inherit the new palette too, with no way to exempt them short of a second override layer. (2) A parallel, additive `--color-elevated-*` primitive namespace (never removing or renaming the existing `-dark`/`-light` primitives) plus a single new scoped class, `.itm-elevated` (`globals.css`), that locally re-overrides the semantic custom-property names (`--color-background`, `--color-ink-primary/secondary/muted`, `--color-border-hairline`, `--color-primary`/`--color-primary-foreground`, plus two new tokens, `--color-accent` and `--color-negative-tint`) within whatever DOM subtree it wraps — chosen. Applied to the Results (`simulation-result-client.tsx`) and Simulator (`simulator/page.tsx`) page-level `<main>` wrappers only.
- **Final Decision (elevated palette architecture)**: Option 2. Because every existing component already reads color exclusively through the semantic custom-property names (never a primitive directly — an invariant this codebase already enforced before this pass), wrapping a subtree in `.itm-elevated` is sufficient to re-skin every descendant (Card, Button, Badge, Input, the disclosure `<details>` elements) with zero component-level changes — the same "layer, don't rewrite" principle `semantic.css`'s own light/dark switch already uses, one level more scoped. `.itm-elevated` also remaps `--color-primary`/`--color-primary-foreground` to the accent pair (so the Simulator's existing `<Button variant="primary">` becomes the mockup's gold CTA with no `Button` component change at all) and `--button-radius` to the mockup's sharper 2px corner. The root `/` and `/dev/playground` pages are never wrapped in `.itm-elevated` and are therefore visually untouched by this pass, satisfying the "leave the stale root page as-is" instruction literally rather than by convention.
- **Options Considered — the chart's data-line color**: (1) Recolor `--color-chart-portfolio` to the mockup's warm gold/amber accent, matching the mockup pixel-for-pixel. (2) Keep `--color-chart-portfolio` (the existing CVD-validated blue) as the line/fill/endpoint-marker color, porting only the mockup's chart *treatment* (gradient area fill, invested-amount baseline, endpoint label, mono ticks) — chosen.
- **Final Decision (chart color)**: Option 2. `docs/BRAND_CONSTITUTION.md` §7 and `docs/frontend_design_system.md` §3/§8 both treat the chart-data palette as a closed, CVD-validated set a UI accent must never be substituted into ("reusing a chart hue for UI chrome, or inventing a new chart hue outside the validated set" is named explicitly as a misuse in both documents) — and this task's own STEP 3 instruction list, when it says "keep... exactly as they are" for the chart's other mechanics, never asks for a data-color change specifically. Recoloring the chart line was judged to be exactly the class of visual-identity decision those two documents flag as requiring deliberate governance beyond a single implementation pass's own judgment call, so the existing validated hue was kept and the mockup's *structural* treatment was ported around it instead. This is a disclosed, deliberate partial-port, not an oversight — flagged again in the Tradeoffs section below.
- **Options Considered — the scramble hook's guardrail compliance**: (1) Use the rAF callback's own timestamp argument for elapsed-time math (matching the mockup's own JS verbatim) — this was the first implementation attempted, and it silently never completed under jsdom: a real bug, caught by this pass's own new `use-scramble.test.ts` (`waitFor` timing out on an unresolved animation), traced to the rAF callback's `now` argument and a separately-called `performance.now()` not sharing the same epoch in this specific test environment. (2) Call `performance.now()` again inside every rAF callback rather than trusting the callback's own timestamp parameter — chosen; portable, spec-safe, and the fix that made the animation actually reach completion under test. Relational-operator comparisons (`i < locked`, the `for` loop bound) required individually justified `eslint-disable-next-line no-restricted-syntax` comments, matching this codebase's existing per-line-disclosure convention (ADR-033) — none of these are `DecimalString` comparisons, all are animation-progress or character-index arithmetic. The `!active` branch's synchronous `setState` inside `useEffect` (flagged by `react-hooks/set-state-in-effect`) was removed entirely rather than suppressed: mirroring `useSettleIn`'s own established pattern (ADR-041), `active` is captured once via a lazy `useState` initializer, and when reduced motion is preferred, that initializer alone already produces the final target text — no effect body needs to run at all for that path.
- **Final Decision (scramble hook)**: `src/hooks/use-scramble.ts` — one hook, `useScramble(target, active, { duration, delay })`, returns `{ text, glow }`. Mirrors `useSettleIn`'s once-per-mount capture pattern exactly (same `useState(active)` lazy-capture idiom, same "no re-trigger on a later prop change" contract, same reduced-motion hard gate). Called from `opening-sequence-heading.tsx` (the two hero figures, delays 0/100ms, duration 600ms) and `results-sections.tsx`'s `Fact` component (the three Supporting Facts figures, delays 0/100/200ms, duration 450ms) — the exact per-figure duration/stagger budget FD-018 rule 1 sets, and the same cadence shape the mockup's own `componentDidMount` schedule used (each successive figure starting 100ms after the last).
- **Options Considered — the invested-amount baseline's plot value**: (1) Add a second, independent numeric-coercion function for this one value, outside `chart-plot-value.ts` — rejected; would create a second, differently-audited exception surface where ADR-043 established exactly one. (2) Reuse the existing, already-audited `toChartPlotNumber` a second time, at a second call site within `growth-chart.tsx` — chosen.
- **Final Decision (baseline plot value)**: Option 2. `growth-chart.tsx` now calls `toChartPlotNumber(sim.investment_amount)` once more, for the `ReferenceLine`'s `y` position only — never displayed; the on-chart label text is `formatCurrency(sim.investment_amount, { decimals: 0 })`, the original `DecimalString`, matching every other displayed figure on this chart. ADR-043's "kept to exactly one call site" claim is superseded by this ADR to "exactly one function, now two call sites, both narrowly scoped to chart-geometry-only Y/position values within `growth-chart.tsx`" — the underlying safety argument (chart geometry only, never displayed, never compared, never fed into a calculation) is unchanged and still holds at the new call site.
- **Options Considered — what to exclude from the mockup**: The mockup's containing folder also holds `Dark Pool Liquidity Tide.dc.html`, an unrelated, non-approved atmosphere exploration using `bandDrift`/`whaleDrift`/`rippleEcho` infinite-loop `@keyframes` — an animated-background concept, not part of the founder's approved direction. (1) Silently ignore it (never mention it, since it isn't the approved file) — rejected; Founder Decision 018 rule 4 and this task's own STEP 6 instruction both require it be named explicitly, not just quietly excluded, so a future contributor who finds the file doesn't have to re-derive why it was never ported. (2) Name it explicitly, in both Founder Decision 018 and here, as the "discarded looping background" — chosen.
- **Final Decision (discarded looping background)**: Option 2. `Dark Pool Liquidity Tide.dc.html`'s infinite-loop drift/ripple animations are never ported, referenced, or reproduced anywhere in this implementation — FD-018 rule 4's "no infinite/looping animation of any kind, anywhere" is a direct, explicit rejection of this specific file's approach, not a generic restatement of prior motion policy. The approved mockup's own atmosphere (a static gradient wash plus a static fractal-noise grain texture, `--grain-texture` in `primitives.css`) is the only background treatment implemented.
- **Rationale**: Every choice above follows the same pattern this project's prior design-system passes established: extend the existing token/hook architecture additively (parallel elevated primitives, a scoped class, a hook mirroring an established sibling hook's contract) rather than rewriting it in place, and where a mockup's literal treatment collides with an already-documented governance rule (the CVD-validated chart palette) or an already-shipped guardrail (`no-restricted-syntax`, `toChartPlotNumber`'s single-audited-function design), resolve in favor of the existing rule with the deviation explicitly disclosed — never silently overridden and never silently dropped.
- **Tradeoffs**: The chart's data line stays validated-blue while nearly everything else on the Results/Simulator surfaces uses the mockup's gold accent — a deliberate, disclosed visual inconsistency with the mockup itself, not a hidden gap; if a future founder review specifically wants the chart recolored to match, that is a new, separate visual-identity decision this ADR does not make on its own. `docs/BRAND_CONSTITUTION.md` §6 (typography) and `docs/frontend_design_system.md` §4 ("no serif or display face anywhere") are now factually superseded for the Results/Simulator surfaces by this pass's font change, but neither document was edited in this pass — out of this task's explicit STEP list, and a large enough edit (a full brand-typography rewrite) to warrant its own dedicated pass rather than a side effect of this one; tracked as a new `docs/KNOWN_ISSUES.md` entry (see below) rather than left as a silent contradiction between two design documents. The mockup source file itself remains untracked/uncommitted in this repository (no git authority available to this implementation pass to fix that) — also tracked in `docs/KNOWN_ISSUES.md`.
- **Future Implications**: If a future pass wants the chart's data-color to actually adopt the mockup's accent, that decision should re-run the dataviz skill's CVD-safety validation against the new hue before shipping it (the same rigor the original blue received), not simply copy the mockup's hex/oklch value. `chart-plot-value.ts`'s "one function, multiple disclosed call sites" pattern (this ADR) is now the template for any future chart-geometry-only value that isn't a plotted data point itself (an average line, a target line, any other reference value) — extend the same function's call sites rather than inventing a second conversion utility. `.itm-elevated`'s scoped-remap pattern (re-pointing semantic custom properties within a wrapped subtree) is now this project's template for any future "this one surface looks different from the rest of the app" requirement, in preference to either a global token rewrite or a component-by-component visual fork.
- **Related**: Founder Decision 018 (Motion & Atmosphere Amendment — the motion rules this ADR's `useScramble`/section-entrance implementation satisfies), Founder Decision 013/017, ADR-041 (`useSettleIn`, the pattern `useScramble` mirrors), ADR-043 (`toChartPlotNumber`, extended to a second call site here), `docs/BRAND_CONSTITUTION.md` §6–7 (typography/color — factually superseded for the elevated surfaces, not yet edited), `docs/frontend_design_system.md` §3–4/§8 (chart palette / typography — same), `docs/KNOWN_ISSUES.md` (new entries: the mockup's untracked/mis-pathed source, and the BRAND_CONSTITUTION/frontend_design_system.md documentation-sync gap).

## ADR-045 — M7 Phase 3D-1 Craft & Coherence: The ProductShell Full-Bleed Architecture, the Shared Disclosure Primitive, the Custom-Property Re-Declaration Bug, and the Chart's Domain/Label-Position Fixes

- **Date**: 2026-07-25
- **Status**: Accepted
- **Context**: M7 Phase 3D (Design Elevation) shipped a visual direction but left individual component states, a product shell (header/footer), and several chart-craft details under-specified — this pass ("Craft & Coherence") closes that gap under FD-018's existing motion law, without a new Founder Decision (no new philosophy is introduced; this is implementation-level craft work applying already-approved rules more completely). This ADR is the engineering record for the choices left to implementation, several of which were only discovered by actually running the app in a real browser via Playwright (available in this environment, previously assumed absent) rather than static code review.
- **Problem**: Five engineering questions this pass's own verification pass surfaced: (1) how to make the elevated atmosphere genuinely full-bleed (no black margins at wide viewports) without duplicating `.itm-elevated` per page; (2) how to unify five different native-`<details>`-based disclosures into one animated, accessible primitive; (3) why a CSS custom property explicitly declared as `var(--color-primary)` rendered the *wrong* color inside `.itm-elevated` despite `--color-primary` itself being correctly overridden there; (4) why the Growth Chart wasted up to half its vertical space and why its invested-amount baseline label visually collided with the price line on a loss trajectory specifically; (5) why a live dividend-reinvestment simulation against the newly-extended `dev_seed` fixture (backend, task F) failed outright.
- **Options Considered — full-bleed atmosphere**: (1) Keep `.itm-elevated` on each page's own `max-w-[...]` content container, and separately paint a duplicate full-width background behind it — two things to keep in sync, and the exact "visible seam" bug this pass was tasked with fixing. (2) Move `.itm-elevated` to a new shared wrapper, `ProductShell` (`src/components/shell/product-shell.tsx`), composing `AppHeader` + `<main>` + `AppFooter`, with the page's own max-width applied to `<main>` only — chosen. `.itm-elevated`'s existing CSS (unchanged) already paints its background/grain via `position: relative` + `::before`/`::after` pseudo-elements sized to whatever element it's placed on; moving the class from a `max-w-[1120px]` `<main>` to a `min-h-screen w-full` outer wrapper required zero CSS changes, only a component-tree change.
- **Final Decision (full-bleed atmosphere)**: Option 2. `ProductShell` is used by `/simulator` and both branches of `/simulation/[id]` (all four states: loading skeleton, error, completed, pending/failed) — every product route now shares one shell, one atmosphere-painting element, and one header/footer. The root `/` and `/dev/playground` pages remain unwrapped by construction, not convention, satisfying "leave the stale root page as-is" literally.
- **Options Considered — unifying disclosures**: (1) Leave five independent native-`<details>` implementations, each with its own ad hoc "+"/chevron treatment — the status quo this pass was tasked with fixing. (2) A shared component built on native `<details>`/`<summary>`, adding only CSS for a rotating marker — rejected; `<details>` offers no CSS hook for animating its own open/close reveal (the content simply snaps), so a real height/opacity transition is not achievable without abandoning the native element. (3) A shared component, `Disclosure` (`src/components/ui/disclosure.tsx`), built from a real `<button aria-expanded>` + a controlled `<div>` using the CSS grid `0fr`→`1fr` row-track technique (the standard way to animate to an intrinsic "auto" height without JS-measured pixel heights) plus an opacity fade — chosen.
- **Final Decision (unifying disclosures)**: Option 3. `inert` (native, zero extra JS, supported in every evergreen browser this project targets) keeps collapsed content out of the tab order and accessibility tree without unmounting it — so the transition always animates real, present content, never a remount, and screen readers never encounter a collapsed disclosure's interactive contents. All five existing disclosures (Supporting Facts' "Source," The Proof, the Simulator's "More options," Results' "Technical details," and `ErrorState`'s own "Technical details") now use this one component; tests updated from `.closest('details')`/native-`toggle`-event assertions to `getByRole('button', { name })`/`aria-expanded` assertions, which are also simpler and more semantically direct than what they replaced.
- **Options Considered — the custom-property bug**: `globals.css`'s `.itm-elevated` block re-declares `--color-primary: var(--color-accent-*)` so the Simulator's CTA button and Results' text links automatically become accent-colored with zero per-component change (ADR-044's own established pattern). `components.css` separately declares, once, at plain `:root`: `--focus-ring-color: var(--color-primary); --input-border-focus: var(--color-primary);`. Live keyboard-Tab verification (a real focus event, not a synthetic `.focus()` call, which does not reliably trigger `:focus-visible` matching) showed every focused input inside an elevated surface rendering a *navy* outline, not accent gold — confirmed via `getComputedStyle` that `--focus-ring-color` resolved to `#3e6bb0` (the base navy hex) while `--color-primary`, queried on the exact same element, correctly resolved to the accent color. (1) Assume this was a browser inconsistency and move on — rejected; reproduced identically after a hard reload, ruling out a stale-cache artifact. (2) Root-cause via the CSS custom-properties specification: a `var()` reference inside a custom-property declaration is resolved once, at that declaration's own computed-value time, at the element the declaration matches — the *resolved* value then inherits down the tree as an ordinary (now-static) value. `--focus-ring-color`'s only declaration is at plain `:root`; there, `var(--color-primary)` resolves against `:root`'s own cascade-winning `--color-primary` (navy, in dark mode, since `.itm-elevated`'s override doesn't apply to `:root` itself). That resolved navy value then inherits unchanged into `.itm-elevated` and every descendant, because nothing re-declares `--focus-ring-color` at that more specific scope — re-declaring `--color-primary` alone does not make an already-declared-elsewhere property "dynamically re-follow" it. This is a general property of CSS custom properties, not a Tailwind- or React-specific quirk.
- **Final Decision (the custom-property bug)**: `--focus-ring-color` and `--input-border-focus` are now explicitly re-declared inside all three `.itm-elevated` blocks (the light default, the `prefers-color-scheme: dark` override, and the explicit `[data-theme='dark']` override), pointing directly at `--color-accent-light`/`--color-accent-dark` — mirroring exactly how `--color-primary`/`--color-primary-hover`/`--color-primary-foreground`/`--button-radius` are already re-declared there. Documented in `docs/frontend_design_system.md` §15 as a standing rule: any future token that composes another token via `var()` and needs to vary between the base and elevated scopes must be re-declared at `.itm-elevated` directly, not assumed to "inherit the composition."
- **Options Considered — the chart's wasted space and label collision**: Live screenshots (Playwright, both light and dark, gain and loss scenarios) showed two related defects neither unit tests nor static review had caught: the Y-axis auto-included a zero baseline regardless of the actual data range (Recharts' `Area` component defaults to a zero-based domain), wasting up to half the chart's height on a series that never approached zero; and the invested-amount reference line's label (`position: 'insideTopLeft'`), which worked fine when the baseline sat near the bottom of the domain (a gain), visually collided with the price line's own starting point on a loss trajectory, where the invested amount *is* the series' own maximum and therefore sits at the domain's upper edge with minimal headroom for a label pushed "up and away." (1) Leave the zero-baseline domain and add only more top/bottom margin — rejected; margin pads the chart's outer edges, not the *proportion* of height the actual data occupies, so it does not fix wasted internal space. (2) An explicit, padded Y-axis `domain` (`[(min) => Math.floor(Math.min(min, investedPlotValue) * 0.92), (max) => Math.ceil(Math.max(max, investedPlotValue) * 1.08)]`) that always includes the invested-amount baseline (so `ifOverflow="extendDomain"` on the `ReferenceLine` becomes a defensive fallback, not the primary mechanism) plus a dynamic label-position flip (`isNetGain ? 'insideTopLeft' : 'insideBottomLeft'`, mirroring the endpoint label's own existing `endpointLabelFlipped` pattern) so the label always renders toward whichever side of the reference line actually has room — chosen, and re-verified live after implementation (both a gain and a loss chart screenshotted cleanly, full height used, no label collision).
- **Final Decision (chart domain/label)**: Option 2 for both. `investedPlotValue`/`last.plotValue` (already-disclosed `toChartPlotNumber` outputs — chart geometry only, never displayed) are compared with plain relational operators for this positioning decision, the same category of narrow, disclosed exception this file's own `wasDecimated`/`endpointLabelFlipped` calculations already use.
- **Options Considered — the dividend-date/price-row mismatch**: A live end-to-end simulation against the newly-added KO dividend fixture (backend task F) failed with `MISSING_HISTORICAL_DATA` on `2020-02-15` — a Saturday. `DevSeedProvider.fetch_dividends` generated ex-dividend dates on a fixed "the 15th" of each quarter month regardless of day-of-week, while `fetch_prices` only emits weekday rows (matching how real equity data has no weekend rows) — the Simulation Engine requires an exact price row on a dividend's own ex-dividend date to reinvest it. (1) Leave the fixed 15th and accept that some years' dividend-reinvestment simulations against KO would fail — rejected outright; a fixture whose own documented "dividend payer" scenario cannot reliably complete a simulation defeats the entire purpose of adding it. (2) Shift any weekend-landing date back to the preceding Friday (`ex_date -= timedelta(days=weekday - 4)` when `weekday >= 5`) — chosen; deterministic, and verified for every seeded year (2020–2024) to land on a real weekday.
- **Final Decision (dividend dates)**: Option 2, implemented in `dev_seed_provider.py::fetch_dividends`, with a new regression test (`test_every_dividend_date_falls_on_a_trading_day_and_has_a_matching_price_row`) asserting every generated dividend date is both a weekday and exactly matches one of that same symbol's own generated price dates — not just "not a weekend," the stronger and more directly relevant guarantee. The already-seeded local dev database's 20 incorrectly-dated KO dividend rows were deleted and re-seeded with the corrected dates as part of this pass's live verification (a one-time local data correction, not a schema or migration change).
- **Rationale**: Every fix in this ADR was found by actually exercising the running application — three of the five (the custom-property bug, the chart's wasted space/label collision, the dividend weekend-date mismatch) were invisible to `tsc`/`eslint`/the unit test suite/static code review, and would have shipped as real, user-visible defects (a wrong-colored focus ring, a chart reading as less precise than intended, a fixture scenario that silently couldn't complete) without a live check. This is the same "a passing test suite and a correct live render are not the same claim" discipline this project has applied before (KI-043, KI-046) — reconfirmed here for a frontend-CSS-cascade class of bug this project had not previously encountered.
- **Scope**: Governs `src/components/shell/*.tsx` (new), `src/components/ui/disclosure.tsx` (new), `src/app/globals.css`'s `.itm-elevated` block, `src/components/simulation-result/growth-chart.tsx`'s Y-axis domain and reference-line label logic, and `backend/app/ingestion/providers/dev_seed_provider.py::fetch_dividends`.
