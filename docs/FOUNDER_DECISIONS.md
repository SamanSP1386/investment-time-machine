# FOUNDER_DECISIONS.md

Founder Decisions record moments where the founder explicitly overrides, amends, or resolves an ambiguity in the Founder Specification itself. They differ from an ADR (`ARCHITECTURE_DECISIONS.md`): an ADR documents an *engineering* decision made within the bounds of the spec; a Founder Decision documents a change to what the spec itself requires, made by the one person with the authority to change it. Every Founder Decision should also generate a corresponding ADR explaining the engineering implementation of that decision. Never edit a previous Founder Decision — if a decision changes, record a new one that supersedes it, per [.claude/DOCUMENTATION_POLICY.md](../.claude/DOCUMENTATION_POLICY.md).

## Founder Decision 001 — Simulation Engine Uses `close_price`, Not `adjusted_close_price`

- **Date**: 2026-07-08
- **Status**: Approved
- **Context**: The Founder Specification's own text, in three separate places (Part 2.6.7, Part 2.6.20, Part 2.6.22), states that the Simulation Engine "shall primarily use adjusted close prices" for MVP, with raw price fields and `stock_splits` retained only "for transparency and future advanced simulations." Part 2.14 (Simulation Engine Architecture) itself is silent on which specific price column its formulas use, referring only to abstract `initial_price`/`final_price` terms.
- **Decision**: The Simulation Engine (starting at Milestone 3) uses **`close_price`** as the sole price input to all growth, return, and CAGR calculations — not `adjusted_close_price`. The engine explicitly:
  - Processes dividend events from the `dividends` table and calculates dividend reinvestment explicitly, share by share, event by event.
  - Preserves `adjusted_close_price` for validation, comparison, and audit purposes only — it is ingested and stored as before, and may be surfaced as a cross-check figure, but is never read by the calculation path.
  - Preserves `stock_splits` primarily for transparency and auditing during MVP — split events are disclosed alongside results but do not drive share-count math (see `docs/simulation_formulas.md` §3 for the technical reasoning: raw `close_price` is already retroactively split-adjusted by the data provider, so applying `split_ratio` on top would double-adjust).
  - Avoids any dividend or split double-counting by construction (single code path for each, documented in `docs/simulation_formulas.md` §7).
- **Rationale**: `adjusted_close_price` bundles an opaque, provider-defined adjustment (the Founder Specification itself notes dividend adjustment within it is "provider dependent," Part 2.6.20) into a single number that cannot be decomposed or audited after the fact. Building the engine around explicit, stored, per-event dividend and split records instead means every dollar of return can be traced to a specific historical event — directly serving the Founder Specification's own stated priorities for the Simulation Engine (Part 2.14.1: Accuracy, Reproducibility, Auditability, Explainability) and its "Full auditability" approved decision (Part 2.14.19). Notably, this is not a deviation the specification's own text finds objectionable in principle — Part 2.6.20 and Part 2.6.22 both describe raw-price-based calculation as the natural destination for "future advanced simulations," and this decision simply brings that destination forward to be the MVP model rather than a later one.
- **Scope**: This decision governs simulation calculation only. It does not change the M1 database schema (both `close_price` and `adjusted_close_price` continue to be stored and ingested exactly as before) and does not change the Data Ingestion Pipeline (M2) in any way.
- **Deferred**: A full Advanced Corporate Actions Engine (handling spin-offs, mergers, special dividends, and incremental-reimport-safe split reconciliation) is explicitly out of scope for the current model and reserved for a future milestone.
- **Tradeoffs**: More implementation complexity now (an explicit event-processing loop vs. a single price lookup) in exchange for auditability and transparency the specification identifies as core platform values. Carries one verification obligation before production use: the assumption that raw `close_price` is already split-consistent within a single ingestion fetch has not yet been empirically verified in this codebase and must be confirmed by a known-answer test against a real historical split during M3 (see `docs/simulation_formulas.md` §3 and §6).
- **Related**: ADR-015 (`docs/ARCHITECTURE_DECISIONS.md`), `docs/simulation_formulas.md`, closes `docs/KNOWN_ISSUES.md` KI-008.

## Founder Decision 002 — Identity Management: Token, Cookie, Anonymous-Use, and Lifecycle Model

- **Date**: 2026-07-11
- **Status**: Approved
- **Context**: The Founder Specification names `JWT_SECRET` as a required environment variable and states "Authentication -> JWT" (Part 2.15.3), but specifies no token lifetime, refresh mechanism, revocation/logout behavior, cookie security attributes, CSRF stance, account-lockout parameters, or password-reset/email-verification flow anywhere in its 408 pages (confirmed by exhaustive term search during the M5 design review — see `docs/KNOWN_ISSUES.md` KI-006). Separately, Part 2.8.8 ("Create Simulation... should require authentication") and Part 2.6.24/2.17.7 (anonymous simulations explicitly allowed, and the founder's own MVP-options text preferring accounts "if development effort remains reasonable") are in direct tension — already tracked as KI-022, resolved at M4 as public/anonymous with rate limiting, but not yet re-confirmed with the fuller citation trail the M5 design review surfaced.
- **Decision**: The founder approved, as project policy, the full Identity Management design proposed in the M5 design review:
  - **Authentication**: email + password only (Argon2). OAuth and MFA remain explicitly deferred, matching Part 2.8.4/2.8.18.
  - **Tokens**: short-lived (15-minute) stateless JWT access token + opaque, database-backed refresh token (30-day sliding expiration) with rotation-on-every-use and reuse detection (a replayed, already-rotated token revokes every active session for that user).
  - **Cookies**: both tokens delivered exclusively via httpOnly, Secure, SameSite=Strict cookies — never in a JSON response body, never in `localStorage`.
  - **Anonymous use, re-confirmed**: anonymous users may run simulations, view results, and share simulation links; they may not save simulation history, generate AI reports, create portfolios, or sync across devices. This re-confirms and narrows KI-022's M4 resolution with an explicit permission boundary, rather than leaving "anonymous vs. authenticated" as an unstated implication.
  - **Simulation History**: requires authentication (unchanged from KI-023's deferral — the middleware now exists, but the history-listing endpoint itself remains unbuilt, out of M5's explicit scope).
  - **Roles**: exactly two — User, Admin — matching Part 2.8.6 precisely, no additional roles.
  - **User lifecycle**: Anonymous -> Registered -> Active -> Suspended -> Soft Deleted. Users are never hard-deleted (matches the existing `is_active` soft-deactivation design from M1).
  - **Session lifecycle**: Login -> Issue Access Token -> Issue Refresh Token -> Refresh -> Logout -> Revoke.
  - **Explicitly deferred past M5** (not excluded, not forgotten): OAuth, MFA, email verification, password reset/account recovery. This is a deliberate, founder-approved sequencing decision — `.claude/SECURITY_POLICY.md`'s prior recommendation to resolve password reset "before Auth milestone completion" is superseded by this decision for M5's scope specifically; password reset must still ship before any production launch (tracked in `docs/KNOWN_ISSUES.md`), just not as part of this milestone.
- **Rationale**: Every parameter above fills a confirmed Founder Specification silence (KI-006) with an explicit, deliberate choice rather than an unstated engineering default — matching this project's standing rule that AI-authored code must not resolve a security-relevant spec gap unilaterally. Rotation-with-reuse-detection and httpOnly/Secure/SameSite=Strict cookies are the two decisions doing the most security work: the former means a stolen-but-not-yet-used refresh token is useless after the legitimate user's next refresh, and the latter means no JavaScript (including an XSS payload) can ever read either token, and cross-site requests cannot carry the session cookie at all.
- **Scope**: Governs Identity Management (M5) only. Does not change the Simulation Engine (M3), which remains fully independent of authentication (Founder Specification Principle 4) — `user_id` continues to flow through as an optional, opaque value, never a control-flow dependency.
- **Deferred**: OAuth (Google/GitHub), MFA, email verification, password reset, and multi-device session management UI (individual/all-device logout) are all explicitly out of scope for M5 — the underlying schema and repository primitives for multi-device support are included now (see ADR-017) precisely so none of these require a schema change later.
- **Related**: ADR-017 through ADR-020 (`docs/ARCHITECTURE_DECISIONS.md`), `docs/MILESTONE_REPORTS/M5_REPORT.md`, resolves `docs/KNOWN_ISSUES.md` KI-006, re-confirms KI-022.

## Founder Decision 003 — Educational AI System (M6): Renamed from "AI Analyst," Scope, Provider, Privacy, Caching, and Integrity Rules

- **Date**: 2026-07-12
- **Status**: Approved
- **Context**: The M6 design review (conducted before any code was written, per this project's design-review-then-implement process established at M3/M5) identified several genuine Founder Specification silences that a prior milestone's precedent doesn't resolve: Part 2.7.6/2.7.7 name "Historical Event Database" and "Educational Knowledge Base" as approved AI inputs, but neither exists in the M1–M5 schema, and Part 2.7.14 itself frames a knowledge corpus as future RAG work, not MVP-mandatory; Part 2.7.15 names OpenAI/Anthropic/Google/Local Models as candidate providers without selecting one; and no milestone before this one has sent any platform data to a third-party commercial vendor, an entirely new category of decision the Founder Specification does not address at all.
- **Decision**: The founder approved, as project policy, the full M6 design proposed in the design review, with the milestone itself renamed from "AI Analyst" to **M6 — Educational AI System** to better reflect its two approved components:
  - **Core philosophy**: "The Simulation Engine calculates. The AI teaches." — restated as the operative one-line test for every implementation decision in this milestone.
  - **Explanation Engine**: generates one structured educational explanation per completed simulation, covering ROI, CAGR, inflation adjustment, dividends, and time horizon — never calculating, modifying, forecasting, recommending, or inventing a financial fact.
  - **Financial Tutor**: supports educational follow-up questions scoped to exactly one simulation's own context — no long-term memory, no cross-user memory, no unrelated financial advice.
  - **Required output structure**: Summary, What Happened, Why It Happened, Financial Concepts, Key Takeaways, Limitations, Educational Disclaimer. Implemented with one disclosed hardening beyond the literal instruction: the Educational Disclaimer is a fixed, code-appended string (`app.ai.service.EDUCATIONAL_DISCLAIMER`), never something the model is asked to generate — a prompt instruction is a request, a fixed string is a guarantee, and this is strictly safer for a compliance-relevant section.
  - **Privacy allowlist**: the AI provider may never receive email, display name, user ID, IP address, session ID, request ID, or auth information. Allowed inputs are exactly: asset symbol, investment amount, start/end date, simulation outputs, dividend/inflation options, and the follow-up question text itself when applicable.
  - **Provider strategy**: Anthropic is the first configured provider, behind a `Protocol`-based abstraction (`app.ai.providers.base.AIProvider`, mirroring the ingestion milestone's capability-protocol precedent, ADR-013) with a `NullProvider` fallback selected by default (`AI_PROVIDER=none`) — no vendor SDK import exists outside `app/ai/providers/anthropic_provider.py`.
  - **Cost control**: explanations are cached and never re-generate the model call when `simulation_id` + `prompt_version` + the configured model are unchanged (a prior *failed* attempt is also served from cache for the Explanation Engine specifically, since it has an explicit `regenerate` override; a Financial Tutor follow-up re-tries automatically on a prior failure instead, since it has no such override); a per-simulation regeneration cap and a separate per-simulation follow-up-question cap are both enforced; output tokens are capped; `model_name`/`prompt_version`/`input_summary` are tracked on every row, matching Part 2.7.10–2.7.12 exactly.
  - **AI integrity check**: every generation is validated post-hoc for (1) the six required section headers being present, (2) every numeric token in the narrative tracing back to the simulation's own structured output within a small tolerance, and (3) no directive investment-advice language. A failure on any check is rejected outright — the row is marked `failed`, `explanation_text` stays `NULL` forever, and the unsafe content is never returned or persisted.
  - **AI availability**: identical to the founder-approved fallback text — "Simulation completed successfully. AI explanation is temporarily unavailable. Your financial results remain accurate." — used for every failure category (provider unavailable, integrity violation, structure violation, unsafe content), returned as a normal successful HTTP response, never an error envelope.
  - **Explicitly out of scope for M6** (per direct instruction): frontend, portfolio simulation, advanced analytics, RAG, and a Historical Event Database. The Simulation Engine is not modified in any way.
- **Rationale**: Every parameter above fills a confirmed Founder Specification silence or introduces a genuinely new category of risk (third-party data sharing) with an explicit, deliberate choice — matching this project's standing rule that AI-authored code must not resolve a security-relevant spec gap unilaterally. Narrowing the AI's approved inputs to a strict allowlist that excludes every user-identifying field closes off the platform's first-ever third-party data-sharing decision and the most obvious prompt-injection vector in the same stroke.
- **Scope**: Governs the Educational AI System (M6) only. Does not change the Simulation Engine (M3) or the API's existing simulation endpoints (M4) — `app.ai` never imports `app.simulation`, never receives a database session, and every explanation/follow-up request is a separate, subsequent, optional call, never inline with simulation creation (Founder Specification Principle 3/4).
- **Deferred**: A Historical Event Database and Educational Knowledge Base (named in Part 2.7.6/2.7.7 but never built in any prior milestone) remain deferred to the already-anticipated future RAG milestone (Part 2.7.14) — M6 scopes the Explanation Engine and Financial Tutor to simulation output plus static, source-controlled educational content only, not a live knowledge corpus.
- **Related**: ADR-021 through ADR-024 (`docs/ARCHITECTURE_DECISIONS.md`), `docs/MILESTONE_REPORTS/M6_REPORT.md`, `docs/KNOWN_ISSUES.md` KI-032 through KI-034.

## Founder Decision 004 — M7 Design Foundation: Visual Identity Approval, Theme Architecture, Scope Exclusions, Growth-Chart Consistency, Anonymous AI Access, and Brand Philosophy

- **Date**: 2026-07-13
- **Status**: Approved
- **Context**: Before any Next.js/React/Tailwind code was written, M7 (Frontend) underwent a three-part design review — an initial frontend design system, a pressure-test against installed UI/UX design skills, and a brand-identity deep-dive — consolidated into `docs/BRAND_CONSTITUTION.md`, the product's authoritative visual-identity standard. That review surfaced nine decisions requiring founder-level sign-off: some resolve genuine Founder Specification ambiguities (anonymous AI access vs. Part 2.8.8's "sensitive endpoint" language; Asset Comparison/Report Generation's core-vs-optional status per KI-007), others are new product-direction choices with no specification precedent at all (theme architecture, brand-identity philosophy). This entry formalizes all nine — previously recorded only as FD-004 through FD-012 inside `docs/BRAND_CONSTITUTION.md` itself — using this document's own numbering convention, closing the documentation-consistency gap noted in that milestone's own DEVLOG entry.
- **Decision**: The founder approved, as project policy, all nine design-review decisions:
  1. **Visual design system approved.** The design system and brand direction developed across the M7 Phase 0 review — `docs/frontend_design_system.md` (implementation detail) and `docs/BRAND_CONSTITUTION.md` (brand philosophy) together — is approved as the binding baseline for all frontend work from Phase 1 forward.
  2. **Theme architecture from day one.** Light and dark themes are supported architecturally from the first design token committed, not deferred to a later pass — dark mode is a first-class expression of the brand, equal to light mode, matching the Vercel/Linear reference precedent named in the design review.
  3. **M7 feature scope exclusions.** Asset Comparison and Report Generation remain out of M7 scope, regardless of their ambiguous core-vs-optional status between the PRD and Functional Requirements documents (`docs/KNOWN_ISSUES.md` KI-007) — resolved in favor of excluding both from this milestone rather than investing UI effort against unconfirmed scope.
  4. **Minimal Account/Settings scope.** M7 builds only a minimal Account/Settings experience: email display, password change, and logout-of-all-sessions. Multi-device session management UI and any richer settings surface are explicitly out of scope for M7, even though the backend primitives for multi-device sessions already exist (ADR-017).
  5. **Growth chart consistency is a backend precondition.** `growth_series`/`disclosed_splits` persistence (`docs/KNOWN_ISSUES.md` KI-021) is reclassified from an accepted, permanent frontend design-around to a blocking prerequisite: the Results screen is not considered complete until these fields are reliably available on every retrieval, not only at simulation creation.
  6. **Anonymous educational AI access, permitted.** Anonymous (unauthenticated) users may generate educational AI explanations and ask Financial Tutor follow-up questions. This resolves, in the anonymous-permissive direction, the ambiguity Founder Decision 002 itself left open between its own "may not... generate AI reports" text and the as-built `explanations` router's actual behavior (`get_current_user_optional`).
  7. **Anonymous AI access is rate-limit-protected, not access-gated.** The mechanism protecting anonymous AI usage from abuse is rate limiting and abuse controls, not an authentication gate — and any limit reached must be communicated to the user in plain, friendly language, never as a punitive or blaming error. (The exact anonymous-specific limit value, if different from the existing 20/min authenticated bucket per `docs/api_design.md`, is an implementation detail for Phase 1/M7 backend work, not part of this decision.)
  8. **Trust and education over excitement.** The visual identity reinforces trust, historical accuracy, and education ahead of excitement in every instance — every visual element must help a user understand a historical investment outcome; none may encourage speculation or trading-style behavior.
  9. **Confidence without ego.** Investment Time Machine earns trust through correctness, transparency, and restraint — never through visual spectacle. The product prefers admitting uncertainty (a data gap, an AI outage, a genuine limitation) over presenting misleading confidence, in both copy and visual design.
- **Rationale**: Decisions 1–5 and 8–9 are genuinely new product/brand-direction choices with no prior Founder Specification text to resolve — recorded here, as this project's standing practice requires, so a brand or scope choice carries the same explicit, traceable authority as a specification-silence resolution. Decisions 6–7 do resolve a real specification-level tension (Part 2.8.8's "sensitive endpoint" framing vs. Part 2.6.24's explicit anonymous-use design, previously left ambiguous even after Founder Decision 002's initial pass at the anonymous-use boundary) and are recorded with the same rigor as Founder Decision 002 itself. Reclassifying KI-021 (decision 5) as a blocking prerequisite, rather than a permanent accepted gap, follows directly from this project's own "Historical Truth Is Sacred" principle applied to the one screen where the growth chart is the actual product being shown.
- **Scope**: Governs the Frontend milestone (M7) and the brand/visual-identity standard (`docs/BRAND_CONSTITUTION.md`) it establishes. Does not change the Simulation Engine, API, Identity Management, or Educational AI System's own backend behavior, except where decision 5 requires a backend change (persisting `growth_series`/`disclosed_splits`) as a precondition of M7's Results screen — that persistence work is backend-layer and must be scheduled before or alongside M7 Phase 1's page-level implementation, not as a redesign of any existing milestone's approved scope.
- **Deferred**: The exact anonymous-specific AI rate-limit value (decision 7) is left to M7/backend implementation, not fixed by this decision. Formalizing the growth-chart persistence fix (decision 5) as its own ADR is left to whichever milestone implements it. Multi-device session management UI (decision 4) remains deferred, not excluded — the same distinction Founder Decision 002 already drew for M5.
- **Related**: `docs/BRAND_CONSTITUTION.md` (full brand/visual-identity standard, all nine decisions' complete reasoning), `docs/frontend_design_system.md` (implementation-level design system this decision approves), `docs/KNOWN_ISSUES.md` KI-007, KI-021, resolves the anonymous-AI-access ambiguity left open by Founder Decision 002, `docs/PROJECT_STATE.md` (M7 Phase 0 completion record).

## Founder Decision 013 — Experience Philosophy

- **Date**: 2026-07-19
- **Status**: Approved
- **Context**: M7 Phase 0 through Phase 2 established what the product looks like (`docs/BRAND_CONSTITUTION.md`) and how its first screen was built (`docs/frontend_design_system.md`, the shipped Simulator). Neither document, by design, states the interaction-level philosophy that should govern every *future* screen's feel — loading states, transitions, disclosure patterns, error tone — leaving that judgment to be re-derived per screen rather than stated once, authoritatively, above both existing documents. `docs/EXPERIENCE_CONSTITUTION.md` was drafted (documentation only, no code changed) during M7 Phase 2's closing pass with a complete, ready-to-formalize proposal for this decision, mirroring exactly how Founder Decision 004 was first drafted inside `docs/BRAND_CONSTITUTION.md` before a follow-up pass formalized it here.
- **Decision**: The founder approves `docs/EXPERIENCE_CONSTITUTION.md` as the product's highest-level UX and interaction philosophy, upstream of `docs/BRAND_CONSTITUTION.md` (visual identity) and `docs/frontend_design_system.md` (implementation), governing every future interaction decision. Specifically approves:
  1. **The three-layer experience model** (Identity/Worked Example, Trust/Proof, Behavior/Instrument, §3) as the standing test applied to every future screen and interaction before it ships.
  2. **The Worked Example as the product's sole content identity** (§4) — every current and future screen must anchor to a specific, personal "if I had invested…" question; no screen may become an abstract dashboard or portfolio-analytics surface.
  3. **The Proof principles** (§5) — progressive disclosure as the default posture for depth beyond a screen's one question, every number carrying a reachable source, assumptions always stated rather than silently applied, and the AI layer's permanent experiential subordination to the Simulation Engine's calculated output.
  4. **The Instrument principles** (§6) — motion communicates state only, an identical reveal treatment regardless of a result's sign, determinism and interruptibility as first-class interaction properties, and the permanent rejection of theatrical/celebratory treatment for any financial outcome.
  5. **The interaction anti-pattern list** (§8) as a standing rejection list, with the same permanence `docs/BRAND_CONSTITUTION.md` §13's "Things We Never Do" already carries for the visual layer.
- **Rationale**: Every future interaction decision currently has no single authoritative philosophy to check against above the visual/implementation layer, meaning the same judgment call (how should this loading state feel? how should this error read?) risks being re-derived inconsistently, screen by screen, by whoever happens to build it. This decision closes that gap the same way Founder Decision 004 closed it for the visual layer — before, not after, the next several screens (starting with M7 Phase 3B's Results foundation, built the same day this decision was formalized) are built.
- **Scope**: Governs the frontend's interaction/UX philosophy for all future work (M7 Phase 3 onward). Does not change any already-approved visual token, component spec, or backend behavior; does not retroactively require rework of the shipped Simulator, which this document was written to describe and validate against, not to invalidate — every principle stated in it already has a shipped, working example cited beside it.
- **Deferred**: No open items — this decision is complete as written and does not depend on any unresolved question elsewhere in the project.
- **Related**: `docs/BRAND_CONSTITUTION.md`, `docs/frontend_design_system.md`, `docs/EXPERIENCE_CONSTITUTION.md` (this document, in full — its own "Status" header updated from PROPOSED to APPROVED in the same pass as this entry), `docs/MILESTONE_REPORTS/M7_PHASE_2_REPORT.md` (the Simulator build this document cites as evidence throughout).

## Founder Decision 014 — Growth Series Persistence (Option A)

- **Date**: 2026-07-19
- **Status**: Approved (full implementation scheduled; not yet built as of this decision)
- **Context**: KI-021 (found during the M4 API design review) established that `growth_series`/`disclosed_splits` are computed fresh by the Simulation Engine on every `POST /api/v1/simulations` request but never persisted — no `simulations` table column exists for either. A subsequent `GET /api/v1/simulations/{id}` (the Results screen's own retrieval path) therefore returns both as empty lists, a gap Founder Decision 004 (decision 5, internally "FD-008") already reclassified from an accepted design-around to a blocking prerequisite for the Results screen specifically. That reclassification approved the *principle* (persistence must happen before the Results screen is complete) but left the *mechanism* open. This decision approves the specific mechanism, presented as four options during the M7 Phase 3B design review: (A) persist `growth_series` at simulation creation, backfill existing completed simulations, and version the stored series against `calculation_version`; (B) persist only `disclosed_splits` now and continue computing `growth_series` on demand at retrieval time (cheaper to build, but re-derives the same figure twice and doesn't fully close KI-021); (C) compute `growth_series` lazily on first `GET` and cache it (defers the backfill problem but adds request-time computation the engine's own reproducibility guarantees don't currently cover); (D) leave growth_series ephemeral permanently and document the Results screen as chart-free (rejected outright — direct conflict with Founder Decision 004 decision 5 and Founder Specification Part 3.3.2's explicit "Growth Chart" requirement).
- **Decision**: The founder approves **Option A**:
  1. `growth_series` is persisted on the `simulations` row at simulation creation time (a new nullable JSON/JSONB column), not recomputed on every subsequent retrieval.
  2. Existing completed simulations created before this column existed are backfilled by a one-time migration/script that re-runs `calculate_growth_series` against each one's already-stored inputs and stamps the result with that simulation's own `calculation_version` — never a different, newer version than the simulation was originally computed under.
  3. `calculation_version` (already present on every `Simulation` row since the first migration, per `docs/ARCHITECTURE_DECISIONS.md` ADR-002) is the version tag stored alongside the persisted series, so a future change to `calculate_growth_series`'s methodology never silently rewrites a historical result out from under a user who already viewed it.
  4. `GET /api/v1/simulations/{id}` must not return an empty `growth_series` for any simulation whose `status` is `completed` — this is the literal closing condition for KI-021, not merely an improvement to it.
- **Rationale**: Option A is the only one of the four that fully satisfies Founder Decision 004 decision 5's own text ("`growth_series`... must be reliably available on every retrieval, not only at creation time") rather than narrowing or deferring the gap again. Persisting at creation (rather than lazily at first read) keeps the Simulation Engine's reproducibility guarantee intact — the exact series a user saw immediately after creating a simulation is the same series stored and later served, with no re-derivation step that could theoretically diverge if the engine's own code changes between the two reads. Tying the stored series to `calculation_version` is what makes a future engine change safe: a stored series is permanently labeled with the methodology that produced it, so `docs/ARCHITECTURE_DECISIONS.md`'s reproducibility principle (ADR-002) extends naturally from the scalar outputs (`final_value`, `cagr_percentage`, already versioned) to the series data.
- **Scope**: Governs the Simulation Engine's persistence layer and the `GET /api/v1/simulations/{id}` retrieval path only. Does not change how `growth_series` is *calculated* (`app.simulation.formulas.calculate_growth_series` is unmodified by this decision) and does not change the `POST /api/v1/simulations` response shape, which already includes a correct `growth_series` today. `calculation_version` itself is additionally exposed on `SimulationResponse` as of M7 Phase 3B (a small, unconditionally safe schema addition made alongside this decision, independent of the persistence work below) — see `docs/ARCHITECTURE_DECISIONS.md` ADR-036.
- **Deferred**: The actual schema migration (new `simulations.growth_series` column), the backfill script, and the repository/engine wiring to read-through the persisted column instead of recomputing are **not implemented by this decision** — M7 Phase 3B builds only the Results foundation (hero numbers, snapshot, technical details; explicitly no growth chart, per direct instruction), so there is nothing yet that reads `growth_series` from the frontend. The persistence implementation is scheduled for the milestone that actually builds the growth chart (M7 Phase 3C or equivalent), tracked as KI-021's continued-open status below, not silently dropped.
- **Related**: `docs/KNOWN_ISSUES.md` KI-021 (status updated by this decision — see below), Founder Decision 004 decision 5 (internally "FD-008" in `docs/BRAND_CONSTITUTION.md`), `docs/ARCHITECTURE_DECISIONS.md` ADR-002 (`calculation_version` non-negotiable), ADR-016 (engine-computed growth series), ADR-036 (`calculation_version` exposure, M7 Phase 3B).

## Founder Decision 015 — Anonymous Educational AI Limits (Option D)

- **Date**: 2026-07-19
- **Status**: Approved — **Implemented** (M7 Phase 4, 2026-07-23; see `docs/ARCHITECTURE_DECISIONS.md` ADR-049 for the full engineering record)
- **Context**: Founder Decision 004 (decisions 6–7, internally "FD-009"/"FD-010") already approved that anonymous users may generate educational AI explanations and Financial Tutor follow-up questions, protected by rate limiting rather than an authentication gate, with any limit reached communicated in "plain, friendly language." It left the exact limit value as "an implementation detail for Phase 1/M7 backend work... not part of this decision." As-built, `app.core.rate_limit.rate_limit_ai` (`backend/app/api/v1/dependencies.py`) applies one identical 20-requests-per-minute limit (Founder Specification Part 2.8.13) to every caller, keyed purely by IP address, with no distinction between anonymous and authenticated traffic and no daily cap of any kind — the M7 Phase 3B design review presented four options to close this remaining gap: (A) leave the limit identical for both groups permanently (rejected — never actually distinguishes anonymous from authenticated as Founder Decision 004 decision 7 anticipated it eventually would); (B) authentication-gate AI entirely for high-volume use, keeping the anonymous tier at today's 20/min with no cap (rejected — direct conflict with Founder Decision 004 decision 7's explicit "rate limiting... not an authentication gate" mandate); (C) a per-minute-only reduction for anonymous users with no daily cap (partial — narrows abuse within a minute but not sustained low-and-slow usage across a day); (D) a lower per-minute limit plus a new daily cap for anonymous users, unauthenticated and with no auth wall, alongside an unchanged or higher limit for authenticated users, with friendly limit-reached messaging throughout.
- **Decision**: The founder approves **Option D**:
  1. Anonymous users continue to receive educational AI explanations and Financial Tutor follow-up questions with **no authentication wall** — this decision only tightens the numeric limits, it does not revisit Founder Decision 004 decision 6's access approval itself.
  2. Anonymous requests are limited to a **lower per-minute rate** than authenticated requests (proposed default: 5/min, down from the current shared 20/min — a specific implementation value, adjustable by the engineering team implementing it without requiring a new Founder Decision, provided the *relative* anonymous-lower/authenticated-higher shape is preserved).
  3. A **new daily cap** applies to anonymous users specifically (proposed default: 15/day), keyed the same way the existing limiter already keys anonymous traffic (IP address) since no other anonymous identifier exists — the first daily-scoped limit anywhere in this codebase; today's `rate_limit.py` is a per-minute-only fixed window.
  4. Authenticated users keep the existing **20 requests/minute** (Founder Specification Part 2.8.13's explicit, spec-mandated value — unchanged) and receive a materially higher daily cap than the anonymous tier (proposed default: 150/day), reflecting that a signed-in user is both identifiable (reducing abuse risk) and a more likely genuine repeat user of the platform's core educational loop.
  5. Whenever either limit is reached, the response uses the existing calm, non-punitive `RATE_LIMIT_EXCEEDED` copy convention (`frontend/src/lib/api/errors.ts`: "Slow down a little" / "You've reached the limit for now. Please try again shortly.") — extended, at implementation time, with wording that distinguishes a daily cap ("come back tomorrow") from a per-minute cap ("try again shortly"), so neither reads as punitive or implies the user did something wrong, per Founder Decision 004 decision 7's own standing requirement.
- **Rationale**: Option D is the only one of the four that actually delivers on Founder Decision 004 decision 7's anticipated "anonymous-specific limit value, if different from the existing 20/min authenticated bucket" — Options A and C only partially close that gap, and Option B directly contradicts the same decision's explicit rejection of an authentication gate. A daily cap specifically (not just a lower per-minute rate) is the correct mechanism for a real cost-control concern this project has not previously had to solve: AI generation costs real money per call regardless of how it's throttled per-minute, and a determined anonymous caller respecting a 5/min limit could still generate hundreds of calls across a day without one. Keeping the mechanism itself unchanged (IP-keyed, Redis-backed fixed window, fails open on Redis outage) — only the specific limits and the addition of a second, daily-scoped window — means this decision extends `app.core.rate_limit` rather than replacing it.
- **Scope**: Governs the Educational AI System's (M6) rate-limiting configuration only. Does not change the AI explanation/follow-up generation logic itself, the privacy allowlist, the caching rules, or the AI integrity check (all Founder Decision 003, unchanged). Does not revisit anonymous AI *access* (Founder Decision 004 decisions 6–7, unchanged) — only the numeric limits protecting it.
- **Deferred**: ~~Implementation (the daily-cap mechanism in `app.core.rate_limit`, the distinct anonymous/authenticated limit values wired into `rate_limit_ai`, and the frontend's daily-cap-specific limit-reached copy) is **not built by this decision** — M7 Phase 3B builds only the non-AI Results foundation (no AI panel exists yet to need a limit-reached state).~~ **Resolved 2026-07-23 (M7 Phase 4)**: the daily-cap mechanism, the anonymous/authenticated split (keyed by user id when authenticated, IP when anonymous), and the AI panel's rate-limit-reached copy are all implemented exactly as this decision specifies — see ADR-049. The exact 5/min and 15/day (anonymous) and 150/day (authenticated) figures shipped as the proposed MVP defaults, unchanged, per this decision's own explicit permission to adjust them without a new Founder Decision provided the relative shape is preserved; still not load-bearing precision — explicitly revisit based on real usage once real traffic exists, the same revisit posture `docs/api_design.md`'s own rate-limit table already applies to its non-spec-mandated buckets.

## Founder Decision 016 — CAGR Percentage Scale Correction

- **Date**: 2026-07-21 (proposed), **Approved 2026-07-22**.
- **Status**: **Approved.** This entry previously existed as two duplicate proposal drafts (a documentation-consistency defect from the pass that recorded them, itself now corrected by this edit — the two drafts differed only in a stale branch reference, not in substance); consolidated into one entry on approval, per this document's own "never edit a previous Founder Decision" rule read together with `.claude/DOCUMENTATION_POLICY.md`'s general instruction to keep documentation synchronized with reality — a duplicated in-progress draft is not itself a decided, citable decision requiring a superseding entry, only a defect blocking one. `docs/KNOWN_ISSUES.md` KI-045 is resolved by this decision — see its entry for the resolution record.
- **Context**: A live, real-backend end-to-end check (M7 Phase 3B.2 damage-assessment pass) surfaced a simulation showing **"+73.01% Total Return"** alongside **"+0.15% CAGR"** for the same result — the correct CAGR for that scenario is in the ~10-15% territory, a factor-of-100 error. Traced end to end (full detail in KI-045): `backend/app/simulation/formulas.py::calculate_cagr` returned a raw fraction (`ratio ** (1/years) - 1`, no `× 100`), matching its own docstring and `docs/simulation_formulas.md` §4's literal documented formula. `calculate_total_return_percent`, its sibling in the same module, *does* multiply by 100. Both values are stored into identically-typed `NUMERIC(10, 6)` columns named with the same `_percentage` suffix (`backend/app/models/simulation.py`), with nothing on either column distinguishing the two conventions. The backend's own known-answer test (`tests/simulation/test_engine_known_answer.py`) asserted the raw-fraction value — the test suite was certifying the bug as correct. `docs/api_design.md`'s own worked-example response was internally inconsistent with the code it documented: for its stated inputs (`investment_amount=1000`, `initial_price=100`, `final_price=250`, 2015-01-01 to 2025-01-01), the independently hand-derived correct value is `9.594448` (not the pre-fix draft's `9.596872`, which was itself a drift error, not a literal preview of the intended fix). `frontend/src/lib/format/percentage.ts::formatPercentage` performs no scaling (by design, and correctly so for `total_return_percentage`) — it is simply the layer where the pre-existing 100x error became visible to a user. A third, previously non-user-facing surface (`backend/app/api/v1/services/explanation_service.py`) fed the same wrong-scale value into the Educational AI System's prompt context.
- **Problem**: Decide which layer owns the correction, and — specifically — whether correcting it is a pure display-layer fix or a Simulation Engine calculation change requiring a `calculation_version` bump and a backfill of every existing `completed` simulation's stored `cagr_percentage`, per `docs/ARCHITECTURE_DECISIONS.md` ADR-002's reproducibility principle ("identical [calculation logic] yields identical results" — ADR-002 was written for the data-reimport case, but the same mechanism generalizes directly to a formula-correctness fix, and Founder Decision 014 already extends it exactly this way for `growth_series`).
- **Options Considered**:
  1. **(A) Fix at the source** — correct `calculate_cagr` (or the point in `engine.py` where its result is stored) to produce a percentage-scaled value, matching `calculate_total_return_percent`'s existing convention; update `docs/simulation_formulas.md` §4 and the known-answer test's assertion to match; bump `DEFAULT_CALCULATION_VERSION` (`backend/app/simulation/engine.py`, previously a hardcoded `"v1"` with no versioning logic implemented anywhere yet) to a new version; **backfill every existing `completed` simulation row** — multiply its already-stored `cagr_percentage` by exactly 100 (a lossless, purely mechanical rescale of an already-computed value, not a re-derivation from raw price data that could theoretically diverge from what a user originally saw) and re-stamp its `calculation_version` to the new value. This is the only option that also closes the dormant AI-prompt-context leak (`explanation_service.py`) and stops the test suite from continuing to certify the wrong value.
     - **1a — backfill immediately** (rewrite in place, one-time migration/script): every stored simulation, past and future, is correct after the fix ships; no simulation a user has ever created or shared keeps showing a wrong number.
     - **1b — do not backfill, only re-stamp forward** (leave pre-fix rows at the old `calculation_version` with their uncorrected value, document the version boundary): avoids a write pass over historical data, but means any simulation created before the fix — including ones already shared via the "Copy link" affordance — keeps showing an incorrect CAGR forever unless a user re-runs it, which is silently wrong, not a stated limitation (`docs/EXPERIENCE_CONSTITUTION.md` §5's "silence about a limitation is never acceptable" reads as a strong argument against this sub-option specifically).
  2. **(B) Fix at the API response boundary only** — rescale `cagr_percentage` by 100 inside `SimulationResponse` construction (`backend/app/api/v1/schemas/simulations.py`), leaving `calculate_cagr`, the Simulation Engine's internal representation, and every already-stored DB row exactly as they are today. Every client (existing and historical rows alike) sees the corrected value immediately on their next `GET`, with **no `calculation_version` bump and no backfill migration needed** — the correction happens at serialization time, not storage time. Leaves the Simulation Engine's own internal/stored convention (a raw fraction) permanently diverging from what the API actually serves, and does not fix `explanation_service.py`'s prompt-context leak unless that call site is also updated to rescale independently (a second, easy-to-forget special case).
  3. **(C) Fix at the frontend only** — a dedicated display helper (e.g. `formatCagrPercentage`) that multiplies by 100 before calling `formatPercentage`, touching nothing on the backend. Fastest to ship, zero backend risk, but leaves the underlying stored/served value wrong at its source for every other consumer — the AI explanation context (already affected), any future analytics/reporting surface, any second frontend client, or a direct API consumer reading `docs/api_design.md`'s own documented contract — and treats a genuine calculation-correctness defect as a formatting nuance, which this project's own engineering priority order (`.claude/SYSTEM.md`: Correctness before Aesthetics) argues against.
- **Final Decision**: **Option 1a** (fix at the source, backfill immediately) — approved as recommended.
- **Rationale**: The Simulation Engine is this platform's sole source of financial truth by explicit design (`.claude/SYSTEM.md` Principle 3, restated everywhere from `docs/simulation_formulas.md`'s own opening line to `docs/BRAND_CONSTITUTION.md` §1) — a wrong value stored at the source would keep resurfacing in every future consumer of that value, as it already had once (the AI prompt context) before the AI panel had even shipped. Options B and C would each fix exactly one visible symptom while leaving the actual defect in place at its origin, trading a one-time, mechanically-safe backfill for an open-ended maintenance burden of remembering to rescale `cagr_percentage` correctly at every future call site, forever. `calculation_version` and a backfill migration were not new infrastructure this decision had to invent — Founder Decision 014 already established the identical pattern (persist, version, backfill) for `growth_series` one decision prior.
- **Scope**: Governs `backend/app/simulation/formulas.py::calculate_cagr`, `backend/app/simulation/engine.py`'s `DEFAULT_CALCULATION_VERSION`, `docs/simulation_formulas.md` §4/§4a, `backend/tests/simulation/test_engine_known_answer.py`'s CAGR assertions, `docs/api_design.md`'s worked example, and the one-time backfill migration (`backend/alembic/versions/0004_cagr_percentage_v2_backfill.py`) over the `simulations` table. Does not change `calculate_total_return_percent` (already correct), `calculate_final_value`, `calculate_inflation_adjusted_value`, `calculate_growth_series`, or any frontend code (`formatPercentage` already handles a correctly-scaled input properly — no frontend change was needed under Option A, confirmed).
- **Deferred**: None — implemented in full in the same pass as this approval. See `docs/ARCHITECTURE_DECISIONS.md` ADR-040 for the complete engineering record (version-string choice, backfill mechanism, and the `NUMERIC(10,6)` overflow bound check for the one edge case a lossless rescale can't safely reach).
- **Related**: `docs/KNOWN_ISSUES.md` KI-045 (the underlying defect, now Resolved), `docs/ARCHITECTURE_DECISIONS.md` ADR-002 (`calculation_version` non-negotiable), ADR-040 (this decision's engineering implementation), Founder Decision 014 (the identical persist/version/backfill pattern already approved for `growth_series`), `docs/simulation_formulas.md` §4/§4a, `docs/api_design.md` (the `POST /api/v1/simulations` worked example).

## Founder Decision 017 — Results Opening Sequence: Staged Reveal Rejected, Editorial Components Kept

- **Date**: 2026-07-22
- **Status**: Approved
- **Context**: M7 Phase 3B.1 built the Results Opening Sequence: a ~2.75-second staged composing→pause→reveal timeline (phrases of the worked-example sentence appearing one at a time, a ~1-second silent pause, then the answer, then settling) gating the entire rest of the Results page behind it, plus a "Skip" affordance for users who didn't want to wait. - **Decision**: The staged composing→pause→reveal timeline is removed in full. The sentence and all Results content render **immediately** on data arrival — no phrase-by-phrase composing, no silent pause, no gated reveal, no "Skip" affordance (there is nothing left to skip). - **Rationale**: This is an enforcement of Founder Decision 013's own Instrument principles (§6: "motion communicates state only... determinism and interruptibility as first-class interaction properties... the permanent rejection of theatrical/celebratory treatment for any financial outcome"), applied retroactively to a component that shipped before this specific ruling was requested. A ~2.75-second mandatory wait before a user can see their own requested result — regardless of how calm or well-choreographed — is a theatrical treatment the Experience Constitution's own standing principles already argue against; the fact that a "Skip" button was needed at all is itself evidence the default experience was imposing a cost on the user's time that a genuinely restrained design wouldn't require in the first place. The one remaining ~200ms settle is kept, not eliminated entirely, because a zero-motion instant swap between "page shell" and "hero sentence" reads as a layout flash/glitch rather than an intentional design choice — a single short, subtle, fully-interruptible, fully-reduced-motion-respecting transition is the minimum polish consistent with "motion communicates state only," not a reintroduction of the rejected choreography at a smaller scale.
- **Scope**: Governs `frontend/src/components/simulation-result/opening-sequence-heading.tsx`, the hook previously named `use-opening-sequence.ts` (replaced by `use-settle-in.ts`), `frontend/src/hooks/use-just-created-flag.ts` (removed), `frontend/src/components/simulator/simulation-form.tsx`'s post-submit navigation (no longer appends `?new=1`), `frontend/src/app/simulation/[id]/page.tsx` (the `Suspense` boundary it required is removed along with the `useSearchParams` call that needed it), and the dev-only `frontend/src/app/dev/opening-sequence-preview/` route (removed — nothing left to preview). - **Deferred**: None — implemented in full in the same pass as this ruling. See `docs/ARCHITECTURE_DECISIONS.md` ADR-041 for the complete engineering record, which also marks ADR-038 (the `?new=1` mechanism this decision removes) as Superseded.

## Founder Decision 018 — Motion & Atmosphere Amendment

- **Date**: 2026-07-25
- **Status**: Approved
- **Context**: M7 Phase 3D (Design Elevation) implements a founder-approved visual mockup — explored externally and approved as the product's new visual direction — at `DOCUMENTS_AND_IDEAS/Investment Time Machine Results/Investment Time Machine.dc.html` (a `.dc.html` design-comp export; the file and its containing folder are untracked in this repository at the time of this decision, a documentation/commit-hygiene gap noted here rather than silently worked around — see this milestone's DEVLOG/ADR entries). The mockup's own script implements a one-shot digits-only "scramble/settle" reveal on every financial figure (hero sentence figures, supporting-fact stat values), a per-section entrance rise, a chart draw-in with an endpoint marker pop, and a static grain/gradient atmosphere. Founder Decision 017 (2026-07-22) established an "immediate-render" principle for the Results page — content renders on data arrival, with at most one ~200ms settle transition and no staged, multi-stage reveal choreography — specifically rejecting a ~2.75-second composing→pause→reveal timeline as theatrical. The mockup's scramble effect is a new, distinct motion pattern FD-017 did not anticipate or rule on, and it is JS-`requestAnimationFrame`-driven (not a CSS transition), meaning `globals.css`'s existing global `prefers-reduced-motion` CSS override cannot stop it by itself — a CSS media query cannot halt an already-scheduled JS animation loop. This amendment is required before that scramble pattern (or the mockup's other motion/atmosphere elements) can be implemented without contradicting FD-017's still-standing immediate-render principle.
- **Decision**: FD-017's immediate-render principle stands. It is amended to permit exactly the following, and nothing beyond it:
  1. **A digits-only scramble/settle on financial figures.** Runs exactly once per page load, progressive left-to-right character lock (ease-out), non-digit characters (`$`, `,`, `.`, `%`, `+`, `−`, spaces, letters) never cycle and are locked from the first frame. Maximum duration 600ms on hero figures (the worked-example sentence's two emphasized figures), 450ms on supporting stat figures. Stagger between elements is ≤150ms. The first scramble starts ≤150ms after the underlying data has rendered (never gating the surrounding sentence — see clause 5 below). On settle, a single brief glow pulse plays (≤650ms) and does not repeat. This is the literal mechanism already implemented in the mockup's `Component.scramble()`/`componentDidMount()` and is ported as-is, not reinvented.
  2. **One-shot section entrances.** A section may rise into place on first mount: ≤16px translate, ≤350ms, ease-out, a light stagger between sections (the mockup's `sectionRise` keyframe, ≤0.35s with ≤0.07s per-section delay steps). Never re-triggers on scroll, re-render, or revisit — mount-once only, the same standing rule `useSettleIn` already enforces for the hero sentence's settle transition (FD-017/ADR-041).
  3. **A one-shot chart draw-in and endpoint marker pop.** The growth line may draw in once (≤800ms) with an endpoint marker pop on completion, mirroring the mockup's `chartDraw`/`popScale` keyframes — but see Scope below: this clause authorizes the *pattern* as compliant with the Experience Constitution's motion rules; it does not by itself reopen ADR-043's specific decision to disable Recharts' own draw-in animation for the shipped Growth Chart, which remains in force unless a future ADR explicitly revisits it.
  4. **Static atmosphere only.** A fine grain texture and a static gradient wash are permitted as background treatment. No infinite or looping animation of any kind, anywhere in the product — this expressly rejects the "discarded looping background" design exploration found alongside the approved mockup (`Dark Pool Liquidity Tide.dc.html`, an unrelated, non-approved atmosphere concept using `bandDrift`/`whaleDrift`/`rippleEcho` infinite-loop keyframes), which is not part of the approved direction and must never be ported.
  5. **Hard gate — `prefers-reduced-motion` via `matchMedia`, everywhere, including the JS-driven scramble.** Every rule above — including the scramble, which a CSS media query alone cannot stop — must check `window.matchMedia('(prefers-reduced-motion: reduce)')` (the existing `useReducedMotion` hook, FD-013 §9/ADR-041's established pattern) and, when reduced motion is preferred, render final values and final layout instantly, with zero motion, on the very first frame. This is a **test-enforced requirement**: no scrambled/intermediate character state may ever be observable under reduced motion, verified by an automated test, not left to visual inspection.
  6. **Identical treatment for gains and losses.** Every rule above applies without exception or variation to a gain or a loss outcome — no faster/slower timing, no different easing, no different glow intensity keyed to a result's sign. This restates, and does not relax, FD-013 §6/§7 and FD-017's own standing "identical reveal treatment regardless of outcome" rule.
  
  Nothing else FD-017 rejected is reopened by this amendment: there is still no multi-second composing→pause→reveal timeline, no "Skip" affordance, no gating of the surrounding sentence text behind any of the above — the sentence's prose renders immediately and unconditionally; only the numeric figures within it scramble in place, in parallel with (never blocking) the rest of the page.
- **Rationale**: FD-017 correctly rejected a *staged, multi-second, gating* reveal as theatrical — but a sub-second, non-gating digit scramble that never delays content, never repeats, and degrades to an instant final state under reduced motion is a categorically different thing: it communicates "this number just arrived and is now settled," the exact purpose FD-013 §6/`docs/BRAND_CONSTITUTION.md` §8 already assigns to motion in this product, and the Experience Constitution's own text (§6) explicitly anticipated exactly this shape of change ("the specific motion timings and easing curves... may evolve freely... as long as the rules... are unchanged"). The founder reviewed the mockup directly and ruled this specific, bounded set of motion permissions compliant with that standing rule; nothing here relaxes the permanent rejection of theatrical/celebratory treatment, gain/loss asymmetry, or content-gating motion.
- **Scope**: Governs motion and static-atmosphere treatment for the M7 Phase 3D Design Elevation implementation only — `frontend/src/components/simulation-result/opening-sequence-heading.tsx`, `results-sections.tsx`, `growth-chart.tsx`, the Simulator's restyled surfaces, and any new scramble/settle hook(s) introduced to implement clause 1. Does not reopen or relax FD-017's rejection of the staged composing→pause→reveal timeline, does not reintroduce a "Skip" affordance, and does not by itself change ADR-043's Recharts `isAnimationActive={false}` decision (clause 3 authorizes the pattern; the shipped chart's own animation state is unchanged unless a future ADR revisits it explicitly). Does not change any backend behavior.
- **Deferred**: None — this decision is complete as written. The mockup folder's untracked/commit-hygiene status (see Context) is not resolved by this decision; the implementing engineering pass has no git-commit authority and defers that action to the founder or a subsequent session.
- **Related**: Founder Decision 013 (Experience Philosophy), Founder Decision 017 (Results Opening Sequence — the immediate-render principle this decision amends), `docs/EXPERIENCE_CONSTITUTION.md` §6, `docs/ARCHITECTURE_DECISIONS.md` ADR-041 (the settle-in pattern clause 2 generalizes), ADR-043 (the Growth Chart's current motion decision, unchanged by clause 3 alone), the approved mockup at `DOCUMENTS_AND_IDEAS/Investment Time Machine Results/Investment Time Machine.dc.html`, and the explicitly-rejected `DOCUMENTS_AND_IDEAS/Investment Time Machine Results/Dark Pool Liquidity Tide.dc.html` (named in clause 4 as the discarded looping-background exploration).

## Founder Decision 018.1 — Scramble v2 Amendment

- **Date**: 2026-07-27
- **Status**: Approved
- **Context**: M7 Phase 3D-3 (founder review round 2, item 3) revisited FD-018 clause 1's digits-only scramble/settle after a live review: at the original 600ms (hero)/450ms (supporting stats) durations, re-rolling unlocked digits every animation frame (~16ms), the effect passed by too quickly and too fast-flickering to actually read as "cycling" — it registered as a brief flash rather than a legible effect, undercutting the "this number just arrived" motion purpose FD-018's own rationale assigns it.
- **Decision**: FD-018 clause 1 stands, amended to specify:
  1. **Duration raised.** Hero figures (the worked-example sentence's two emphasized figures): 900-1000ms (was "maximum 600ms," now a tighter 900-1000ms band — shipped at 950ms). Supporting stat figures: 600-700ms (was "maximum 450ms" — shipped at 650ms). The ≤150ms inter-element stagger clause is unchanged (shipped unchanged at 100ms between elements in both groups).
  2. **A legible tick rate.** Unlocked digits re-randomize at a fixed ~55ms cadence, not every animation frame — the character-lock progression (`eased`, left-to-right) is still computed every frame for a smooth lock curve, but the *visible content* of a still-cycling digit holds for ~55ms before changing, long enough to read as digits cycling rather than a blur.
  3. **A glow during the cycle, not only after.** A subtle, constant-intensity accent glow (`text-shadow`, `--color-accent`) is now visible for the full cycling phase — not only the brighter pulse already specified for the 650ms post-settle hold. The in-cycle glow is deliberately lower-intensity than the settle pulse (shipped at 8px hero / 6px supporting-stats, vs. 20px / 16px at settle) so the settle moment still reads as the more emphatic one.
  4. Every other FD-018 clause-1 guarantee is unchanged and re-confirmed: still exactly once per page load, still digits-only (non-digit characters — `$`, `,`, `.`, `%`, `+`, `−`, spaces, letters — never cycle, locked from the first frame), still `matchMedia`-gated under `prefers-reduced-motion` to an instant final render with zero intermediate scrambled state ever observable (test-enforced, unchanged hard gate), still never intensity- or duration-scaled by a result's sign (a gain and a loss get identical treatment at every clause above).
- **Rationale**: The founder's own review is direct evidence the shipped timing failed the "legible" bar FD-018's rationale itself set ("communicates 'this number just arrived and is now settled'" — a signal that can't do its job if it's too fast to perceive as a signal at all). Slowing the tick rate independently of the overall duration (rather than just lengthening the duration alone) is what actually fixes legibility — a longer duration re-rolling every 16ms would still blur, just for longer. The in-cycle glow closes a second, related gap: the original design communicated "arrival" only at the very end (the settle pulse), leaving the multi-hundred-millisecond cycling phase itself visually inert relative to the surrounding static page — a "not only after settle" glow makes the whole cycle read as one continuous, alive moment rather than a dead wind-up followed by the only real payoff.
- **Scope**: Governs `frontend/src/hooks/use-scramble.ts` and its two call sites (`opening-sequence-heading.tsx`'s `ScrambleFigure`, `results-sections.tsx`'s `Fact`) only. Does not reopen any other FD-018 clause, does not change the settle-pulse duration (still 650ms) or its own non-scaling-by-sign rule, and does not change any backend behavior.
- **Deferred**: None — implemented in full in the same pass as this amendment.
- **Related**: Founder Decision 018 (the clause this amendment revises), `docs/frontend_design_system.md` §15 (Component State System — the "Working/loading" row's spinner precedent for a motion detail that may "evolve freely... as long as the rules... are unchanged," the same standing permission FD-018 itself already invoked for its own approval).

- **Context**: M7 Phase 4 replaces the stale root page (`/`, a Phase 1 foundation-verification placeholder tracked in `docs/KNOWN_ISSUES.md`) with the product's real Landing page. The hero is the product's own worked-example question, set in the serif display face — the one piece of net-new motion this milestone introduces, distinct from every existing motion pattern (`useScramble`'s digit cycling, `useSettleIn`'s opacity/translate settle) since it reveals prose character-by-character rather than a financial figure. FD-017's immediate-render principle and FD-018's own scoped, bounded-motion amendments govern the Results page specifically; this decision extends the same standing motion law (§6 of `docs/EXPERIENCE_CONSTITUTION.md` — motion communicates state only, one-shot, never theatrical, `matchMedia`-gated reduced motion) to the Landing page's one new pattern.
- **Decision**: The Landing hero line may type itself **once** on load:
  1. Total duration ~1.2s, easing into a natural, steady per-character cadence (not a constant-speed typewriter) — ported in-house as `frontend/src/hooks/use-typewriter.ts` (~30 lines, no new dependency).
  2. No deletion, no loop — the sentence is revealed forward, exactly once per mount, matching every other one-shot pattern already in this codebase (`useScramble`, `useSettleIn`, both ADR-041/FD-018 precedent).
  3. A cursor bar blinks **at most twice**, then fades out permanently — never a persistent blinking cursor sitting in the finished sentence.
  4. Every other page element — the one-line product description, the primary CTA, the example-simulations list — renders **immediately** and is **never gated** behind the typing animation completing, matching FD-017's immediate-render principle applied to this page.
  5. Hard gate, matching FD-018 rule 5's precedent exactly: `window.matchMedia('(prefers-reduced-motion: reduce)')` renders the full sentence instantly, with no cursor at all, on the very first frame — no intermediate typed-partial state is ever observable under reduced motion.
- **Rationale**: A silent, static hero would under-communicate the product's own worked-example identity (`docs/EXPERIENCE_CONSTITUTION.md` §4) on the one page whose entire job is to state that identity in one sentence; a hero animation with no bound on duration, repetition, or gating would violate the same Instrument principles (§6) FD-017 already enforced once for the Results page's staged-reveal rejection. A short, one-shot, non-gating, non-looping reveal is the same shape of compromise FD-018 already approved for the Results page's figure-scramble — applied here to prose instead of a number, with the identical non-negotiables (once, bounded, reduced-motion-safe, never blocks content).
- **Scope**: Governs `frontend/src/components/landing/typed-hero-heading.tsx` and `frontend/src/hooks/use-typewriter.ts` only. Does not change any Results/Simulator motion pattern, any backend behavior, or FD-017's still-standing rejection of a staged, gating, multi-second reveal — the Landing hero's total duration (~1.2s) is a small fraction of the ~2.75s timeline FD-017 rejected, and — critically — it never gates anything else on the page, which was FD-017's actual objection, not the mere presence of motion.
- **Deferred**: None — implemented in full in the same pass as this approval.
- **Related**: Founder Decision 013 (Experience Philosophy, §6 Instrument principles), Founder Decision 017 (immediate-render principle), Founder Decision 018/018.1 (the Results page's analogous one-shot, bounded, reduced-motion-gated figure motion — the direct precedent this decision extends to prose), `docs/EXPERIENCE_CONSTITUTION.md` §6.

## Founder Decision 018.3 — Pre-Deploy Polish Amendment: Magnetic CTAs and the Button Light Sweep

- **Date**: 2026-07-14
- **Status**: Approved
- **Context**: M7 Phase 3D-4 (a founder-directed, 14-item pre-deploy polish pass) named two genuinely new motion patterns not covered by any existing FD-018 clause or its 018.1/018.2 amendments: a proximity-driven "magnetic" pull on the two primary CTAs, and a one-shot light sweep across the primary button on hover. Both were specified with concrete, bounded parameters directly in the pass's own instructions (item 10, item 11) rather than left to open-ended interpretation. Per FD-018.1/018.2's own established precedent — amend only once a pattern is actually built, recording what shipped rather than a pre-approval of an abstract concept — this entry records both, matching that same practice.
- **Decision**: Both patterns are approved exactly as shipped:
  1. **Magnetic primary CTAs** (`frontend/src/hooks/use-magnetic.ts`, `frontend/src/components/ui/magnetic-wrap.tsx`). A `requestAnimationFrame`-lerped pull toward the cursor within an ~80px proximity radius, capped at ≤6px of translate, ease-out return to (0, 0) the instant the cursor leaves the radius or the element unmounts. Same "settle to zero and stop scheduling frames" contract as `use-proximity-rows.ts` (FD-018's own no-infinite-loop rule extended to this pattern) — verified by a controllable-rAF test asserting the frame count reaches zero. Applied to exactly two elements — the Simulator's "Run simulation" button and the Landing page's primary CTA — never a general-purpose hover effect on every interactive element. Hard-disabled (no listener ever attached) for a coarse/touch pointer and for `prefers-reduced-motion`.
  2. **Button light sweep** (`.btn-sweep`, `globals.css`; wired to the `primary` button variant in `button-variants.ts`, covering every primary CTA app-wide rather than a hand-picked list). Rest state: a faint, static gradient border (`--color-accent` — already this codebase's literal gold/amber hue, ADR-044 — so no new color was introduced). Hover: a single gradient pass, 600ms, ease-out, CSS-only via a `:hover`-triggered keyframe animation that naturally restarts on every fresh hover-enter (removed along with `:hover` on mouse-leave) — no JS state needed to guarantee "once per hover-enter." Augments the existing `hover:bg-primary-hover` background-tint feedback; does not replace it.
  3. Both patterns are gated by this codebase's existing global `prefers-reduced-motion` mechanisms (the magnetic pull via an explicit `matchMedia` check in its own enabling logic, matching FD-018 rule 5's `useReducedMotion`/`matchMedia` precedent exactly; the button sweep via the file's pre-existing global CSS override that collapses every `animation-duration` to ~0.01ms, the same mechanism `.skeleton-shimmer`/`.rise-fade-in`/`.typewriter-cursor` already rely on) — no new reduced-motion mechanism was invented for either.
  4. Neither pattern's intensity, duration, or presence varies with a financial result's sign or magnitude — both are pure interaction/hover feedback, entirely independent of any calculated value, so FD-013/017/018's "identical treatment regardless of outcome" guarantee is trivially unaffected (there is no outcome-dependent branch in either implementation to begin with).
- **Rationale**: Both patterns fill exactly the shape of gap FD-018.1/018.2 already established a pattern for: a concrete, bounded, already-specified motion addition that needed a record of what actually shipped (exact pixel/millisecond values, exact gating mechanism) rather than a re-litigation of whether motion of this kind belongs in the product at all — that question was already settled by FD-013 §6 and FD-018's own existing carve-out for hover/press/focus feedback ("Hover/press/focus feedback is state communication and is permitted and expected"). Scoping the magnetic pull to exactly two elements (rather than every button) and the sweep to the `primary` variant only (rather than an ad hoc per-component class) both follow this codebase's own accent-discipline precedent (`docs/frontend_design_system.md` §16) applied to motion instead of color: a effect scattered everywhere stops reading as meaningful.
- **Scope**: Governs `frontend/src/hooks/use-magnetic.ts`, `frontend/src/hooks/use-coarse-pointer.ts` (extracted from `example-simulations-list.tsx` in the same pass — a pure refactor, not a new decision), `frontend/src/components/ui/magnetic-wrap.tsx`, and the `.btn-sweep` CSS class (`globals.css`)/`button-variants.ts` wiring. Does not change target-brackets' hover/focus treatment (item 12 of the same pass) — a plain opacity/transform state transition already covered by the existing Component State System (`docs/frontend_design_system.md` §15), not a new keyframe-driven motion pattern in FD-018's sense, and therefore not a new clause here. Does not change any Results/Simulator scramble/settle/dissolve pattern, and does not change any backend behavior.
- **Deferred**: None — implemented in full in the same pass as this approval.
- **Related**: Founder Decision 013 (Experience Philosophy, §6 Instrument principles — the standing "hover/press/focus feedback is permitted" carve-out both patterns rely on), Founder Decision 018/018.1/018.2 (the direct procedural precedent — amend once built, record exact shipped parameters), `docs/frontend_design_system.md` §15 (Component State System) and §16 (Accent Discipline, the precedent for scoping a treatment narrowly rather than applying it everywhere), `docs/KNOWN_ISSUES.md` KI-051 (the M7 Phase 3D-4 polish pass this decision's two patterns were built under).

## Founder Decision 019 — AI Provider Changed from Anthropic to Groq (Cost Control, M7 Phase 4)

- **Date**: 2026-07-23
- **Status**: Approved — Implemented
- **Context**: Founder Decision 003 selected Anthropic as "this project's first configured provider" behind `app.ai.providers.base.AIProvider`, a `Protocol`-based abstraction chosen specifically so a provider could be swapped without touching prompt, safety, caching, or router/service code. M7 Phase 4 (the Educational AI Panel — the first real, user-facing surface for the Financial Tutor follow-up endpoint FD-003/FD-004 already approved) is also this project's first milestone built against a concrete, real-money constraint: the platform's only deployment target is the Render free tier (`docs/DEPLOYMENT.md`, ADR-047/048), which has no budget line for a paid, per-token AI vendor. Anthropic's API is not free at any volume; Groq's chat completions API (OpenAI-compatible REST, no vendor SDK required) has a real, usable free tier sufficient for this feature's bounded, short-answer educational use case (2-4 short paragraphs per `app.ai.prompt._FOLLOWUP_SYSTEM_PROMPT`, `ai_max_output_tokens=800`).
- **Decision**: The founder approves swapping the Educational AI System's configured provider from Anthropic to **Groq**, effective immediately, superseding Founder Decision 003's provider selection **specifically and only on vendor choice**:
  1. `app.ai.providers.anthropic_provider.AnthropicProvider` is removed; `app.ai.providers.groq_provider.GroqProvider` (new) implements the identical `AIProvider` Protocol against Groq's OpenAI-compatible `/chat/completions` endpoint via `httpx` (already a project dependency — no new vendor SDK added).
  2. `Settings.ai_provider` now accepts `"none" | "groq"` (was `"none" | "anthropic"`); `Settings.groq_api_key` (env var `GROQ_API_KEY`) replaces the prior generic `ai_provider_api_key`/`AI_PROVIDER_API_KEY` naming, now that exactly one real provider exists. The startup fail-fast guard (a real provider configured with no key refuses to boot) is unchanged in shape.
  3. Default model: `llama-3.1-8b-instant` (was `claude-3-5-haiku-20241022`) — Groq's fastest general-purpose free-tier model, an appropriate match for this feature's short, latency-sensitive, non-creative educational Q&A shape.
  4. **Every other part of Founder Decision 003 is explicitly unchanged and unaffected by this decision**: the privacy allowlist (no email/name/user id/IP/session id/request id ever sent to the provider), the fixed system prompts and their prompt-injection defenses (`app.ai.prompt`), the post-generation safety gates (`app.ai.safety`'s numeric-integrity and advice-language checks, reject-don't-sanitize), the `AIProvider` Protocol abstraction itself, the caching/regeneration-cap rules, and the founder-approved fixed unavailable-message text (`explanation_service.SAFE_UNAVAILABLE_MESSAGE`) all remain exactly as FD-003 approved them — confirmed unmodified by direct review of `app/ai/prompt.py`, `app/ai/safety.py`, and `app/ai/service.py`, none of which changed in this pass. This decision touches only the adapter layer FD-003 itself designed to be swappable.
  5. Per `docs/FOUNDER_DECISIONS.md`'s own "never edit a previous Founder Decision" rule, Founder Decision 003's text is left completely unedited and remains the accurate historical record of the original Anthropic selection and its full rationale — this entry is the superseding record for the vendor-choice clause only, not a correction of FD-003, which was correct and fully justified for the milestone it governed.
- **Rationale**: A provider that cannot be run at all under this project's own accepted deployment topology (free-tier-only, per Founder Decision-level acceptance of ADR-047's free-tier deploy) is not a real option for shipping the Educational AI Panel — the Simulation Engine's own "100% functional with AI removed" guarantee (Founder Specification Principle 3) means AI staying `AI_PROVIDER=none` forever is *safe* but not the milestone's actual goal. Groq's OpenAI-compatible REST shape means the adapter swap costs exactly one new ~60-line file and zero prompt/safety/service changes — the strongest possible confirmation that FD-003's Protocol-based abstraction did what it was built for.
- **Scope**: Governs `app/ai/providers/` (the adapter layer only), `Settings.ai_provider`/`Settings.groq_api_key`/`Settings.ai_model_name`, `backend/requirements.txt` (the `anthropic` SDK dependency is removed, not replaced — Groq needs none), `.env.example`, and `render.yaml`'s AI env var block. Does not change `app/ai/prompt.py`, `app/ai/safety.py`, `app/ai/service.py`, `app/api/v1/services/explanation_service.py`, the `AIExplanation` persistence model, or any Founder Decision 003 privacy/integrity/caching rule.
- **Deferred**: None — implemented in full in the same pass as this approval. No live call against the real Groq API was made in this session (no `GROQ_API_KEY` available in this environment) — verified instead via a fully mocked provider test suite (`tests/ai/test_providers.py`) and the graceful-degradation path (`AI_PROVIDER` unset, `NullProvider` selected, confirmed via `tests/api/test_explanations.py`'s existing safe-fallback tests, unaffected by this swap). Live verification against a real key is the founder's to perform post-deploy, the same posture already established for this project's other un-live-verifiable infrastructure (`docs/DEPLOYMENT.md`).
- **Related**: Founder Decision 003 (the original Anthropic selection and full Educational AI System design — unedited, remains the authoritative record for everything this decision does not change), Founder Decision 015 (the anonymous/authenticated rate-limit tiers implemented in the same pass as this provider swap, ADR-049), `docs/ARCHITECTURE_DECISIONS.md` ADR-049 (this decision's full engineering record).
