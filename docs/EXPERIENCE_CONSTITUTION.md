# Experience Constitution — Investment Time Machine

**Status: APPROVED — Founder Decision 013 (2026-07-19), see `docs/FOUNDER_DECISIONS.md`.** This is the highest-level UX and interaction reference for the product: not a UI specification, not an animation specification, not a design system — the philosophy that governs how a person *experiences* Investment Time Machine, one level above both. It answers exactly one question, for every future interaction decision: **how should this feel to use?**

**Relationship to the documents beneath it**: `docs/BRAND_CONSTITUTION.md` governs how the product *looks* (visual identity, color, typography, iconography) and remains the authoritative source for those decisions. `docs/frontend_design_system.md` governs how the product is *built* (tokens, component specs, page inventory). This document sits upstream of both — where a future visual or implementation choice would require relaxing a principle stated here, that is a signal the choice needs to change, not this document. It does not restate detail already settled in those two documents (an exact color hex, an exact spacing value, an exact component prop) — it states the philosophy that detail must obey, the same relationship the Brand Constitution already holds toward the Design System, one level further up.

Synthesized from `docs/BRAND_CONSTITUTION.md`, `docs/frontend_design_system.md`, `docs/FOUNDER_DECISIONS.md` (001–004), and `docs/PROJECT_STATE.md`, plus the completed M7 Phase 2 Simulator build itself — the product's first working proof that these principles hold up in real, shipped interaction, not only in prose. Where a principle below is stated, a concrete example already built is cited beside it wherever one exists, so this document is falsifiable against real behavior, not just aspirational.

---

## 1. Purpose

This document exists because a design system can specify every token correctly and a product can still *feel* wrong to use — cold where it should be calm, cluttered where it should be quiet, or quietly manipulative where it should be plainly honest. `docs/BRAND_CONSTITUTION.md` answers "what does Investment Time Machine look like." This document answers the question underneath that one: **what does it feel like to sit down and use it?**

It exists to give every future interaction decision — a loading state, an error message, a transition, a disclosure, a screen's reading order — a single, consistent test to pass, so that decision doesn't have to be re-derived from first principles by whoever happens to be building it. A contributor who has never read `docs/frontend_design_system.md`'s component specs should still be able to read this document and correctly predict how a brand-new interaction *should* behave, because the philosophy — not a component library — is the actual source of consistency.

This is also, concretely, the frontend's expression of `.claude/SYSTEM.md`'s four non-negotiable engineering principles and the Founder Specification's own stated belief (restated in `.claude/MVP_RULES.md`) that "users value understanding historical investment outcomes." If the Simulation Engine's discipline is *never fabricate, never estimate*, this document's discipline is *never dramatize, never rush, never hide*.

## 2. Product Experience

Using Investment Time Machine should feel like sitting down with a former quant who now teaches — someone who has already done the math, already checked it twice, and is now walking you through exactly what happened and why, at your pace, never theirs (`docs/BRAND_CONSTITUTION.md` §2). It should never feel like opening a trading app, a dashboard, or a portfolio tracker. There is no ticker scrolling, no number that pulses when it changes, no reason to check back in five minutes. The product is retrospective by design — the outcome it shows already happened, years or decades ago — and every part of the experience should carry that stillness. Patience is not a stylistic choice here; it is the correct emotional register for a tool whose entire subject is the past.

Concretely, a session should move through three felt stages, in order, every time: **ask** (a specific, personal question — "what if I had invested $X in Y, starting on this date"), **see** (a precise, sourced answer, never an approximation dressed up as one), **understand** (an explanation of *why* that answer is what it is, available exactly as deep as the user wants to go and no deeper than they ask). A screen that skips straight to "see" without a real "ask" reads as a dashboard. A screen that buries "understand" behind friction reads as withholding. Both are experience failures, independent of whether every individual component on that screen is pixel-perfect.

## 3. The Three Layers

Every interaction in the product operates at one of three layers — Identity, Trust, and Behavior — and a well-formed feature engages all three deliberately, not just the one that happens to be easiest to build.

- **Identity — the product knows what it is.** It is a worked example generator, not a dashboard. Every screen should be answerable with "whose question is this answering, and what was the question?" A screen with no clear worked-example anchor has drifted from the product's actual identity, regardless of how polished it looks.
- **Trust — the product earns belief, never demands it.** Nothing is asserted without a path to its proof. A number with no reachable source is not acceptable output, no matter how confident it looks.
- **Behavior — the product acts like a precise instrument, not a performer.** Motion, timing, and feedback exist only to communicate state, time, or understanding — never to entertain, never to editorialize a result's size or sign.

These are not independent checklists — they compound. A worked example (Identity) that hides its assumptions is not trustworthy (fails Trust). A trustworthy explanation that arrives with a celebratory animation undermines its own credibility (fails Behavior, which then retroactively damages Trust — a theatrical product reads as a persuading one, and a product that's trying to persuade is not one whose numbers can be taken at face value). The three layers are best read as a single question asked three ways: *does this interaction know what it is, prove what it claims, and behave the way a precise instrument would?* If any one answer is no, the interaction is not finished, even if it ships.

## 4. The Worked Example

Investment Time Machine's central and only real content is the worked example: *if you had invested this amount, in this asset, starting on this date, and held until this date — here is precisely what would have happened.* Users arrive already thinking in this shape ("if I had bought Apple in 2015…"), and the product's job is to answer in that exact language, not to translate the question into an abstract portfolio-management vocabulary the user didn't ask for.

**Why it exists**: it is the concrete expression of `.claude/MVP_RULES.md`'s founding belief that people value *understanding* a historical outcome, and of Founder Decision 001's insistence on an explicit, auditable, event-by-event calculation over an opaque adjusted-price shortcut — the worked example is only trustworthy because every dollar of it can be traced to a real historical event, not a black-box number.

**Where it belongs**: the Simulator (asking the question) and the Results screen it feeds (answering it) are the worked example's home — this is the "simulate-and-explain loop" `.claude/MVP_RULES.md` names as the product's core success signal, and the only loop that exists in the product today (M7 Phase 2). Any future screen that shows a number *about* an investment without first anchoring it to a specific worked example (a specific asset, a specific amount, a specific date range someone actually chose) has left this layer's identity behind, regardless of what other value it adds.

**How it should evolve**: an Asset Explorer extends the worked example by making it easier to *pose* the question (browsing toward a good example, not computing anything itself); a Simulation History extends it by making a user's own past worked examples revisitable, not by aggregating them into portfolio-style analytics that would blur the product's identity toward a dashboard. Any future comparison or reporting feature (both explicitly excluded from M7 scope, Founder Decision 004) must, if ever built, still resolve to one or more individually legible worked examples — never an abstracted summary that no longer answers a specific "if I had invested…" question a real person asked. This is a governing test for that future work, not an implementation instruction.

## 5. The Proof

Trust here is never asked for — it is earned, continuously, by making every claim checkable. A user should never be in the position of simply believing the system; they should always be able to see *why* a number is what it is, at the depth they choose to look.

**Progressive disclosure**: every important value is available at a glance and explainable on demand, never one or the other. The Results screen's hero numbers each expose their source formula on tap or hover (`docs/BRAND_CONSTITUTION.md` §9) — the single highest-leverage detail in the whole product for signaling "this was built by people who care about being checked," and now a working, shipped pattern (`StatTile`'s source disclosure, extended in M7 Phase 2's closing pass to `ErrorState`'s own "Technical details" disclosure for Request ID and error code). Disclosure exists to deepen understanding on request, never to bury information a user actually needs to proceed — a required field's validation error, for instance, is never behind a disclosure; only *supplementary* detail is.

**Evidence and provenance**: every number the product ever shows carries a legible source — a field name, a formula, or an "as of" date reachable from the value itself (`docs/BRAND_CONSTITUTION.md` §9). A figure that cannot answer "where did this come from" is not acceptable output at any layer of the product, full stop.

**Assumptions, stated plainly, never hidden**: when the product makes a choice on a user's behalf — a day-count convention, a rounding rule, a decision not to move a date the user picked — that choice is stated, not silently applied. The Simulator's trading-day guidance is this principle in miniature: rather than silently choosing the nearest trading day for a user (a hidden assumption dressed as convenience), the product explains the actual constraint in advance and lets the user decide, because "historical accuracy matters more than convenience" is a *stated* value, not an implementation detail smoothed away.

**Educational transparency over persuasion**: an explanation teaches a concept once, plainly, and states a limitation directly when one exists, rather than omitting it (`docs/BRAND_CONSTITUTION.md` §2). A genuine data gap — a missing CPI observation, an AI explanation that safely failed, a simulation that could not complete — is stated as a plain fact, in the same calm register as a successful result, never smoothed over and never dressed up as an error the user caused.

**The AI layer's subordinate position is Trust's clearest test case**: the Educational AI System explains, it never calculates (Founder Decision 003) — and the experience must make that hierarchy *visible*, not just technically true. An AI panel that competes visually with a calculated result, or that is styled to look more special or more certain than the Simulation Engine's own output, has broken Trust at the experience layer even if the underlying calculation is untouched.

## 6. The Instrument

Motion, timing, and feedback in this product exist for exactly one purpose: to communicate that a correct thing has happened, is happening, or is about to happen. Never to entertain. Never to editorialize.

**Motion philosophy**: every animation in the product answers "what state changed" and nothing else. A button press is a 120–160ms tint, not a bounce — confidence reads as an immediate, certain response, not a playful one (`docs/BRAND_CONSTITUTION.md` §8). A result reveals itself once, the same way, regardless of whether it is a gain or a loss — a +150% outcome and a −40% outcome get the identical reveal treatment, because color and an explicit sign already carry the meaning, and motion is never allowed to add a second, editorializing layer on top. This is named, in the Brand Constitution, as the single hardest rule to violate by accident and the one most worth enforcing deliberately — it is restated here because it is as much an experience principle as a motion-token specification.

**Transitions**: the product's one recurring transition — a subtle horizontal slide along the time axis — is the only screen-level transition anywhere in the product (`docs/BRAND_CONSTITUTION.md` §5). One motion signature, applied consistently, is what makes motion legible as *communication* rather than decoration; a different transition per screen would read as ornamental variety for its own sake, exactly what this layer forbids.

**Interaction rhythm**: deliberate, never rushed, never urgent. Navigation gives instant active-state feedback (a confident product never makes a user wait to see where they are), while the one genuinely slow operation in the product — the simulation calculation itself — is named honestly while it runs ("Calculating historical returns…"), never disguised behind a generic spinner standing in for an unexplained wait. The rhythm of the product should always make clear which parts are instant and which parts are doing real work, and never blur the two together.

**Loading philosophy**: a skeleton shaped like the real layout it stands in for, for anything expected to take over 300ms — never a spinner floating in empty space, which communicates nothing about what's coming.

**Chart behavior** (governing future work, since no chart exists yet in the shipped product): a growth line draws in once, on first render only, and never re-animates on a re-render it didn't cause; event markers appear with a slight, deliberate delay after the line completes, so the causal read is "here's what happened, then here's what happened to it" — sequence communicating causation, never spectacle communicating excitement.

**Determinism**: the Simulation Engine is deterministic by design — identical inputs always produce an identical, auditable result. The interface should feel like the visible surface of that same determinism: the same input always produces the same interaction outcome, no randomized micro-variation in timing or presentation, nothing that would make two identical sessions *feel* different from each other. An instrument you can calibrate against is one that behaves the same way every time you use it.

**Interruptibility**: a precise instrument lets you stop, correct, and retry without penalty or friction. A user can always navigate away mid-flow, correct an input and resubmit, or have a stale, superseded request quietly cancelled rather than left racing a newer one — the Simulator's `AbortSignal`-based search cancellation (M7 Phase 2) is this principle already built, not aspirational: a keystroke that supersedes an in-flight search cancels it outright rather than letting two answers arrive out of order. Nothing in the product should ever trap a user mid-action or make correcting course feel costly.

**Never theatrical**: no confetti, no celebratory bursts, no bounce/spring physics on a gain, no animation whose intensity scales with a number's size or sign (`docs/frontend_design_system.md` §9). This is not a matter of taste — a product that celebrates a good outcome and stays silent on a bad one is a product nudging its user toward a feeling, which is exactly the trading-app energy this product exists to reject.

## 7. Experience Principles

- **Never overwhelm.** One clear primary question per screen; a screen needing a fourth visual weight to make sense is doing too much and should be split (`docs/BRAND_CONSTITUTION.md` §4).
- **Never surprise.** A constraint the user is about to hit (a date range with no data, a rate limit, a required field) is explained *before* it's hit wherever the product can reasonably anticipate it, not only after.
- **Always explain.** Every number has a reachable source; every error states cause and remedy in plain language; every limitation is named directly, never omitted.
- **Reveal complexity gradually.** Progressive disclosure is the default posture for anything beyond the one thing a screen is answering — never force a user to see everything at once, and never hide something they specifically came to see.
- **Reward curiosity, never manufacture it.** A user who taps a disclosure or asks a follow-up question is met with real, useful depth — the product should never bait engagement with an artificial hook it doesn't actually pay off.
- **Respect user attention.** Nothing on screen exists to hold attention for its own sake; every element serves the one question the screen is answering, or it doesn't belong (`docs/BRAND_CONSTITUTION.md` §13).
- **Trust grows through evidence, not assertion.** The product never asks to be believed — it shows its work, every time, at whatever depth is asked for.
- **The same calm register, regardless of outcome.** A −40% result and a +150% result are narrated, colored, and animated identically in every dimension except the number and its sign — this is the clearest single tell distinguishing an education platform from a trading one (`docs/BRAND_CONSTITUTION.md` §2).
- **Silence about a limitation is never acceptable; a guess is worse.** When the product doesn't know something — a data gap, an AI outage, an unverified assumption — it says so plainly, and never fills the gap with a plausible-sounding invention.
- **Speed is a form of respect, not a goal in itself.** Instant feedback where a user is waiting on the interface (navigation, validation); honest, named waiting where the product is doing real work — never the two confused for each other.
- **Expert and first-time users are served by the same interface, not two.** Progressive disclosure is what makes this possible: a first-time user can ignore every disclosure and still complete the task; an expert can open every one and move faster, without either persona needing a separate mode.

## 8. Interaction Anti-Patterns

Explicitly and permanently rejected — not because they're unfashionable, but because each one either dramatizes an outcome, manufactures urgency, or asks for belief instead of earning it, all of which are direct violations of this document's core test:

- **Decorative motion** — any animation that doesn't communicate a state change is noise, and noise erodes the instrument-like precision the product depends on for credibility.
- **Gambling mechanics** — streaks, near-miss framing, variable-reward reveals, or anything that makes checking a result feel like pulling a lever. This product's outcomes already happened; there is nothing to gamble on and nothing to be teased about.
- **Endless dashboards** — an unbounded feed of metrics with no anchoring worked example is a betrayal of Identity (§3) before it's anything else; if a screen can't say whose question it's answering, it doesn't belong in this product.
- **Information overload** — every additional visible number past what a screen needs to answer its one question is a tax on the next user's attention, not a bonus for a more sophisticated one (that user has progressive disclosure instead).
- **Unexplained numbers** — a figure with no reachable source is worse than no figure at all, because it borrows the product's earned credibility for something that hasn't actually earned it.
- **Urgency** — no countdown timers, no "prices moving now," no implied expiration on anything. Nothing in this product is time-sensitive; the outcome already happened, and pretending otherwise is manipulative by construction.
- **Fake scarcity** — nothing about a historical fact is scarce or limited-availability, and implying otherwise borrows persuasion tactics from a category of product this one explicitly is not.
- **Celebration of gains** — confetti, exclamation points, "you won" framing of any kind. A gain is a fact to be reported accurately, not a moment to be marketed.
- **Manipulative engagement mechanics** — streaks, daily-return prompts, notification bait, anything optimized for return visits rather than for understanding. This product's job is done the moment a user understands their answer; it does not need them back tomorrow to be a success.

## 9. Accessibility

Accessibility here is inseparable from the experience philosophy above — the same principles that make the product feel calm and trustworthy to any user are what make it actually usable by every user, not a separate compliance layer bolted on afterward.

- **Reduced motion**: every rule in §6 degrades to an instant state change with no information loss when `prefers-reduced-motion` is set — the count-up shows the final number immediately, the chart renders fully drawn, the transition becomes a cut. Nothing communicated by motion is *only* communicated by motion; it is always redundantly available as a static state.
- **Keyboard navigation**: every interaction — the Simulator's asset search, its checkboxes and disclosures, a future results tab set or history table — must be fully operable without a mouse, matching `docs/frontend_design_system.md` §11. The shipped `AssetSearchCombobox`'s full ARIA combobox pattern (arrow-key navigation, Enter to select, Escape to close) is this principle already proven in production, not aspirational.
- **Screen readers**: structure is real structure — semantic headings, not styled `<div>`s standing in for them; a disclosure is a native `<details>`/`<summary>`, operable and announced correctly with zero additional JavaScript, the same pattern used for `StatTile`'s source disclosure and (as of M7 Phase 2's closing pass) `ErrorState`'s technical details.
- **Mobile**: the primary persona (a student, per `.claude/MVP_RULES.md`) is mobile-heavy; progressive disclosure is what lets a dense, considered-decision screen like the Simulator still resolve to a small, mobile-comfortable primary path, with secondary options behind a "More options" disclosure rather than a scaled-down version of the desktop layout.
- **Slower users**: nothing in the product times out an interaction, penalizes hesitation, or loses state if a user steps away mid-task. A calm product is, definitionally, one that never rushes anyone.
- **Expert users**: the same interface that never overwhelms a first-time user must never slow down an expert one — progressive disclosure again does this work, letting depth be opt-in rather than a tax every user pays regardless of whether they wanted it.

## 10. Future Evolution

This document is written to hold as the product grows past M7 — a Results screen, an AI panel, new asset classes, a possible mobile app — without needing to be rewritten for each. A useful, permanent test for any future interaction addition: **does it know what it is (Identity), does it earn belief rather than assert it (Trust), and does it behave like a precise instrument rather than a performer (Behavior)?** A proposed feature that requires relaxing one of these three to ship is a signal the feature needs to be redesigned, not that this document needs an exception — the same standing rule `docs/BRAND_CONSTITUTION.md` §14 already applies to the brand layer, extended here to the experience layer above it.

**What may evolve freely**: the specific motion timings and easing curves (as long as the *rules* in §6 — communicates state, never theatrical, identical treatment regardless of outcome sign — are unchanged); new disclosure patterns for new content types, provided they follow §5's progressive-disclosure posture; new worked-example surfaces (Asset Explorer, Simulation History, and beyond) built per §4's evolution guidance; platform-specific adaptations (a future native mobile app would express these same three layers through native idiom, not a literal copy of the web interaction language).

**What must never change**: the three-layer model itself (§3) as the lens every future interaction decision is checked against; the rule that a gain and a loss are never treated differently by color intensity, motion, or language (§6, §7); the rule that every number carries a reachable source (§5); the rule that the AI is always experientially subordinate to the Simulation Engine's own calculated output (§5); and the core felt target this whole document exists to protect — a former quant who teaches, never a trading app that persuades.

## Founder Decision 013: Experience Philosophy — Approved

*Approved 2026-07-19, formalized in `docs/FOUNDER_DECISIONS.md` in full. The text below is retained verbatim as the original proposal record; the authoritative decision entry is in `docs/FOUNDER_DECISIONS.md`.*

- **Context**: M7 Phase 0 through Phase 2 established what the product looks like (`docs/BRAND_CONSTITUTION.md`) and how its first screen was built (`docs/frontend_design_system.md`, the shipped Simulator). Neither document, by design, states the interaction-level philosophy that should govern every *future* screen's feel — loading states, transitions, disclosure patterns, error tone — leaving that judgment to be re-derived per screen rather than stated once, authoritatively, above both existing documents.
- **Decision**: The founder approves `docs/EXPERIENCE_CONSTITUTION.md` as the product's highest-level UX and interaction philosophy, upstream of `docs/BRAND_CONSTITUTION.md` (visual identity) and `docs/frontend_design_system.md` (implementation), governing every future interaction decision. Specifically approves:
  1. **The three-layer experience model** (Identity/Worked Example, Trust/Proof, Behavior/Instrument, §3) as the standing test applied to every future screen and interaction before it ships.
  2. **The Worked Example as the product's sole content identity** (§4) — every current and future screen must anchor to a specific, personal "if I had invested…" question; no screen may become an abstract dashboard or portfolio-analytics surface.
  3. **The Proof principles** (§5) — progressive disclosure as the default posture for depth beyond a screen's one question, every number carrying a reachable source, assumptions always stated rather than silently applied, and the AI layer's permanent experiential subordination to the Simulation Engine's calculated output.
  4. **The Instrument principles** (§6) — motion communicates state only, an identical reveal treatment regardless of a result's sign, determinism and interruptibility as first-class interaction properties, and the permanent rejection of theatrical/celebratory treatment for any financial outcome.
  5. **The interaction anti-pattern list** (§8) as a standing rejection list, with the same permanence `docs/BRAND_CONSTITUTION.md` §13's "Things We Never Do" already carries for the visual layer.
- **Rationale**: Every future interaction decision currently has no single authoritative philosophy to check against above the visual/implementation layer, meaning the same judgment call (how should this loading state feel? how should this error read?) risks being re-derived inconsistently, screen by screen, by whoever happens to build it. This decision closes that gap the same way Founder Decision 004 closed it for the visual layer — before, not after, the next several screens are built.
- **Scope**: Governs the frontend's interaction/UX philosophy for all future work (M7 Phase 3 onward). Does not change any already-approved visual token, component spec, or backend behavior; does not retroactively require rework of the shipped Simulator, which this document was written to describe and validate against, not to invalidate — every principle stated above already has a shipped, working example cited beside it.
- **Deferred**: No open items — this proposal is complete as written and does not depend on any unresolved question elsewhere in the project.
